# ESP32 灌溉项目协作规范

## 当前阶段

本项目已完成新版电路板和新版产品架构的结果性方案整理。当前仓库里的旧代码、现有固件骨架、旧设计文档、旧硬件文档和此前整理的需求，都不能直接视为最新方案或实现依据。

当前新版方案入口是 `docs/plans/`。该目录只保留结果性设计文档，覆盖硬件边界、配置模型、运行调度、安全状态、流量保护、RS485 通信、Web/API、告警记录和验证验收。

`docs/prototypes/web/` 只能作为早期交互沟通参考，不能替代 `docs/plans/` 中的新版方案文档。当前仍未进入实施计划和代码实现阶段。

## 基本规则

- 默认使用中文沟通，表达直接、清晰、务实。
- 不要假设当前代码、旧文档或旧规则仍然正确；使用前必须说明其状态和不确定性。
- 用户提出新版方案、PCB 信息、交互想法或需求变更时，先判断是否需要更新 `docs/plans/` 的结果性方案，不要直接进入实现。
- 在实施计划确认前，不新增正式业务代码，不按旧架构继续扩展功能。
- 可以阅读旧资料和现有代码作为背景，但不能把它们当作约束，除非用户明确指定某一部分继续沿用。
- 旧项目归档目录 `legacy/2026-06-10-pre-rewrite/` 不做修改，除非用户明确要求。
- 涉及硬件、水路、泵、流量计、低液位、电磁阀、供电、安全策略、芯片接口和平台能力时，必须优先依据官方文档、数据手册、成熟产品规则或行业实践；无法核实时明确说明依据和风险。
- 涉及删除、覆盖、推送、部署、烧录、修改账号/权限/配置等高影响操作时，先说明影响并取得确认。

- 构建、测试、烧录和实机验证规则
