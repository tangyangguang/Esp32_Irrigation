# ESP32 灌溉项目协作规则

## 依据与边界

- 默认使用中文沟通，结论直接、清晰，并说明依据和剩余工作。
- 硬件事实以 `pcb_irrigation/` 下 2026-07-11 的 BOM 和网表为准；产品与软件方案以 `docs/当前方案/` 为准。
- 当前产品是单板 12V DC 智能浇水控制器，具体功能和边界不在本文件重复，以当前方案文档为准。
- 固件基于同级目录 `../Esp32Base`，优先使用其启动、WiFi、Web、认证、OTA、文件系统、日志、事件、配置、健康、看门狗、时间和 RTC 能力。

## 判断与设计

- 实事求是：先检查当前文档、代码、配置、构建结果和权威资料再作判断；禁止伪造、臆想或把经验当成事实。
- 已确认事实、用户决定、推断和未知信息必须区分；无法核实时说明缺失信息、假设、风险和验证方法。
- 第一性原理：从用户目标、实际场景、物理约束、安全要求和成功标准推导方案；历史实现和同类产品只作参考，不能照搬。
- 优先采用满足当前需求的最简单可靠方案；关键反馈后重新检查原方案，不为未确认需求增加抽象、兼容或扩展设计。
- 产品、页面、文档和代码使用成熟行业术语并保持一致；用户口头表达不自动成为最终命名。

## 实施规则

- 当前只有一个最终版本。发现现有方案或代码不合理时，直接改成当前确认的正确实现，不保留旧字段、旧页面、旧 API、旧流程或迁移分支。
- 配置结构和含义未变时，重新编译、烧录或 OTA 必须保留已有有效配置，默认值不能覆盖用户配置；结构不兼容时安全停机并明确提示重新配置，不做历史迁移。
- 项目业务代码只实现浇水领域能力，不重复实现 `Esp32Base` 已有功能，也不为基础库问题增加旁路或重复框架。
- 只能修改本项目工作区。禁止修改 `../Esp32Base`；发现基础库问题时，只提供问题说明、建议和可转交维护者的提示词。
- 涉及产品信息结构、页面导航、核心交互或关键配置模型时，先给整体方案并等待用户确认，再修改代码或文档。
- 不删除、覆盖或回滚用户改动，不提交密钥、账号、隐私数据或敏感配置。

## 验证与交付

- 修改前确认影响范围；代码改动至少运行 `pio run -d master-esp32`，并按风险补充相关测试或说明实机验证缺口。
- 代码或文档完成后提交并推送本次改动；用户明确要求不提交或不推送时除外。
