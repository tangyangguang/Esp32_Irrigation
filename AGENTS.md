# ESP32 灌溉项目协作规范

## 当前阶段

本项目处于产品方案和 Web 原型重建阶段。旧方案、旧原型、旧固件骨架和历史过程文件已经移入待删除区，不再作为当前方案依据。

当前阶段先确认硬件边界、产品方案、对象模型、状态模型、配置模型、任务流和信息架构；方案确认后再进入高保真 Web 原型。原型确认前不进入固件、业务代码、API、存储或通信协议实现。

## 当前权威入口

- 硬件资料：`pcb/`
- 当前文档入口：`docs/README.md`
- 待删除历史资料：`待删除/2026-06-29-历史过程文件/`

涉及具体方案时，先查 `docs/` 下当前方案文档，不把 `AGENTS.md` 当作产品方案来源。

## 依据规则

- 涉及硬件能力时，只以 `pcb/` 下当前 BOM 和网表为依据。
- 如果 PCB 文件缺失、被删除或状态异常，只报告风险和影响，不自行恢复、覆盖或推断。
- 历史文档、旧 README、旧原型、旧固件和旧计划只允许作为背景线索，不允许作为当前结论依据。
- 当前没有确认的产品能力、页面、API、协议、对象和状态，不得为了延续旧方案而补全。
- 文档之间冲突时，先指出冲突、影响和可选处理方式，再按用户裁决或当前方案文档修正。
- 不要把当前方案细节写入 `AGENTS.md` 作为硬性规则；方案会变化，具体结论应维护在 `docs/` 中。

## 工作方式

- 用户提出想法、疑问、建议或方案时，先判断目标、约束和成功标准是否变化，不要直接执行，除非用户明确要求执行。
- 每次关键反馈后重新检查旧方案是否仍成立；需要推翻时直接指出。
- 信息不足时明确写出缺失信息、当前假设、风险和需要用户裁决的问题。
- 修改文档前先理解上下文和影响范围，不删除、覆盖或回滚用户已有改动，除非用户明确要求。
- 原型确认前不编写实施计划，不进入固件实现，不决定工程目录、类名、存储实现、API 或通信协议细节。

## 高影响操作

删除、覆盖、清理、提交、推送、部署、烧录、发起 OTA、修改账号、权限或配置前，必须先说明影响并取得用户确认。

用户已经要求旧资料进入待删除区，因此旧资料移动到 `待删除/2026-06-29-历史过程文件/` 不再视为当前方案变更。

## 回复要求

默认使用中文沟通。表达直接、清晰、务实；不要为了配合而附和。每次结束回复都说明是否还有剩余工作；如果有，说明下一步、风险或需要用户决策的点。
