# ESP32 灌溉项目协作规则

## 当前阶段

项目现在同时推进：

- Web 原型优化。
- 软件代码实现。

原型和代码都必须基于当前有效方案，不沿用失败原型、旧固件或历史过程文件。

## 有效依据

- 硬件事实以 `pcb/` 下当前 BOM 和网表为准。
- 产品与软件方案以 `docs/当前方案/` 为准。
- `docs/历史参考/` 和 `待删除/` 只能作为背景参考，不能直接继承为当前结论。

资料冲突时，先指出冲突和影响，不自行补全。

## 原型规则

原型面向最终用户，不做后台表单、说明文档或工程调试页。

页面结构、状态表达、操作入口和视觉设计都要服务真实产品体验。无真实数据时，使用空状态、待配置状态、默认值或不可操作状态，不编造运行数据。

列表、详情、编辑、新建、校准等页面职责要清楚，不把复杂内容堆在同一页。

## 软件实现规则

代码只实现当前灌溉产品需要的功能，不做通用平台。

主控基于 `Esp32Base`，优先复用基础库已有能力。分站基于 `Stc8hBase`，基础库缺少通用能力时，优先回基础库补齐，不在本项目里写临时绕过。

协议、引脚、模块边界以 `docs/当前方案/10-软件实现前收口计划.md`、`11-RS485协议与工程骨架.md`、`12-引脚映射与基础库核实.md` 为主要依据。

不要新增云端、多用户、天气联动、复杂统计、通用 IO 平台、复杂权限系统或调试后台，除非用户重新确认目标变化。

## 工作方式

用户讨论想法时，先判断目标、约束和成功标准是否变化，不要机械沿旧方案优化。

用户明确要求修改、实现或修复时，默认直接执行。缺失信息会导致方向错误、数据丢失或不可逆影响时，先问清楚。

修改前先理解影响范围。不要回滚用户已有改动，除非用户明确要求。

代码任务完成后，可以直接提交并推送本次改动；用户明确要求不提交或不推送时除外。

## 高影响操作

删除、覆盖已有资料、部署、烧录、修改权限或配置前，先说明影响并取得确认。

提交和推送不属于高影响操作，除非它们包含上述风险动作。

## 回复要求

默认中文沟通。表达直接、清晰、务实。

每次结束说明是否还有剩余工作。
