# ESP32 灌溉项目协作规范

## 当前阶段

本项目正在进入新版电路板和新版产品架构重新梳理阶段。当前仓库里的旧代码、现有固件骨架、设计文档、硬件文档和此前整理的需求，都不能直接视为最新方案或实现依据。

当前新版方案入口是 `docs/plans/`。其中编号文档记录新版 PCB、硬件启用范围、通信协议、配置模型、运行状态、Web/API、告警记录和复评结论。

`docs/prototypes/web/` 只能作为早期交互沟通参考，不能替代 `docs/plans/` 中的新版方案文档。当前仍处于方案完善和复评阶段，确认完成后再编写实施计划和代码。

## 基本规则

- 默认使用中文沟通，表达直接、清晰、务实。
- 不要假设当前代码、旧文档或旧规则仍然正确；使用前必须说明其状态和不确定性。
- 用户提出新版方案、PCB 信息、交互想法或需求变更时，先讨论和整理边界，不要直接进入实现。
- 在新版 PCB 和整体方案确认前，不新增正式业务代码，不按旧架构继续扩展功能。
- 可以阅读旧资料和现有代码作为背景，但不能把它们当作约束，除非用户明确指定某一部分继续沿用。
- 旧项目归档目录 `legacy/2026-06-10-pre-rewrite/` 不做修改，除非用户明确要求。
- 涉及硬件、水路、泵、流量计、低液位、电磁阀、供电、安全策略、芯片接口和平台能力时，必须优先依据官方文档、数据手册、成熟产品规则或行业实践；无法核实时明确说明依据和风险。
- 涉及删除、覆盖、推送、部署、烧录、修改账号/权限/配置等高影响操作时，先说明影响并取得确认。

## 后续更新

等新版 PCB 信息、产品方案、页面原型和交互流程最终确认后，再继续完善本文件。届时再明确：

- 最新硬件能力和接口边界
- 软件架构和基础库边界
- 文档入口和实现计划
- 构建、测试、烧录和实机验证规则
- Web/API、本地交互、记录和告警策略
