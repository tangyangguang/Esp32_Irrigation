# 灌溉系统核心逻辑说明

最后更新：2026-06-05

本文档定义当前固件采用的核心业务模型。旧的固定 4 路、Road 命名、每路一个流量计和启动补偿校准模型均不再作为实现依据。

## 一、核心模型

系统分为两个层次：

- Flow：流量计和供水计量资源，最多 2 个。
- Zone：电磁阀控制的浇水水路，最多 6 个。

每个 Zone 配置 `flowId = 1` 或 `flowId = 2`，可以任意归属：

```text
示例 A：Zone 1/3/5 -> Flow 1，Zone 2/4/6 -> Flow 2
示例 B：Zone 1 -> Flow 1，Zone 2/3/4/5/6 -> Flow 2
示例 C：全部 Zone -> Flow 1
```

运行规则：

- 同一个 Flow 下，同一时刻最多一个 Zone 运行。
- 不同 Flow 下的 Zone 可以并行运行。
- 不增加 Supply/Pump 软件资源；接线说明中提醒用户只有两个 Flow 确实来自独立供水支路时才建议并行。

## 二、默认配置

- Flow 1 默认启用。
- Flow 2 默认禁用。
- Zone 1/2 默认启用。
- Zone 3..6 默认禁用。
- 所有 Zone 默认归属 Flow 1。

禁用的 Zone 不在首页和本地屏幕主操作列表中展示，也不能启动；但可在 Zone 设置页重新启用。

## 三、板级硬件

引脚不开放页面自由选择。当前板级配置：

| 对象 | 引脚 |
| --- | --- |
| Flow 1 | GPIO32 |
| Flow 2 | GPIO35 |
| Zone 1 Valve | GPIO16 |
| Zone 2 Valve | GPIO14 |
| Zone 3 Valve | GPIO13 |
| Zone 4 Valve | GPIO27 |
| Zone 5 Valve | GPIO4 |
| Zone 6 Valve | GPIO5 |

GPIO35 是输入脚，流量计输入必须确认外部或板载上拉。GPIO5 是启动采样相关引脚，Valve6 实机接线后必须验证不会影响启动。

## 四、阀门控制

阀门输出使用 PWM：

```text
开阀吸合阶段：100% 占空比
吸合时间：IrrigationPins::ValvePullInMs
保持阶段：IrrigationPins::ValveHoldDutyPercent
关阀：0% 占空比
```

这些参数是软件默认值，不等于最终硬件结论。必须根据实际电磁阀、电源、驱动模块、温升和吸合可靠性实测调整。

## 五、Zone 状态机

Zone 状态：

```text
DISABLED -> IDLE -> STARTING -> RUNNING -> IDLE
                         |          |
                         v          v
                       ERROR      ERROR
```

启动条件：

- Zone enabled。
- Zone 无故障。
- 归属 Flow enabled。
- 归属 Flow 当前空闲。
- 归属 Flow 无待机异常流动保护。
- 当前没有 Flow 校准任务。
- 目标时长在系统最大浇水时长以内。

## 六、手动与计划任务

手动任务和计划任务使用同一套运行记录结构。区别只在启动来源：

- Web 页面
- HTTP API
- 本地按钮
- 计划调度器

停止来源包括：

- Web 页面
- HTTP API
- 本地按钮
- 目标时长到达
- Flow 监控
- Flow 待机漏水监控
- 配置变更
- 恢复出厂流程

## 七、计划调度

每个 Zone 最多 6 条计划，总计最多 36 条计划。计划按 Zone 保存，但启动时遵守 Flow 互斥。

同一时间多个计划到点时：

- 不同 Flow 可同时启动。
- 同一 Flow 只有第一个可启动，其余进入内存队列。
- 队列等待超过 `queuedPlanMaxDelaySec` 后过期并记录事件。

不实现停水等待后自动恢复补浇；缺水、无脉冲或确认的流量异常默认停机并进入故障。

## 八、Flow K+Offset 校准

Flow 使用固定点 K+Offset 模型：

```text
measuredMilliHz = pulses * 1000000 / durationMs
correctedMilliHz = measuredMilliHz + offsetMilliHz
flowUlPerMin = correctedMilliHz * kUlPerMinPerHz / 1000
totalMl = flowUlPerMin * durationMs / 60000 / 1000
```

参数语义：

- `kUlPerMinPerHz`：每 1 Hz 脉冲频率对应多少 uL/min 流量。
- `offsetMilliHz`：频率偏移量，用于修正低流量或传感器零点偏差。
- `minValidFreqMilliHz`：低于该频率时水量估算可以继续，但应提示精度下降。
- `warningFreqMilliHz`：低流量提醒阈值。
- `maxValidFreqMilliHz`：最大可信频率，0 表示不限制。

手工修改和采样校准都先写入 pending；只有用户明确 apply 后才替换 active，旧 active 进入 rollback。

## 九、Zone 正常流量基线

Zone 基线用于判断运行中的低流量和高流量：

- `learnedFlowMlPerMin`
- `lowFlowPermille`
- `highFlowPermille`
- `flowFaultConfirmSec`
- `noPulseTimeoutSec`
- `lowFlowAction`
- `highFlowAction`

低流量和高流量必须连续超过 `flowFaultConfirmSec` 才触发动作。无脉冲保护独立判断，只要超过 `noPulseTimeoutSec` 没有任何脉冲，就停机。

基线手工修改也先保存 pending；apply 后才进入 active，旧 active 进入 rollback。

## 十、待机漏水保护

待机漏水是 Flow 级异常，不伪装成某个 Zone 故障。

当没有任何 Zone 运行、也没有校准任务时，如果某个 Flow 在 `idleLeakWindowSec` 窗口内累计脉冲数达到 `idleLeakPulseThreshold`：

- 写入 Flow 级告警。
- 记录 `flow:<id>` 业务事件。
- 阻断归属该 Flow 的 Zone 启动。
- 不影响另一个没有告警的 Flow。

## 十一、本地交互

本地按钮只做最基本功能：

- 选择上一个 enabled Zone。
- 选择下一个 enabled Zone。
- 对当前 Zone 启动或停止，必须同键二次确认。
- Stop All，必须同键二次确认。
- 可选信息键切换显示页。

本地不做校准、学习、容量输入或复杂配置。I2C 屏幕只显示当前选择、Flow、状态、阻断原因、估算水量和故障摘要。

## 十二、Web/API

当前业务页面：

- `/irrigation`
- `/irrigation/flows`
- `/irrigation/zones`
- `/irrigation/plans`
- `/irrigation/calibration`
- `/irrigation/settings`
- `/irrigation/records`
- `/irrigation/events`

所有改变状态的页面表单必须使用 POST，并带 JavaScript `confirm`。服务端仍必须重新校验 enabled、Flow busy、Zone busy、Flow disabled、Flow leak protected 等条件。

## 十三、记录和事件

浇水记录保存到 `/irr/records_v1.bin` 定长环形文件。

业务事件写入 Esp32Base App Events，不另建第二套事件存储。事件对象使用新模型命名：

- `zone:<id>`
- `flow:<id>`
- `plan:<id>`
- `system:irrigation`
- `config:<store>`
