# ESP32 灌溉项目协作规范

## 当前阶段

本项目硬件能力边界和产品逻辑已经基本确定，当前处于最终高保真 Web 原型修改阶段。原型确认前不进入代码实施，不新增或重写正式业务代码。

当前任务优先级：

1. 保持硬件依据、结果性方案和最终原型一致。
2. 修正 `docs/prototypes/web/final-prototype.html` 的交互、信息架构和视觉问题。
3. 原型确认后，再编写实施计划。
4. 实施计划确认后，才进入代码实现。

## 无效依据

旧原型、中间原型、旧设计文档、过程记录、提示词、复评记录和 `legacy/` 内容都不能作为当前方案依据。它们即使存在，也只能在用户明确要求时作为历史背景查看，不能覆盖 `pcb/`、`docs/plans/` 和最终原型。

## 已确认的一版边界

- 一个从站对应一个真实水源。
- 一个从站最多 8 路阀输出、最多 1 个流量计、最多 1 个主阀。
- 电压监测只用于显示、告警和记录，不参与自动停机、启动禁止或入口电源控制。
- 从站只做固定硬件抽象、本地安全、RS485 业务命令和 OTA 接收；不放计划、Web、复杂规则、通用 IO、通用寄存器或脚本逻辑。
- Web/API 必须符合 `Esp32Base` 当前能力：固定路径 + query 参数；不使用 REST 路径参数、请求体 JSON、`PUT`、`DELETE`。
- 第一版不支持多主控、从站主动上报、一个从站多个流量计、同站多水路并行、Web 任意 RS485 透传。

## 原型修改规则

- 修改 `final-prototype.html` 前，先核对 `docs/plans/01..09`，不能凭视觉效果发明功能。
- 原型可以表达最终显示和交互，但不能引入 `docs/plans/` 未确认的新功能边界。
- 可点击原型变更后，同步检查是否需要更新 `docs/plans/09-Web界面与交互原型.md`。
- 事件和浇水记录必须分开：事件处理告警、阻断、同步、OTA 等状态；记录只追溯水路级运行事实。
- 设置页不能变成系统设置页。系统时间、LCD、SHT30、RS485、刷新间隔、记录保留等不作为普通配置项；需要表达时进入诊断详情、本地说明或对应状态提示。
- OTA 只能从维护页进入，不能放顶层导航、事件页、诊断详情、通知或 LCD。
- 水源列表、配置表单和水路编辑必须保持职责分离；列表视图和完整编辑表单不得默认同屏。
- 运行入口只在总览和运行页；设置、维护、诊断、事件、记录、统计、天气、通知和 LCD 不能越权发起普通 Web 手动灌溉。
- 天气只显示、解释和建议，不自动停机、不禁止启动、不自动跳过计划、不自动改时长、不自动改配置。

## 文档维护规则

- `docs/plans/` 只保存结果性方案，不新增过程记录、提示词、复评日志或临时草稿。
- 新增或修正方案内容必须写入对应编号文档；跨多个文档且确实需要长期入口时，才更新 `docs/plans/README.md`。
- 不要沿用旧方案惯性。用户提出新想法、PCB 信息、交互反馈或需求变更时，先判断它属于对象、页面职责、任务、流程、视觉规则、原型还是代码，再决定是否修改文档或原型。
- 发现资料冲突、硬件能力疑问、产品边界变化、过度设计风险或命名分歧时，先说明冲突和影响，再给推荐处理方式。

## 实施前限制

- 原型确认前，不进入代码实施。
- 实施计划确认前，不决定工程目录、具体类名、文件拆分、存储介质实现、EEPROM/IAP 算法、分区表或基础库 API 修改。
- 可以阅读旧代码和现有固件骨架作为背景，但必须说明其状态和不确定性，不能按旧架构继续扩展功能。
- 如果基础库缺少通用能力，不在业务项目里绕路补丁，应先沉淀到对应基础库。

## 高影响操作

删除、覆盖、清理、提交、推送、部署、烧录、清空从站配置、发起 OTA、修改账号/权限或配置前，必须先说明影响并取得用户确认。

代码任务完成后，只有在用户明确要求提交和推送时才执行 git 提交或推送。

## 回复要求

默认使用中文沟通。表达直接、清晰、务实；信息不足时说明缺失信息、当前假设和风险。每次结束回复都说明是否还有剩余工作；如果有，说明下一步、风险或需要用户决策的点。
