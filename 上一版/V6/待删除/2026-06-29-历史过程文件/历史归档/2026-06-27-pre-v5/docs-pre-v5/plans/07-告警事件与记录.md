# 07 告警事件与记录

## 概念边界

| 概念 | 回答的问题 | 是否当前态 | 是否历史态 |
|---|---|---|---|
| `Alarm` 告警 | 现在用户需要处理什么 | 是 | 可归档 |
| `Fault` 故障 | 设备为什么不能运行或为什么停机 | 是 | 可记录 |
| `Event` 事件 | 过程中发生了什么可解释动作 | 否 | 是 |
| `RunRecord` 浇水记录 | 一个水源下某一路区域的一次运行或运行尝试结果 | 否 | 是 |
| `Notification` 通知 | 哪些结果或异常需要主动提醒用户 | 否 | 投递结果可记录 |

规则：

- 清除告警不删除事件和浇水记录。
- 清除故障不表示真实问题已经修好。
- 历史记录不能被当前状态覆盖。
- 一个故障通常会产生一个告警和一个事件。
- 一次故障停机会结束一个 `RunSession` 并生成浇水记录。

## 影响范围

| 范围 | 含义 |
|---|---|
| `system` | 影响主控或整体系统 |
| `station` | 影响一个从站 |
| `zone` | 影响一个分支水路 |
| `run` | 只影响一次运行 |
| `config` | 影响配置可用性 |

从站级故障默认不影响其他从站。水路级锁定默认只影响该水路。

## 告警生命周期

```text
raised -> active -> acknowledged -> condition_cleared -> cleared -> archived
```

规则：

- 用户确认 `acknowledged` 不等于清除。
- 条件恢复 `condition_cleared` 不等于用户已经处理。
- 只有满足清除条件后，才能进入 `cleared`。
- 进入 `cleared` 后仍保留历史事件。

## 故障生命周期

```text
triggered -> protected -> waiting_clear -> cleared
```

规则：

- 故障触发后必须先进入安全状态，再允许清除。
- 不可清除故障只能通过修复条件或重新同步配置解决。
- 清除故障后不能自动启动运行。

## 告警生成规则

| 条件 | 范围 | 是否禁止运行 |
|---|---|---|
| 低液位 | `station` | 是，禁止该从站 |
| 启动无水 | `station` 或 `zone` | 是，按锁定范围 |
| 运行中无水 | `station` 或 `zone` | 是，按锁定范围 |
| 高流量 | `zone` | 是 |
| 低流量 | `zone` | 按策略 |
| 待机漏水 | `station` | 按 `standby_leak_action`，第一版默认禁止该从站新运行 |
| 通信丢失 | `station` | 是，直到确认状态 |
| 地址疑似冲突 | `system` 或 `station` | 是，禁止相关地址 |
| 配置未同步 | `config` | 是，禁止相关从站 |
| 配置无效 | `config` | 是 |
| 电压检测未校准 | `station` | 否，仅提示电压监测未启用 |
| 已启用电压监测并确认电压低/高 | `station` | 否，仅告警和记录 |
| 主控时间无效 | `system` | 禁止自动计划 |

低/高流量告警只在流量计输入有效、目标水路标准流速已学习、对应保护已启用后生成。标准流速未学习时只记录保护未启用，不生成阻断性低/高流量告警。

`low_flow_action=alarm_only` 时只生成告警和事件，不禁止启动、不触发从站停机；`low_flow_action=stop` 时按低流量停机策略处理。`high_flow_action=stop` 时按高流量停机并锁定处理。

电压检测未校准或电压监测未启用时，不生成电压低/高告警。

电压低/高告警不禁止启动、不触发停机、不进入锁定；它只用于提示用户检查电源、线路、负载和现场供电质量。

## 清除规则

| 告警或故障 | 清除条件 |
|---|---|
| 低液位 | 液位恢复，或用户确认并允许清除 |
| 无水 | 用户确认检查水源、泵、阀、流量计 |
| 高流量 | 用户确认检查管路、脱管、阀门和流量计 |
| 低流量 | 用户确认，或策略允许自动恢复 |
| 待机漏水 | 用户确认检查后，且当前无持续脉冲 |
| 通信丢失 | 从站重新上报安全状态后清除 |
| 地址疑似冲突 | 重新扫描后异常消失 |
| 配置未同步 | 同步成功 |
| 配置无效 | 重新同步有效配置 |
| 电压检测未校准 | 完成校准或用户确认暂不启用 |
| 已启用电压监测并确认电压低/高 | 电压恢复到告警阈值内，或用户确认后清除 |
| 主控时间无效 | 时间恢复有效 |

规则：

- 清除失败必须显示原因。
- 清除后不自动启动。
- 如果清除后条件仍存在，应重新生成告警。

## 事件记录

事件类型：

| 类型 | 例子 |
|---|---|
| `run_started` | 手动或计划启动水路 |
| `run_stopped` | 正常停止或用户停止 |
| `run_faulted` | 本地保护停机 |
| `schedule_triggered` | 计划触发 |
| `schedule_skipped` | 计划项跳过 |
| `auto_watering_paused` | 用户暂停自动浇水 |
| `auto_watering_resumed` | 到期或用户提前恢复自动浇水 |
| `config_changed` | 用户保存配置 |
| `config_sync` | 配置同步成功或失败 |
| `fault_cleared` | 用户或维护动作清除故障 |
| `maintenance` | 扫描、识别、恢复配置 |
| `calibration` | 流量计校准和标准流速学习 |
| `firmware_update` | ESP32 OTA、从站 RS485 OTA 开始、成功、失败、重试或中断 |
| `power_restored` | 主控重新上电或断电后恢复在线 |
| `notification_sent` | 通知发送成功 |
| `notification_failed` | 通知发送失败 |

事件不能被清除故障删除。

## 浇水记录

`RunRecord` 必须包含：

```text
run_id
sequence_id
run_type
trigger_type
trigger_name
station_id
zone_index
started_at
ended_at
requested_duration_sec
actual_duration_sec
result
stop_reason
pulse_count
total_ml
flow_summary
config_version
had_lost_contact
disabled_protections
estimation_quality
created_by
```

记录定义：

- 一条 `RunRecord` 对应一个水源下某一路区域的一次运行或运行尝试。
- `trigger_type` 表示触发方式：手动、计划或维护。
- `trigger_name` 表示用户可理解的来源，例如“Web 手动运行”“本地 LCD 手动运行”“早晨灌溉”“流量计校准”。
- `sequence_id` 可为空；Web 手动顺序运行中的多条水路记录使用同一 `sequence_id` 关联，但记录页仍以水路级记录为主。
- 一个计划触发多个区域时，每个区域分别生成一条 `RunRecord`。
- 未启动即跳过的计划项如果需要向用户解释，也按对应区域生成一条 `RunRecord`，结果为 `skipped`，并保留跳过原因。
- 自动浇水暂停导致的计划项未启动，应记录为 `skipped`，跳过原因指向全局暂停或对应水源暂停。
- 不再单独定义“计划整体记录”作为记录页对象；计划触发过程中的解释性信息进入事件，水路级结果进入 `RunRecord`。

运行结果：

| 结果 | 含义 |
|---|---|
| `completed` | 按预期完成 |
| `stopped_by_user` | 用户停止 |
| `skipped` | 未启动即跳过 |
| `failed_to_start` | 启动失败 |
| `stopped_by_fault` | 故障或保护停机 |
| `lost_contact` | 最终状态无法确认 |
| `cancelled` | 启动前取消 |

计划触发规则：

- 计划触发本身可以生成 `schedule_triggered`、`schedule_skipped` 等事件。
- 计划中的每一路区域按 `RunRecord` 记录结果。
- 跳过项必须保留跳过原因。
- 删除计划不删除已生成的浇水记录和事件。

## 关键处理流程

### Web 手动顺序运行记录流程

1. 用户确认启动后，生成一条手动顺序运行上下文 `sequence_id`。
2. 每一路实际执行时分别生成 `RunSession`，结束后生成对应 `RunRecord`。
3. 当前水路正常完成后，主控再尝试下一路。
4. 当前水路故障停机、启动失败或状态无法确认时，后续水路不继续执行；已纳入本次序列但未启动的水路记录为 `cancelled` 或 `skipped`，并写明原因。
5. 用户停止序列时，当前水路按从站真实停止原因生成记录，后续未启动水路记录为 `cancelled`。

### 自动浇水暂停记录流程

1. 用户暂停自动浇水时生成 `auto_watering_paused` 事件，影响范围为全局或指定水源。
2. 到期恢复或用户提前恢复时生成 `auto_watering_resumed` 事件。
3. 暂停期间自动计划触发时，受影响水路未启动即跳过，生成 `RunRecord(result=skipped)`，原因指向全局暂停或水源暂停。
4. 暂停不会生成从站停止原因，因为没有启动从站运行。
5. 暂停和恢复通知只是提醒，最终依据仍是事件和运行记录。

### 阻断事件处理流程

1. 阻断条件出现时，生成告警、事件和当前状态；如果发生在运行中，同时结束对应 `RunSession` 并生成 `RunRecord`。
2. 阻断未清除期间，受影响水源禁止手动运行，后续自动计划项未启动即跳过并记录原因。
3. 用户可以确认或清除当前阻断状态，但清除动作不删除事件和运行记录。
4. 清除成功后只解除当前阻断状态，不自动启动运行，也不补跑被跳过的计划项。
5. 清除失败必须保留原状态并记录失败原因，例如条件仍存在、通信未恢复、配置仍未同步或状态无法确认。

### 配置和同步记录流程

1. 保存主控侧配置生成 `config_changed` 事件。
2. 如果影响从站安全配置，对应从站进入待同步状态，并生成配置待同步事件。
3. 用户发起同步后生成 `config_sync` 事件，记录目标从站、配置版本、CRC 和结果。
4. 同步成功解除待同步阻断；同步失败保留待同步状态，并记录失败原因。
5. 配置保存和同步事件不删除历史运行记录，也不改变已有计划历史。

### 校准和学习记录流程

1. 流量计校准和标准流速学习都生成 `calibration` 事件。
2. 校准或学习期间的实际水路运行仍生成水路级 `RunRecord`，触发来源分别为“流量计校准”或“标准流速学习”。
3. 成功事件记录新参数摘要；失败、取消、保护停机或通信丢失事件记录原因。
4. 失败或取消不覆盖原 `pulses_per_liter` 或 `baseline_ml_min`。

### OTA 记录流程

1. 从站 OTA 开始、准备、传输、校验、提交、成功、失败、中断和重试都生成 `firmware_update` 事件。
2. OTA 事件记录目标从站、当前版本、目标版本、固件校验状态、阶段和结果。
3. OTA 期间拒绝运行命令不生成浇水记录；如有计划触发，应按未启动跳过记录原因。
4. OTA 成功或失败不删除主控侧配置、计划、事件或运行记录。

### 主控上电恢复记录流程

1. 主控每次启动后先记录本次启动事实，并保持所有自动运行处于未启动状态。
2. 主控读取上次持久化的 `last_alive_at`、预期维护重启标记和复位原因。
3. 如果上次不是 OTA、用户维护重启或其他预期重启，生成 `power_restored` 系统事件。
4. `power_restored` 事件记录最近确认在线时间、本次恢复在线时间、估算离线持续时间、复位原因、系统时间恢复时间、断电区间说明和通知投递状态。
5. 主控扫描断电区间内本应触发的计划；没有对应运行记录的计划项生成 `schedule_skipped` 或错过执行事件，原因指向 `power_restored`。
6. 如果断电前存在未闭合的运行会话，恢复后生成对应水路级 `RunRecord`，结果按状态无法确认处理，并关联 `power_restored` 事件。
7. 没有备用电源或独立断电检测电路时，断电时间和持续时间只能表达为估算区间，不能写成精确断电时刻。
8. 上电恢复事件本身不生成浇水记录；断电期间错过的计划仍按错过执行规则记录，不补跑。

## 通知记录

第一版需要主动通知：

- 每次灌溉完成结果。
- 保护停机、阻断事件、通信异常、状态无法确认等异常。
- 自动浇水暂停、提前恢复、到期恢复。
- 主控断电后重新上电或恢复在线。

规则：

- 通知只表达事实、影响和建议入口，不作为唯一证据。
- 事件和浇水记录是最终可追溯依据。
- 通知发送成功或失败需要可查，不能静默丢失。
- 上电恢复通知必须包含最近确认在线时间、本次恢复在线时间、估算离线持续时间、错过计划摘要和恢复后自动计划状态。
- 错过计划摘要至少包含错过数量；有错过时显示最早错过计划的名称、时间、水源或水路摘要。
- 恢复后自动计划状态必须明确是已恢复、仍暂停还是暂不可用；暂不可用时必须写明原因。
- 没有硬件断电检测依据时，只能说明断电发生在最近确认在线时间和本次恢复在线时间之间。
- 通知通道实现后续单独裁决；本文只定义产品记录边界。

## 不支持

当前产品不支持：

- 完整手机 App。
- 复杂云端后台。
- 手机端手动启动灌溉、停止当前运行、修改配置、清除故障、发起 OTA。
- 复杂账号审计。
- 用户自定义脚本规则。
- 从站本地长期保存完整运行历史。
- 清除故障时删除历史证据。
