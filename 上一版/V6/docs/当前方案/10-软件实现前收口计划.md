# 10 软件实现前收口计划

## 目标

本文用于在正式写代码前收口软件实现边界。

当前目标不是设计大型软件平台，而是在不重复造基础库能力的前提下，把现有灌溉产品逻辑实现成最终可用形态。

原则：

- 主控基础环境一次性按最终设备能力打开。
- 业务代码只实现灌溉闭环需要的逻辑。
- 暂时用不到的能力可以先空着，但不要写假逻辑、假数据或复杂预留。
- 基础库已经提供的能力，本项目直接使用。
- 基础库缺少通用能力或发现基础库 bug，优先回基础库完善，不在本项目里补丁式绕过。

## 基础库依据

主控使用：

- ESP32 + `Esp32Base`。
- 主控基础库已提供 Web、WiFi、认证、OTA、FS、系统日志、App Config、App Events、RTC 时间、Watchdog、Health 和 RS485 半双工串口基础能力。

分站使用：

- STC8H8K64U + `Stc8hBase`。
- 分站基础库已提供 GPIO、UART、RS485 UART 方向切换、Timer、PWM、ADC、WDT、EEPROM/IAP、EXTI、soft timer、ring buffer、CRC/filter 等基础能力。
- H8K64U 8 路 PWM 基础能力已由基础库示例验证方向覆盖。

当前基础库前置条件：

- 本项目分站 RS485 使用 P3.6/P3.7，也就是 UART1 第二组引脚；`Stc8hBase` 已提供 `STC8H_UART1_PIN_GROUP=1`，项目侧按板级配置启用。

## 主控基础环境

主控建议直接启用完整基础环境：

```ini
ESP32BASE_PROFILE=ESP32BASE_PROFILE_FULL

ESP32BASE_ENABLE_RTC=1
ESP32BASE_RTC_DRIVER=ESP32BASE_RTC_DRIVER_DS3231

ESP32BASE_ENABLE_RS485_PORT=1
ESP32BASE_ENABLE_APP_CONFIG=1
ESP32BASE_ENABLE_APP_EVENTS=1
```

说明：

- `FULL` 启用 WiFi、Web、FS、Watchdog、Health 和 OTA。
- OTA 使用 `Esp32Base` 内置 Web OTA / ArduinoOTA 能力，不在本项目重新设计 OTA 上传流程。
- RTC 使用主控板上的 DS3231，业务时间统一通过 `Esp32BaseTime` 获取。
- RS485 物理层使用 `Esp32BaseRs485Port`。
- App Config 用于系统级标量参数。
- App Events 可以作为近期关键业务事件窗口，但不替代产品完整历史。

## 主控职责

主控负责：

- Web 管理入口。
- 系统参数、水源、水路和计划配置。
- 自动计划调度。
- 手动运行入口。
- 当前运行状态汇总。
- RS485 主站轮询、启动、停止和状态同步。
- 浇水、异常、配置、校准和操作历史。
- 校准流程。
- OTA 基础能力接入。

主控不负责：

- 直接控制灌溉阀。
- 直接采集水源侧流量和低液位。
- 执行从站本地当前任务闭环。
- 重新实现基础库已有的 Web、OTA、RTC、FS、认证和系统诊断能力。

## 主控实现形态

主控采用单 loop / system task 模型，配合 `Esp32Base` 的并发边界。

建议主循环形态：

```text
setup()
  初始化 Wire / RTC 配置
  注册 App Config 字段
  注册业务 Web 页面和 API
  Esp32Base::begin()
  初始化 Esp32BaseRs485Port
  加载业务配置和历史索引

loop()
  Esp32Base::handle()
  IrrigationRuntime::handle()
  Rs485Master::handle()
  Scheduler::handle()
  HistoryStore::handle()
```

要求：

- 不从其他 FreeRTOS task 直接调用 `Esp32BaseConfig`、`Esp32BaseFs`、`Esp32BaseWeb`。
- 不把实时控制放到主控多任务里抢时间。
- 主控负责决策和同步，从站负责已下发当前任务的本地关闭和异常停机。

## 主控业务模块

建议只保留必要模块：

```text
IrrigationModel
  水源、水路、计划、任务、历史的数据结构和枚举

ConfigStore
  系统参数、水源配置、水路配置、计划配置的读写

HistoryStore
  浇水、异常、配置、校准、操作历史

Rs485Protocol
  灌溉帧格式、命令、响应、CRC

Rs485Master
  发现从站、轮询状态、超时、重试、启动任务、停止任务

RuntimeController
  手动任务、计划任务、运行状态、可启动判断、停止原因

Scheduler
  自动计划触发、按水源拆分、跳过原因

CalibrationController
  稳定流量校准、接水量校准

WebPages
  业务页面和 API
```

不建议增加：

- 插件系统。
- 脚本规则引擎。
- 通用任务队列平台。
- 通用 IO 平台。
- 复杂权限体系。
- 云端同步框架。

## 主控存储边界

按数据类型选择基础库能力。

### App Config

用于系统级标量参数：

- 最大单次运行时长。
- RS485 波特率。
- 从站离线判定时间。
- 通信异常判定参数。
- 主阀和水泵默认启停延时。
- 流量保护默认参数。

不用于：

- 水源列表。
- 水路列表。
- 计划步骤。
- 历史记录。
- 校准过程数据。

### Esp32BaseConfig / NVS

用于小型配置对象。

注意：

- namespace 不使用 `eb_` 前缀。
- 单个 blob / POD 最大 256 bytes。
- 多 key 保存不是事务。
- 不要把所有水源、水路或计划打成一个大 blob。

### Esp32BaseFs / LittleFS

用于业务文件：

- 固定容量历史记录文件。
- 计划或对象索引文件。
- 校准结果或过程缓存。
- 需要分页读取的数据。

要求：

- 业务文件放在 `/app/**`、`/data/**` 或项目自定义目录。
- 不写入 `/esp32base/**`。
- 使用 `createFixedFile()`、`readBytesAt()`、`writeBytesAt()` 做固定记录和分页。
- 所有 FS API 必须检查返回值。

### App Events

App Events 只作为近期关键业务事件窗口。

适合写入：

- 计划跳过。
- 保护触发。
- 异常停止。
- 用户停止。
- 关键配置变更摘要。

不适合作为：

- 完整浇水历史。
- 长期统计。
- 流量采样。
- 校准明细。
- 不允许覆盖的业务记录。

完整产品历史仍由业务 `HistoryStore` 保存。

## OTA 边界

主控 OTA 使用 `Esp32Base` 能力。

本项目不做：

- 自定义 OTA 页面。
- 自定义 OTA 上传协议。
- 自定义固件校验流程。
- 差分升级。
- 云端升级系统。

当前只需要保证：

- OTA 期间不启动新的灌溉任务。
- 如果已有从站当前任务，主控重启不影响从站按已下发时长本地到点关闭。
- OTA 成功重启后，主控重新发现从站并同步最近结果。

如果后续启用 `ESP32BASE_OTA_REQUIRE_MARK_VALID=1`，业务只在核心配置、RS485、存储和 Web 初始化通过后调用基础库 mark-valid 接口，不单独设计 OTA 状态机。

分站 OTA 当前不进入灌溉闭环实现范围。

## RS485 协议边界

ESP32 侧使用 `Esp32BaseRs485Port`，STC8H 侧使用 `drv_rs485_uart`。

基础库只负责字节收发和方向控制。本项目负责灌溉协议。

建议采用主控轮询、分站应答模型，避免半双工总线冲突。

建议帧结构：

```text
magic
version
addr
seq
cmd
len
payload
crc16
```

最小命令：

```text
DISCOVER
GET_STATUS
START_TASK
STOP_TASK
GET_LAST_RESULT
GET_INPUTS
PING
```

不做：

- Modbus 寄存器表。
- 通用调试命令。
- 任意 IO 操作。
- 从站主动抢占上报。
- 多主通信。

CRC 算法建议主控和分站使用一致的 CRC16。分站侧优先复用 `util_crc16_modbus()`。

## 分站职责

分站负责：

- 读取拨码地址。
- 接收主控下发的当前任务。
- 按主阀、分支阀、水泵顺序动作。
- PWM 阀全功率启动和维持占空比。
- 本地计时。
- 到点关闭。
- 用户停止命令到达时关闭。
- 低液位、无流量等本地可判断异常停机。
- 电压读数上报。
- 最近任务结果上报。

分站不负责：

- 保存自动计划。
- 保存完整历史。
- 生成 Web 配置模型。
- 做通用 IO 平台。
- 执行脚本规则。
- 独立长期调度灌溉。

## 分站实现形态

分站建议模块：

```text
board
  当前 PCB 引脚、电平、PWM 映射、RS485 DE/RE 控制

protocol
  灌溉 RS485 帧解析和响应生成

task_runner
  当前任务状态机

output_driver
  8 路 PWM 阀、泵干接点、全关安全态

input_monitor
  流量脉冲、低液位输入、电压 ADC

protection
  无流量、低液位、低/过流量动作

status_reporter
  当前状态和最近结果
```

分站主循环必须非阻塞。

不建议：

- 在任务运行中使用长时间 `delay_ms()`。
- 用 EEPROM 保存计划或复杂配置。
- 做运行期动态 IO 映射。
- 把协议和业务状态机写成一个大函数。

## 分站基础库使用

直接使用：

- `drv_rs485_uart`：RS485 半双工 UART。
- `stc8h_pwm`：8 路阀 PWM。
- `stc8h_gpio`：泵输出、输入、状态灯、拨码。
- `stc8h_timer`：1ms tick。
- `stc8h_adc`：电压采样。
- `stc8h_wdt`：看门狗。
- `util_crc`：帧校验。
- `util_ring_buffer`：如需要串口接收缓冲。
- `util_soft_timer`：非阻塞超时判断。
- `util_filter`：简单 ADC 或状态滤波。

暂不使用：

- EEPROM/IAP 保存业务配置。
- nRF24/RF link。
- LCD、红外、TM1637、EC11 等无关驱动。

## PWM 边界

当前从站 PCB 有 `PWM1..PWM8` 到 8 路阀驱动。

实现规则：

- 8 路阀输出使用硬件 PWM。
- 阀打开时先设置 100% duty。
- 全功率启动时间到达后切换到维持 duty。
- 关闭时先 duty 置 0，再 disable 对应通道。
- 主阀和分支阀可以使用不同的全功率时间和维持 duty，但底层 PWM 频率受硬件组限制。

限制：

- `PWMA1..4` 共用 `PWMA` 周期和预分频。
- `PWMB5..8` 共用 `PWMB` 周期和预分频。
- 建议灌溉阀统一使用同一个 PWM 频率，避免用户理解成本和组间差异。

## 状态模型

不要设计一个巨大的系统状态枚举。

主控页面和 API 聚合这些维度：

```text
通信状态：unknown / online / offline
配置状态：complete / incomplete
运行状态：idle / starting / running / stopping
可启动状态：canStart + reason
保护状态：none / warning / blocking / stopped
任务结果：completed / stoppedByUser / stoppedByFault / skipped
```

用户看到的是解释性结论：

- 可以浇水。
- 不能浇水：未发现从站。
- 不能浇水：水源停用。
- 不能浇水：同水源已有任务运行。
- 任务停止：低液位。
- 任务跳过：到点时水源不可用。

## Web 实现边界

Web 使用 `Esp32BaseWeb` 页面壳层、认证、POST 安全检查和基础样式。

业务页面只实现产品页面：

- 总览。
- 运行。
- 计划。
- 水源。
- 水路详情和配置。
- 校准。
- 历史。
- 系统参数。

不做：

- SPA。
- WebSocket。
- 大型前端框架。
- 通用 RS485 调试台。
- 通用 IO 平台。
- 独立统计大屏。

所有危险操作必须使用 POST，并调用 `Esp32BaseWeb::checkPostAllowed()`。

## 实现顺序

不拆产品版本，但编码按依赖推进。

1. 主控和分站工程骨架。
2. 分站板级 pin map 和安全态初始化。
3. RS485 帧协议和 CRC。
4. 分站当前任务状态机。
5. 主控发现、轮询、启动、停止、同步。
6. 主控配置存储。
7. 手动运行闭环。
8. 历史记录。
9. 自动计划调度。
10. 校准流程。
11. Web 页面和 API 完整接入。
12. OTA 场景下的业务启动和同步检查。

## 最小验证

不做大型测试矩阵，只验证关键闭环。

主控：

- FULL profile 编译通过。
- Web/Auth/OTA 页面可访问。
- RTC 时间可用，异常时有诊断状态。
- RS485 port 可发送和读取。
- 配置保存和读取正常。
- 历史文件初始化、写入、分页读取正常。

分站：

- 编译通过。
- 上电所有阀和泵关闭。
- 8 路 PWM 初始化和关闭正常。
- RS485 收发正常。
- 当前任务到点关闭。
- 停止命令关闭。
- 低液位触发关闭。
- 无流量触发关闭。
- WDT 正常喂狗。

系统：

- 主控发现从站。
- 手动启动一条水路。
- 从站到点关闭。
- 主控记录历史。
- 通信中断后从站仍到点关闭。
- 通信恢复后主控同步最近结果。
- OTA 后主控重新发现从站并恢复页面状态。

## 需要继续核实

实现前还需要核实：

- 主控 `485_DIR`、`485_RX`、`485_TX` 对应的 ESP32 GPIO 编号。
- 主控 I2C 引脚和 DS3231 是否与 `Esp32BaseRtc` 初始化方式一致。
- 从站 `PWM1..PWM8` 与 `stc8h_pwm` 通道和 pin select 的一一映射。
- 从站 `IN1..IN4` 对应 MCU 引脚、中断能力和流量脉冲最大频率。
- 从站电压 ADC 通道和换算比例。

如果流量脉冲频率要求超过普通轮询可靠范围，应优先评估在 `Stc8hBase` 中补可复用的脉冲计数基础能力，而不是在本项目里临时写不可复用补丁。

## 当前结论

基础环境可以按最终能力打开。

业务实现必须克制：

- 只做灌溉闭环。
- 不做通用平台。
- 不做复杂预留。
- 不重复基础库能力。
- 缺基础能力先回基础库完善。
