# 02 产品范围与对象模型

## 产品范围

当前产品是本地智能灌溉控制系统。

它要解决：

- 哪些水源可用。
- 哪些水路能浇水。
- 什么时候自动浇水。
- 现在能不能手动浇水。
- 为什么没有浇、为什么停了、哪里需要处理。

它暂时不解决：

- 多用户权限。
- 云端管理。
- 天气联动。
- 通用自动化平台。
- 复杂统计分析。
- 通用设备调试后台。

## 核心对象

### 主控

主控负责：

- Web 管理入口。
- 时间。
- 自动计划调度。
- 运行状态汇总。
- 历史追溯，包括浇水、异常、配置、校准和操作。
- RS485 主站通信。

主控不负责：

- 直接控制灌溉阀。
- 直接采集水源侧流量或低液位。
- 作为一个水源参与运行。

### 从站

从站负责：

- 一个真实水源的执行和输入。
- 阀输出。
- 水泵或自吸泵输出。
- 流量计和低液位输入。
- 状态和异常上报。

从站不负责：

- 保存自动计划。
- 生成复杂记录。
- 执行脚本或条件联动。
- 独立于主控长期运行灌溉逻辑。

### 水源

水源是用户理解的灌溉供水单元。

当前建议：

- 一个从站对应一个水源。
- 水源可启用或停用。
- 水源可以有主阀、水泵或自吸泵、一个流量计、一个低液位输入。
- 同一水源同一时刻只运行一条水路，因此该水源的流量计读数归属于当前运行水路。

### 分支水路

分支水路是真正被浇水的对象。

当前建议：

- 一个分支水路对应一路阀输出。
- 分支水路属于某个水源。
- 自动计划、手动运行和浇水记录都以分支水路为最小单位。
- 主阀不是分支水路。

## 对象关系

```text
主控
  └─ 从站 / 水源
       ├─ 分支水路 1
       ├─ 分支水路 2
       └─ 分支水路 N
```

## 主阀和分支水路

建议规则：

- 不启用主阀时，8 路阀都可作为分支水路。
- 启用主阀时，主阀占用 1 路阀，其余阀作为分支水路。
- 主阀默认使用阀 8。
- 被选为主阀的阀不再出现在分支水路列表。

## 我的建议

- “水源”和“从站”一一对应是当前最重要的简化点。不要设计一个从站多个水源，也不要设计一个水源跨多个从站。
- 水路不要让用户手动选择阀号。阀号应由主阀配置和从站输出顺序推导，否则页面会变复杂。
- 不建议把泵、主阀、流量计、低液位做成独立顶层对象。它们是水源能力，不是用户日常管理对象。
- 从站输入输出要按明确产品用途表达，不要给用户一个“任意输入/任意输出”的配置平台。

## 后续细化

后续页面设计中继续细化：

- 水源和水路列表如何展示。
- 水源详情和水路详情分别承载哪些摘要信息。
- 水源配置和水路配置从详情页如何进入。
