# 07 审查建议与待裁决

## 当前可以先定的方向

用户已确认：

- 主控只做系统中心，不做直接灌溉执行。
- 一个从站对应一个水源。
- 一个水源同一时刻只运行一条水路。
- 同一水源只有一个流量计，当前流量和流速归属于正在运行的水路。
- 不同水源可以并行运行。
- 主控本地阀输出不进入当前产品范围。
- 从站不是通用 IO 平台。
- 从站干接点输出固定用于自吸泵/水泵管理，可启用或停用。
- 从站低液位输入用途固定为低液位传感器，支持常开/常闭，可启用或停用。
- 低流量和过流量默认不停机，只提示；可配置为停机。
- 从站执行当前任务必须带运行时长，并能本地到点关闭输出。
- 从站运行中遇到本地可判断异常时，能停止输出并上报。
- 低液位传感器不是必配；只有启用自吸泵且启用低液位传感器时才参与启动和运行保护。
- 流量计输入、低液位输入在对应水源/从站的 `IN1..IN4` 中选择，放在水源配置中，不做通用 IO。
- 校准保留稳定流量校准和接水量换算每升脉冲数两种轻量能力；标准流量归属水路，每升脉冲数归属水源流量计。
- 先用统一历史页面表达浇水、异常、配置、校准和操作。
- 主阀默认阀 8，允许在水源配置中改阀号。
- 自动计划只保留“每天”和“自定义周期”两个入口。
- 错过计划不补跑。
- Web 当前建议 6 个顶层页面：总览、运行、计划、水源、历史、系统参数。
- 桌面 Web 管理界面优先，移动端适配后置。

继续设计但不硬定：

- 配置页面如何组织，系统、水源、水路是当前自然归属，但页面可以继续调整。
- LCD2004、本地按键、OTA、天气、统计独立页后置，不进入当前主流程。

## 建议推翻或暂缓的旧想法

建议推翻：

- 主控也作为阀控设备参与灌溉。
- 把从站输入输出做成通用 IO 平台。
- 把干接点做成通用继电器。
- 把历史做成底层日志。
- 把所有配置塞进一个设置页。

建议暂缓：

- LCD 菜单。
- 主控和从站本地按键业务。
- OTA 页面。
- 天气联动。
- 独立统计页。
- 通知推送。
- 多用户权限。
- 目标水量自动停止。
- 复杂安全停机状态机。
- 安全停机待同步。

## 关键取舍

### 校准怎么做才不复杂

校准不应被设计复杂。

它可以只是一个工具流程：

- 打开指定水路出水。
- 读取流量计脉冲和流速。
- 等滑动窗口稳定后，保存稳定流量作为水路标准流量。
- 或者用户接固定水量，输入水量后换算每升脉冲数。

我的建议：保留校准，但只做轻量工具，不做复杂向导。标准流量和每升脉冲数要分清楚，前者属于水路并用于低/过流量判断，后者属于水源流量计并用于估算水量。

### 历史是否需要拆分

当前先不拆。

判断标准：

- 如果用户首先关心“浇了什么”，记录应独立。
- 如果用户首先关心“发生过什么”，可以做统一历史列表。
- 如果异常解释太多，事件独立会更清楚。
- 如果历史变成流水账，就应该弱化低价值信息。

我的建议：当前方案先用统一历史页面，通过类型筛选解决信息混杂。只有历史页确实变乱时，再拆成记录和事件。

### 通信参数是否展示

方案 A：展示在系统参数。

- 优点：个人项目可调试、可维护。
- 缺点：页面看起来更技术。

方案 B：隐藏为实现默认值。

- 优点：页面更短。
- 缺点：现场通信问题时缺少入口。

我的建议：展示，但只展示必要项和默认值，不做调试后台。

### 主阀默认阀号

方案 A：默认阀 8，可修改。

- 优点：规则明确，兼容安装差异。
- 缺点：需要处理主阀占用分支阀的页面逻辑。

方案 B：固定阀 8。

- 优点：更简单。
- 缺点：安装灵活性差。

我的建议：默认阀 8，可修改；但水路不直接编辑阀号。
