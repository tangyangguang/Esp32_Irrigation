# 08 验证验收清单

## 验收范围

| 类别 | 目标 |
|---|---|
| 基础库能力 | 确认 `Esp32Base` 和 `Stc8hBase` 能支撑所需能力 |
| 单板硬件 | 确认主控板、从站板启用资源可用 |
| 通信协议 | 确认 RS485 帧、CRC、超时、重试和多地址管理正确 |
| 配置同步 | 确认版本、CRC、配置拒绝和重新同步正确 |
| 运行控制 | 确认手动运行、计划运行、停止和状态跟踪正确 |
| 安全保护 | 确认低液位、无水、高流量、通信丢失、运行超时保护可靠 |
| 电压监测 | 确认电压显示、校准、告警和记录可靠，且不误触发停机 |
| Web 交互 | 确认 Web 覆盖所有业务操作 |
| 手机通知与轻操作 | 确认通知、关键状态和暂停/恢复自动浇水可用，且不越界成完整远程控制台 |
| 本地基础控制 | 确认 LCD2004 + 主控按键可完成现场基础查看、停止、手动启动、暂停/恢复和清除可恢复故障 |
| 天气显示 | 确认天气预报和建议可见，且不参与自动控制 |
| OTA 维护 | 确认 ESP32 OTA 和主控转发从站 RS485 OTA 的边界清楚 |
| 长时间运行 | 确认稳定性、记录、恢复和无误动作 |

## 高保真原型前评审清单

进入高保真原型前，必须先完成以下评审。未通过时回到对应文档修正，不直接改 HTML、不直接进入代码实施。

资料权威：

- `pcb/` 下 4 个硬件文件存在，并作为硬件能力边界依据。
- `docs/plans/01..09` 是当前结果性方案依据。
- 旧代码、旧文档、旧 prototype 和 `legacy` 只作为背景，不覆盖当前方案。

产品边界：

- 一个从站对应一个真实水源。
- 一个从站最多 8 路阀输出、最多 1 个流量计、最多 1 个主阀。
- 从站只做固定硬件抽象、本地安全、RS485 业务命令和 OTA 接收。
- Web/API 使用固定路径和 query 参数，不引入 REST 路径参数、请求体 JSON、通用透传或脚本规则。
- 第一版不引入多主控、从站主动上报、一个从站多个流量计、同站多水路并行或 Web 任意 RS485 透传。

页面职责：

- 总览只回答系统现在是否可用、是否有必须处理的问题。
- 运行只回答本次要浇哪个水源、哪些水路、多久、是否可启动。
- 计划只回答未来什么时候自动浇水。
- 记录只回答过去某一路实际发生了什么。
- 事件只回答系统发生了什么、影响什么、是否需要处理。
- 统计只回答正常浇水的水量和时长趋势。
- 设置组只回答真实水源如何装配、分支水路如何配置、校准如何完成。
- 维护和诊断详情只回答现场服务、同步、OTA 和底层状态证据查看。
- 天气只显示天气和建议，不自动控制灌溉。

关键流程：

- 手动运行、暂停/恢复自动浇水、创建计划、处理阻断、查看记录、配置新从站、同步配置、流量校准、标准流速学习和 OTA 都有明确入口、准入、执行中、结果和失败处理。
- 暂停自动浇水只影响自动计划，不停止当前运行，不影响手动运行，不补跑。
- 清除阻断状态不删除事件和记录，不自动启动运行，不补跑跳过计划。
- 运行、计划、记录和事件之间的对象关系一致：一条运行记录对应一个水源下某一路水路的一次运行或运行尝试。

低保真和信息结构：

- 每个页面先满足 `09-Web界面与交互原型.md` 的低保真结构规则和文字线框。
- 页面首屏只放判断和主任务必需信息，历史证据和维护细节下移。
- 普通用户页面不使用工程后台总表；维护诊断页面允许表格，但必须紧凑分组。
- 单水源和多水源都能自然呈现，不需要用户在正式产品中手动切换规模模式。

视觉和组件：

- 状态语言、动作层级、状态摘要条、水源卡片、水路行、顺序运行编辑器、保护摘要、步骤流程条、证据列表和高影响确认框遵守 `09` 的组件规则。
- 高保真必须覆盖正常、运行中、待同步、保护不完整、阻断、离线、状态无法确认和空状态。
- 危险动作必须二次确认，确认文案说明影响对象和后果，不能只写“确定”。
- 颜色只能辅助理解，不能作为唯一信息。

高保真原型必须覆盖的页面范围：

- 总览、运行、事件：验证日常使用、手动运行和故障处理是否成立。
- 计划、记录、统计：验证自动计划、历史证据和统计表达。
- 设置、水源配置、水路配置：验证安装配置、校准学习和设置组信息层级。
- 维护、诊断详情、天气、手机轻入口和 LCD2004：验证辅助入口和工程入口不越界。
- 当前唯一 Web 原型入口是 `docs/prototypes/web/final-prototype.html`。

禁止进入：

- 不直接在旧 HTML 上继续打补丁。
- 不为高保真临时引入未在 `docs/plans/` 确认的新功能。
- 不把页面做成营销页、工程后台总表、通用 IO 控制台或通用 RS485 透传工具。
- 不在高保真阶段决定正式工程目录、类名、存储介质实现、分区表或基础库 API 修改。

需要用户裁决：

- 第一版产品范围是否变化。
- 高保真视觉方向明显偏家用简洁还是偏工程紧凑。
- Figma 和 HTML 原型谁作为下一阶段主工具。默认推荐 Figma 作为视觉权威，HTML 作为后续可点击验证原型。
- 手机通知通道和实际可用平台约束。

## 硬件启用验证

主控板：

- 12V 输入、5V、3.3V 正常。
- ESP32 启动和复位正常。
- RS485 发送、接收、方向控制正常。
- LCD2004 可显示。
- SHT30 可读取。
- DS3231 可读写时间。
- 主控按键和两个状态灯可用，主控按键只触发主控侧受控业务动作。
- ADS1115、FRAM、主控侧阀输出、干接点、预留输入不被初始化为业务功能。

从站板：

- STC8H 启动和复位后全部输出关闭。
- RS485 通信正常。
- 拨码地址读取正确。
- `IN1..IN4` 可按配置识别。
- 8 路阀输出可单独控制。
- `DRV_SD` 安全关闭有效。
- 干接点输出可控制外接泵继电器。
- 系统电压 ADC 可读取并校准。
- 状态灯和维护按键可用。
- 预留 I2C、预留 ADC、`SPARE_*` 不进入业务逻辑。

## 配置验证

必须验证：

- 参数默认值、最小值、最大值、单位和字段关系符合 `02-配置模型与参数边界.md`。
- 一个从站必须有且只有一个流量计输入。
- 低液位最多一个，可关闭。
- 自吸泵可关闭。
- 主阀最多一个，可关闭。
- 至少一路分支水路启用。
- 同一路输入不能同时配置为流量计和低液位。
- 主阀不能同时作为分支水路。
- 配置版本和 CRC 可校验。
- 配置未同步时禁止相关从站运行。
- 保存影响安全配置的内容后，对应从站显示待同步。
- 同步成功必须验证从站配置版本和 CRC 一致后才解除待同步。
- 同步失败必须保留待同步状态并显示失败原因。
- 增强能力未校准或未学习不阻断运行，只提示保护不完整。
- 非法参数不能保存，不能下发到从站，也不能被运行命令绕过。
- `pulses_per_liter=0`、`baseline_ml_min=0` 能作为未校准或未学习状态保存，且不阻断普通灌溉。

## 通信验证

必须验证：

- `PING`、`GET_INFO`、`GET_STATUS`、`GET_CONFIG_SUMMARY`。
- `SET_SAFE_CONFIG`、`GET_SAFE_CONFIG`。
- `START_ZONE`、`STOP`、`CLEAR_FAULT`。
- `GET_STATUS` 能返回状态、运行 ID、活动水路、停止原因、故障位、输入/输出位图、脉冲计数和电压状态。
- CRC 错误不执行动作。
- `SEQ` 不匹配不被当成有效应答。
- 地址不匹配不被当成有效应答。
- 超时重试有效。
- 多从站轮询稳定。
- 疑似地址冲突能被标记并禁止运行。
- 普通灌溉运行的 `START_ZONE` 不允许关闭无水保护。
- `STOP` 对 `run_id=0` 和重复停止请求保持幂等。
- `CLEAR_FAULT.flags` 未定义位非 0 时必须拒绝。

## 运行验证

手动运行：

- Web 手动运行可在一个水源内选择多个分支水路，并按确认顺序逐路运行。
- Web 手动顺序运行不保存为计划、不修改水路配置、不改变计划配置。
- Web 手动顺序运行中，当前水路故障停机、启动失败或状态无法确认时，后续水路不继续执行。
- 单水路启动、运行、停止。
- Web 手动运行和主控 LCD2004 + 本地按键手动运行都走同一启动准入。
- 本地手动运行启动前可微调本次时长，且不修改水路配置或计划。
- 其他从站运行中仍可启动当前从站。
- 同一从站已有运行时拒绝第二个运行。
- 用户停止和本地保护停机的停止原因不混淆。

计划运行：

- 计划创建和编辑只保存主控侧计划，不下发从站。
- 计划只能加入已启用且未被锁定或阻断的分支水路。
- 删除计划必须二次确认，且不删除历史运行记录和事件。
- 同一从站多水路按顺序运行。
- 多从站可并行运行。
- 同一从站冲突时后触发计划项跳过并记录。
- 全局暂停自动浇水时，所有水源自动计划项未启动即跳过并记录暂停原因。
- 按水源暂停自动浇水时，暂停水源跳过，未暂停水源照常执行。
- 暂停到期自动恢复，提前恢复有效。
- 暂停自动浇水不影响手动运行。
- 暂停自动浇水不停止已经运行中的水路。
- 全部跳过记录为 `skipped`，不记录为 `failed`。
- 暂停恢复后不补跑暂停期间跳过的计划项。
- 错过执行不补跑，只记录事件。
- 主控断电后重新上电不补跑错过计划，并生成上电恢复事件和断电恢复摘要。
- 断电恢复摘要能识别断电区间内本应触发但没有运行记录的计划项，并生成错过事件。

## 安全验证

必须验证：

- 上电、复位、看门狗复位后所有输出关闭。
- 主控上电恢复后只记录事件、通知和必要的错过计划摘要，不自动启动水路。
- 如果断电前存在未闭合运行，恢复后不能写成正常完成，必须显示为状态无法确认或疑似断电中断。
- 通信超时后从站本地安全停机。
- 运行超时后从站本地安全停机。
- 低液位触发后停机。
- 启动无水停机。
- 运行中无水停机。
- 标准流速已学习且高流量确认后停机。
- 电压低/高告警只提示和记录，不自动停机、不禁止启动。
- 从站进入 `OFFLINE_LOCAL_SAFE` 后不自动恢复运行。
- 清除故障后不自动启动。

## 流量验证

必须验证：

- 流量计脉冲计数。
- 计量校准流程。
- 校准失败、取消、保护停机或通信丢失时不覆盖原 `pulses_per_liter`。
- 计量未校准时允许运行并提示水量估算不可用或仅供参考。
- 标准流速学习。
- 标准流速学习失败、取消、保护停机或通信丢失时不覆盖原 `baseline_ml_min`。
- 标准流速未学习时允许运行并提示低/高流量保护未启用。
- 无水保护不依赖标准流速。
- `low_flow_action=alarm_only` 只告警不从站停机，`low_flow_action=stop` 按配置停机。
- `high_flow_action=stop` 默认停机并锁定，`high_flow_action=disabled` 不生成阻断性故障。
- `standby_leak_enabled=1` 时待机漏水可生成告警并按 `standby_leak_action` 禁止该从站新运行。

## Web 验证

必须验证：

- 低保真原型能说明每个页面的对象范围、当前状态、主任务和不能执行时的下一步。
- 总览、运行、计划、记录、事件、配置、维护页面分别使用对应结构模板，不混成一个大后台表格。
- 页面首屏优先级符合 `09-Web界面与交互原型.md` 的低保真结构规则。
- 总览、运行、计划、记录、事件、设置组、维护和诊断详情都有文字线框说明。
- 低保真评审按总览/运行/事件、计划/记录/统计、配置/维护/诊断、天气/手机/LCD 的顺序推进。
- 状态摘要条、阻断事件条、水源卡片、水路行、顺序运行编辑器、保护摘要、步骤流程条、紧凑筛选条和高影响确认框有一致使用规则。
- 低保真阶段不决定最终颜色、图标、字体、动效、组件代码或前端技术栈。
- 高保真原型必须遵守颜色语义：绿色正常、蓝色信息、琥珀需注意、红色阻断、灰色离线或禁用。
- 高保真原型必须区分主动作、次动作、危险动作和禁用动作；危险动作必须二次确认。
- 高保真原型必须区分保护不完整和设备故障，不能把未校准、未学习画成故障。
- 高保真原型必须验证水源卡片、水路行、顺序运行编辑器、保护摘要、步骤流程条、证据列表和高影响确认框的一致性。
- 高保真原型不得把首页做成工程后台总表，不得把设置组做成运行控制台，不得把天气页做成自动控制策略页。
- 移动端必须按状态摘要、主要动作、详情配置、历史证据纵向排列；运行页停止按钮必须容易触达。
- LCD2004 显示必须只覆盖现场基础控制，不承载完整配置、计划编辑、校准或 OTA。
- 首页能显示系统、从站、运行和告警摘要。
- 手动运行能启动、监控、停止。
- 从站页面能同步配置、识别从站、清除故障。
- 水路页面能配置 8 路输出、主阀和分支水路。
- 输入输出页面只覆盖 `IN1..IN4`、自吸泵和电压监测。
- 流量页面能完成校准和标准流速学习。
- 计划页面能创建、编辑、启停和查看结果。
- 新水源/从站配置流程能从水源信息、区域配置、保护与流量配置进入待同步，并在同步成功后解除阻断。
- 记录页面只显示水路级浇水记录；手动、计划和维护作为触发来源字段。
- 记录页面不显示统计卡片，筛选控件贴近记录表格。
- 浇水记录详情能查看停止原因、流量摘要、启用/未启用保护和关联事件。
- 计划触发多个区域时，每个区域各生成一条水路级记录；不在记录页显示计划整体汇总表。
- 统计页面按水源和区域平铺显示每日估算水量和实际浇水时长。
- 统计页面顶部只显示正常浇水相关的紧凑摘要，不把保护停机、阻断告警、未启动作为核心统计信息。
- 事件页面能查看告警、配置、维护、系统和 OTA 事件，并能处理阻断事件状态。
- 事件页面只在事件表格上方显示紧凑处理提醒和筛选控件，不显示最近 OTA 成功等无处理意义摘要。
- 事件详情能查看触发条件、影响范围、处理建议、关联记录和清除状态动作。
- 清除阻断状态不删除事件和运行记录，不自动启动运行，不补跑已跳过计划项。
- 天气页面能显示当前位置、当前天气、未来预报、更新时间和建议。
- 天气设置从天气页进入独立表单页，且只影响天气显示和建议。
- Web API 能保存系统、从站、水路、输入、流量、自吸泵和计划配置。
- 配置类 API 的校验结果和 Web 页面一致。
- 从站安全配置修改后能正确标记待同步。
- Web API 使用固定路径和 query 参数，不使用 REST 路径参数或请求体 JSON。
- Web 不提供通用 IO、通用寄存器、任意 RS485 帧透传。
- Web API 的 `op` 名称全文一致，不混用横线命名和下划线命名。
- `sync_station_config` 下发后必须读取配置摘要确认版本和 CRC。
- `calibrate_flow`、`learn_baseline`、`start_station_ota` 都必须拒绝从站运行中、离线、未同步、故障或锁定等不满足准入的状态。

## 天气验证

必须验证：

- 首页不显示天气摘要、天气建议或天气设置。
- 天气页面能显示当前天气、未来 3 天预报、降雨概率、预计降雨量、风力和更新时间。
- 天气页面能显示当前设置摘要，修改设置进入独立表单页。
- 天气建议明确标注“仅供参考，不自动控制灌溉”。
- 天气数据不可用或过期时只显示数据状态，不生成阻断性告警。
- 天气不禁止手动运行。
- 天气不自动跳过计划。
- 天气不自动延迟计划。
- 天气不自动调整运行时长。
- 天气不作为从站安全配置下发。
- 天气建议可以引导用户手动暂停自动浇水，但不能自动执行暂停。
- 天气 API 使用固定路径和 query 参数，不使用 REST 路径参数、请求体 JSON、`PUT` 或 `DELETE`。

## 手机通知与轻操作验证

必须验证：

- 每次灌溉完成后生成通知。
- 保护停机、阻断事件、通信异常和状态无法确认能生成异常通知。
- 暂停自动浇水、提前恢复和到期恢复能生成通知。
- 主控断电后重新上电或恢复在线能生成通知，通知包含最近确认在线时间、本次恢复在线时间、估算离线持续时间、错过计划摘要和恢复后自动计划状态。
- 手机轻入口可查看当前运行、自动浇水暂停状态、最近一次灌溉结果和阻断事件摘要。
- 手机轻入口可暂停和恢复自动浇水。
- 手机轻入口不能手动启动灌溉、停止当前运行、修改计划或配置、清除故障、发起 OTA。
- 通知发送成功或失败可查，不静默丢失。
- 没有备用电源或独立断电检测电路时，通知和事件不得声称精确断电时刻或精确断电持续时长。

## 本地基础控制验证

必须验证：

- LCD2004 显示当前时间、主控状态、RS485 摘要、当前运行、剩余时间、暂停状态、未处理故障数量和最短故障提示。
- 主控按键可停止当前运行。
- 主控按键可选择已启用水源和水路，发起一次手动运行。
- 主控按键可在启动前微调本次时长。
- 主控按键可暂停/恢复自动浇水。
- 主控按键可清除可恢复故障。
- 主控按键不能创建或修改计划，不能修改安全配置，不能执行 OTA、校准或清空从站配置。
- 从站本地按键不能启动灌溉。

## OTA 验证

必须验证：

- ESP32 主控 OTA 不改变从站业务协议边界。
- 从站 OTA 只针对 `STC8H8K64U-45I-LQFP48`，不宣称支持 STC8H 全系列或 STC8H1K08。
- 从站 OTA 只能在维护状态下发起。
- 从站 OTA 过程中全部输出保持关闭。
- 从站 OTA 过程中拒绝启动灌溉运行。
- 从站 OTA 成功、失败和中断都能生成事件记录。
- 从站 OTA 失败或中断后不自动启动灌溉，不清空主控侧配置、计划、事件或运行记录。
- 主控在发送 OTA 前完成固件下载、签名/hash 校验和完整固件暂存。
- 业务协议只负责可靠传输 manifest/chunk，不私有实现 STC Flash/IAP 分块写入逻辑。
- OTA chunk 必须顺序写入，整包 CRC32 校验通过后才能 commit。
- 新应用启动早期必须先初始化输出为安全态，自检通过后再标记 app valid。
- 升级失败后进入 bootloader 等待重试，不要求继续运行旧应用。
- 从站 OTA 基础能力来自 `Stc8hBase`，业务项目不私有实现 IAP 擦写、commit、app valid、recovery/retry 或 bootloader 保护。
- RS485 OTA 验证覆盖地址选择、总线冲突避免、业务命令暂停、超时重试和 ACK/NAK/状态返回。

## 长时间运行验证

第一版交付前至少完成：

- 24 小时空闲轮询。
- 24 小时单从站周期运行。
- 24 小时多从站并行计划运行。

失败条件：

- 主循环卡死。
- 误动作打开输出。
- 配置丢失。
- 记录写入失控。
- 通信恢复后自动误启动。
- 清除故障后自动误启动。
