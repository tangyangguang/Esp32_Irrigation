# 灌溉系统核心逻辑说明

本文档说明当前灌溉系统的核心业务逻辑，重点覆盖计划管理、计划执行、浇水会话状态机、安全告警和页面/API 的业务边界。文档用于后续继续讨论核心设计，不替代代码级接口文档。

最后更新：2026-05-27

## 一、系统定位和边界

本项目是基于 ESP32 的两路智能灌溉控制器。系统默认按两路水设计：第 1 路和第 2 路默认都启用；如果现场只接一路水，应在“灌溉设置”中关闭不用的那一路，而不是把系统理解成单路产品。

应用层只负责灌溉业务：

- 水路启用状态、默认模式、每路默认时长和流量校准。
- 手动浇水、计划浇水、停止和紧急停止。
- 计划配置、近期计划、单次跳过和历史记录。
- 浇水记录、业务事件、水流异常和待机漏水告警。

基础能力由 Esp32Base 承担，包括 WiFi、NTP、mDNS、Web 底座、OTA、系统日志、系统状态、文件系统和重启等。灌溉业务不复制这些基础能力。

## 二、核心对象

### 1. 水路

系统固定支持最多两路水。每一路有独立的启用状态、名称、默认时长、流量计脉冲当量和校准系数。

水路启用状态的业务含义是“这一路当前可用于浇水”。停用水路仍属于系统结构的一部分，页面仍按两路展示，但停用水路不能被手动启动，也不会被计划实际执行。

### 2. 计划

系统最多保存 8 条计划。每条计划包含：

- 是否启用。
- 一天中的执行时间，精度为 1 分钟。
- 第 1 路目标时长和第 2 路目标时长。
- 执行模式：同时浇水或顺序浇水。
- 循环天数。
- 周期内执行日掩码。
- 循环开始日期。
- 最近一次已处理日期。

默认计划是停用状态，执行时间为 07:00，两路默认各 5 分钟，同时浇水，1 天循环并在第 1 天执行。

### 3. 单次跳过

单次跳过按“计划编号 + 日期”保存。它只影响某一天的某一条计划，不修改计划定义本身。

系统最多保存 32 条跳过记录。加载或写入跳过记录时，会清理早于今天的过期记录。

### 4. 浇水会话

所有手动浇水和计划浇水都会进入同一种浇水会话。系统同一时间最多只有一个浇水会话。

会话包含：

- 会话是否活动。
- 触发来源：本地按键、Web/API、计划或未知来源。
- 执行模式：同时或顺序。
- 每一路的目标时长、当前状态、开始时间、结束时间、开始脉冲数和最近脉冲数。
- 最近一次结束原因。

每一路的状态包括：空闲、等待、运行、完成、已停止、异常。

会话结束原因包括：无、完成、手动停止、紧急停止、被新会话替换、异常、跳过、部分异常。

## 三、初始化流程

系统启动后，灌溉业务按固定顺序初始化：

1. 加载灌溉设置。
2. 加载计划。
3. 加载单次跳过记录。
4. 初始化浇水记录和业务事件。
5. 初始化流量计。
6. 初始化安全管理和阀门控制，并确保阀门关闭。
7. 初始化浇水会话。
8. 初始化计划调度器。
9. 初始化漏水监控。
10. 初始化维护服务。
11. 注册业务 Web 页面和 API。
12. 订阅基础库事件并写入启动事件。

安全原则是：启动早期先关闭所有阀门，再进入业务逻辑；任何异常路径都不应让阀门保持开启。

## 四、计划管理逻辑

### 1. 计划保存

计划保存时会进行基础校验：

- 计划编号必须在 0 到 7 范围内。
- 执行时间必须在 00:00 到 23:59 范围内。
- 单路时长必须在 0 到 14400 秒范围内。
- 执行模式只能是同时或顺序。
- 循环天数必须在 1 到 30 天范围内。
- 循环执行日不能为空，且不能超过循环天数范围。
- 循环开始日期必须是有效日期。
- 如果计划启用，至少要有一路配置了大于 0 的目标时长。

页面保存计划时，会根据当前水路启用状态清理停用水路的时长。也就是说，某一路已经在灌溉设置中停用时，即使旧计划中曾保存过该路时长，保存计划后该路时长也会变为 0。

### 2. 周期规则

计划不使用星期、间隔和临时规则的多套模型，而统一使用“循环天数 + 周期内执行日”模型。

示例：

- 每天执行：循环天数为 1，执行第 1 天。
- 浇 2 天停 1 天：循环天数为 3，执行第 1 天和第 2 天。
- 每周固定执行：循环天数为 7，选择周期内对应的执行日。

判断某天是否执行时，系统会从循环开始日期计算目标日期是周期内第几天，再检查执行日掩码。

### 3. 近期计划

近期计划页面只展示今天、明天、后天。它的定位是面向未来的执行控制，尤其是让用户按单次跳过未来未执行计划。

近期计划展示的是“当前有效执行内容”，不是计划原始字段。有效执行内容等于：

```text
计划中配置的每路时长 ∩ 当前已启用水路
```

如果某条计划配置了第 2 路，但第 2 路后来被停用，近期计划不会显示第 2 路会执行。如果两路都不可执行，该计划显示为不可执行，并且不提供跳过按钮。

### 4. 历史记录

历史记录页面只展示已经发生过的浇水会话事实。历史记录不按当前水路启用状态重新解释，因为历史记录保存的是当时的执行快照。

## 五、计划执行逻辑

### 1. 调度前提

计划调度依赖 NTP 时间。如果时间尚未同步，计划调度器不触发任何计划。

调度器以分钟为粒度运行。同一分钟内不会重复处理同一条计划，系统还会记录当前已处理的日期和分钟，防止时间回调或循环重复导致重复触发。

### 2. 触发条件

一条计划要被触发，必须同时满足：

- 计划已启用。
- 当前分钟等于计划执行分钟。
- 计划当天尚未标记为已处理。
- 该计划当天没有被单次跳过。
- 同一天同一分钟没有被调度器处理过。
- 目标日期符合该计划的循环规则。

### 3. 有效执行内容

计划触发时，系统先计算有效执行内容：

```text
第 1 路有效时长 = 第 1 路已启用 ? 计划第 1 路时长 : 0
第 2 路有效时长 = 第 2 路已启用 ? 计划第 2 路时长 : 0
```

如果没有任何一路有效，系统写入“计划无启用水路”的跳过事件，并把该计划标记为当天已处理。这样可以避免无效计划在当天形成不清晰的待执行状态。

### 4. 与现有会话冲突

如果计划触发时已经存在活动浇水会话，当前策略是跳过本次计划，不等待、不排队、不叠加、不延长。

跳过后系统会写入事件，并把该计划标记为当天已处理。

这个策略简单、可预测，适合嵌入式控制器。它牺牲了自动补偿执行，但避免了多个浇水任务叠加导致不可控的阀门开启时长。

### 5. 启动计划会话

如果没有活动会话且有效执行内容不为空，计划调度器会调用统一的浇水会话启动逻辑，并把来源标记为计划。

启动成功后，计划被标记为当天已处理，并写入浇水开始事件。

如果启动被拒绝，系统会写入“计划被拒绝”的跳过事件。当前实现同一分钟不会重复触发该计划；该情况通常代表配置或运行时状态出现了不应该发生的异常，需要通过事件和日志继续定位。

## 六、手动浇水逻辑

手动浇水可以来自 Web/API 或本地按键。

Web/API 手动启动时，默认参与水路来自当前启用状态：两路都启用时默认两路参与；如果某一路被关闭，则默认不参与。请求也可以显式选择只浇其中一路。

本地按键启动时，系统使用每路默认时长，并只启动当前已启用的水路。

手动启动的校验规则：

- 至少有一路目标时长大于 0。
- 参与水路必须已启用。
- 参与水路的目标时长必须在 1 到 14400 秒之间。
- 执行模式必须是同时或顺序。

如果启动手动浇水时已有活动会话，旧会话会以“被新会话替换”结束，然后启动新会话。

## 七、浇水会话状态机

### 1. 会话级状态

会话级状态可以概括为：

```text
空闲 -> 活动 -> 结束 -> 空闲
```

活动会话只有一个。无论来源是手动还是计划，都会进入同一个状态机。

### 2. 路级状态

每一路在会话中的状态流转为：

```text
空闲
等待
运行
完成 / 已停止 / 异常
```

目标时长为 0 的路保持空闲。目标时长大于 0 的路先进入等待。

同时模式下，所有等待路立即进入运行。顺序模式下，系统按路号从小到大启动；当前运行路结束后，再启动下一条等待路。

### 3. 正常完成

运行中的水路到达目标时长后，该路关闭阀门并进入完成状态。

当没有任何水路处于等待或运行状态时，会话结束。若没有任何路异常，会话结束原因是完成。

### 4. 单路停止

单路停止只影响指定水路。该路如果正在运行或等待，会进入已停止状态。

如果停止后没有任何水路还在等待或运行，整个会话结束，会话结束原因使用本次停止原因。

### 5. 全部停止

全部停止会立即关闭所有阀门，并把所有等待或运行的水路标记为已停止。

Web 的停止全部、本地停止全部和紧急停止都会走统一的全部停止路径，但结束原因不同。

### 6. 水流异常

运行中的水路会持续检查流量计脉冲。如果超过配置的无脉冲超时时间仍没有新的脉冲，该路判定为水流异常，系统关闭该路阀门并写入异常事件。

为了避免刚开阀时管路尚未响应导致误判，系统存在启动缓冲：如果配置的无脉冲超时时间小于 3 秒，首次无脉冲判断仍至少等待 3 秒。

多路运行时，某一路水流异常不会直接停止另一条正常水路。会话结束时，如果有异常路也有完成路，结束原因是部分异常；如果只有异常没有完成，结束原因是异常。

## 八、系统状态机

从业务角度，系统状态可以分为以下几类：

### 1. 启动初始化

系统加载配置和记录，初始化阀门、流量计、计划调度器和 Web/API。启动期间的安全要求是阀门关闭。

### 2. 空闲

没有活动浇水会话。系统持续处理按键、Web/API、计划调度和漏水监控。

### 3. 手动浇水中

由 Web/API 或本地按键触发。手动浇水优先级高于计划浇水；新的手动浇水会替换已有会话。

### 4. 计划浇水中

由计划调度器触发。计划浇水与手动浇水共用同一套会话状态机。

### 5. 当前告警

当前告警主要来自待机漏水或疑似阀门粘连。出现告警后，首页显示当前告警状态。用户确认现场处理后，可以清除告警提示。

### 6. 恢复出厂待处理

GPIO0/BOOT 长按会进入恢复出厂请求状态。该状态下，配置保存、手动浇水、停止、清除告警、计划保存和计划跳过接口都会拒绝新的持久化或控制操作，避免恢复过程中出现状态竞争。

## 九、安全和异常策略

系统安全策略优先级高于便利性：

- 启动和初始化必须关阀。
- 紧急停止优先于所有业务状态。
- 任意浇水都必须有目标时长，不允许永久开启。
- 已停用水路不能被启动。
- 计划冲突时跳过，不排队。
- 水流异常按路处理，不让异常路继续开阀。
- 待机漏水告警会关闭全部阀门，并要求用户处理后清除告警。
- 设备级日志、系统状态、OTA 和基础维护交给 Esp32Base。

## 十、页面和 API 边界

### 1. 首页

首页负责当前状态和即时操作：

- 设备状态。
- 当前浇水状态。
- 手动浇水。
- 当前告警。

### 2. 近期计划

近期计划负责今天、明天、后天的未来执行视图和单次跳过。它不是历史页面。

### 3. 历史记录

历史记录只看已发生的浇水业务记录，不展示系统事件。

### 4. 计划配置

计划配置只维护计划定义。执行结果应去近期计划和历史记录查看。

### 5. 灌溉设置

灌溉设置维护水路、默认模式、默认时长、流量计量和检测阈值。恢复出厂、系统日志、OTA、认证等设备级能力不放在业务设置中。

## 十一、专业审视后的优化建议

以下是基于当前实现继续审视后，建议后续重点讨论的点。这些不是当前已经全部实现的行为，而是下一轮核心逻辑优化方向。

### 1. 抽出统一的有效计划模型

当前调度器和 Web 页面都各自计算“有效执行内容”。逻辑已经一致，但从长期维护看，建议后续抽成统一的业务函数，例如“根据计划和当前设置生成有效计划视图”。

这样可以避免未来页面、API 和调度器对停用水路、不可执行计划、展示文案产生分歧。

### 2. 明确计划触发结果类型

当前计划触发结果通过事件文本区分，例如“已有会话跳过”“无启用水路”“启动被拒绝”。后续可以沉淀成明确枚举：

- 已启动。
- 因已有会话跳过。
- 因无有效水路跳过。
- 因参数或状态异常拒绝。

这样近期计划、事件和诊断 API 可以显示更稳定的状态，而不是依赖文本。

### 3. 讨论计划启动被拒绝后的处理策略

当前实现中，计划启动被拒绝会写事件，但不会像“已有会话”或“无启用水路”那样明确标记当天已处理。由于调度器同一分钟有防重复保护，通常不会造成同一分钟反复启动，但页面语义可能仍不够清晰。

建议后续讨论：任何计划触发尝试，无论启动成功、主动跳过还是被拒绝，是否都应标记为当天已处理，并在近期计划中显示明确结果。

### 4. 区分“计划定义”和“计划实例”

当前系统没有单独持久化未来计划实例。近期计划是根据计划定义、日期和当前设置即时计算出来的。

这个方案简单、节省存储，适合 ESP32。但如果后续需要更强的审计能力，例如精确记录某一天原本应该执行什么、为什么没有执行，可以考虑引入轻量的“计划执行结果记录”，不要直接扩展计划定义本身。

### 5. 保持历史记录只记录事实

历史记录不应该承担系统诊断日志功能，也不应该随着当前配置变化重新解释旧会话。当前方向是正确的，后续应继续保持：历史记录看业务事实，系统级诊断看 Esp32Base 日志和业务事件 API。

### 6. 重新审视恢复出厂待处理时的安全操作边界

当前恢复出厂待处理状态会拒绝配置保存、手动浇水、停止、清除告警、计划保存和计划跳过等操作。这个设计可以减少恢复过程中的状态竞争，但从安全优先原则看，停止和紧急停止属于安全操作，后续建议单独讨论是否应允许它们绕过恢复出厂待处理限制。

更专业的边界可能是：恢复出厂待处理时拒绝新增任务和持久化写入，但仍允许关阀类安全操作。

### 7. 区分永久停用和临时停用水路

当前页面保存计划时，会把停用水路的计划时长清零。这适合“用户实际只用一路”的长期配置，但如果用户只是临时关闭某一路维修，保存计划时清掉原始时长可能会让后续恢复该路时丢失计划内容。

后续建议讨论是否需要区分两种语义：

- 永久停用：该路不再参与计划，保存时可以清零。
- 临时停用：计划定义保留原时长，执行时按当前启用状态忽略该路。

目前系统已经在执行时计算有效水路，所以具备支持“临时停用保留计划定义”的基础。

## 十二、当前核心设计结论

当前最重要的设计结论是：

1. 系统是两路灌溉控制器，单路使用是配置退化，不是产品主模型。
2. 计划定义和实际执行要分开看，实际执行必须受当前启用水路约束。
3. 近期计划面向未来控制，历史记录面向事实回看。
4. 手动和计划共用同一套浇水会话状态机，避免重复实现。
5. 安全优先于补偿执行；冲突计划跳过，不排队。
6. 水流异常按路处理，紧急停止按会话处理。
7. 当前告警是业务告警，设备级问题应主要交给基础库日志和系统工具。
