# ESP32 灌溉控制器系统模型

## 目标

本项目从零实现一个家用/小型智能灌溉控制器。系统核心模型固定为：

```text
1 个流量计，必配
最多 6 路电磁阀 Zone
所有 Zone 串行运行，不允许同时开启
可选 1 路自吸泵启动输出，默认不启用
可选 1 路低液位输入，默认不启用
浇水任务只按时间停止，水量只用于观察、记录和异常判断
```

## 水路模型

推荐水路拓扑：

```text
水源
 -> 过滤/必要的防回流或减压
 -> 可选自吸泵或主执行器
 -> 1 个流量计
 -> 分水器
 -> Zone 1 电磁阀
 -> Zone 2 电磁阀
 -> ...
 -> Zone 6 电磁阀
```

系统一次只打开一个 Zone，因此流量计读数直接归属于当前运行 Zone。这样可以明确判断当前水路是否实际出水，并估算当前水路的浇水量。

当前模型不做：

```text
多个流量计
多个供水域
多个 Zone 同时运行
按目标水量停止
泵 PWM 调速
水塔液位管理或补水控制
天气联动内置控制
```

## 硬件接口

板级接口目标：

```text
FLOW_IN x1
VALVE_OUT_1 ... VALVE_OUT_6
PUMP_START_OUT x1
LOW_LEVEL_IN x1
```

`FLOW_IN` 必接。没有有效流量计时，系统不能满足“确认有水”和“统计浇水量”的目标。

`VALVE_OUT_1 ... VALVE_OUT_6` 固定面向 12V DC 常闭电磁阀。6 路 N 沟道逻辑电平 MOS 管低边驱动集成在同一块控制板上，不拆分独立电磁阀驱动板。

每路阀门驱动支持统一的吸合/保持控制：

```text
吸合阶段：100% 全功率输出
保持阶段：PWM 降低平均电流，减少线圈发热和整机功耗
```

默认参数：

```text
valvePullInMs = 3000
valveHoldDutyPercent = 60
valvePwmFrequencyHz = 20000
```

这 3 个参数作为系统级高级配置，不做每路单独参数。最终默认值必须通过目标电磁阀实测确认；若实测 20kHz 下保持力不足或 MOS/续流器件发热异常，应调整频率或保持占空比。

`PUMP_START_OUT` 是泵启动控制信号，不是泵电源输出。主板不直接承载交流泵或大电流直流泵主回路，应控制外部泵启动继电器、接触器或泵控制模块。

`LOW_LEVEL_IN` 可选，推荐支持两线制浮球液位开关。推荐安全定义：

```text
水位正常：触点闭合，输入为低电平
低液位或线缆断开：触点断开，输入为高电平
```

## Zone 配置

电路板固定提供 6 路阀门输出，但实际现场不一定接满 6 路水路。因此每个 Zone 必须有启用/禁用配置。

推荐默认：

```text
Zone 1 默认启用
Zone 2..6 默认禁用
```

禁用 Zone 的规则：

```text
不出现在本地手动选择列表
计划执行时跳过
API 手动启动时拒绝
不参与故障锁定
不生成浇水记录
```

计划编辑页面可以显示全部 6 路，但禁用 Zone 应明确标注为未启用，默认运行时长为 0。

## 浇水运行逻辑

启动一个 Zone：

```text
1. 检查没有其他 Zone 正在运行。
2. 检查目标 Zone 已启用且未被故障锁定。
3. 如果启用低液位输入，先检查低液位状态。
4. 打开目标 Zone 电磁阀。
5. 延迟约 0.5~1 秒。
6. 如果启用自吸泵控制，启动 PUMP_START_OUT。
7. 进入出水确认窗口，默认 15 秒。
8. 检测到有效流量后进入正常浇水。
9. 到达设定时长、用户停止或故障触发后停止。
```

浇水任务只按时间结束。流量计统计的水量是估算值，用于记录、观察和异常判断，不作为“浇够多少水就停止”的控制目标。

系统任意时刻最多只有一个 Zone 正在运行，因此软件停止逻辑只有一个：

```text
停止当前运行
```

不存在“停止单路/停止多路/Stop All”的业务概念，也不设计软件急停。真正的紧急断电应由外部电源开关或拔电完成，属于硬件供电层。

停止当前运行必须做二次确认，避免误触。确认后：

```text
1. 立即关闭 PUMP_START_OUT。
2. 关闭当前 Zone 电磁阀。
3. 取消当前自动计划剩余未执行 Zone。
4. 清空等待队列。
5. 写入当前 Zone 的浇水记录，停止原因为 user_stop。
```

停止后不保存剩余进度，不支持暂停/继续，不支持恢复未完成计划。

## 自动计划模型

计划采用简化的成熟灌溉控制器模型：

```text
最多 6 个计划
每个计划 1 个开始时间
每个计划独立设置浇水周期
每个计划为 Zone 1..6 设置运行时长
运行时长为 0 的 Zone 跳过
计划到点后按 Zone 编号串行运行
```

周期规则使用统一模型：

```text
cycleLengthDays：周期长度，1~31 天
activeDayMask：周期内哪些天浇水
cycleAnchorDate：周期第 1 天对应的真实日期
```

页面可以提供“每天”“每周”“自定义周期”等快捷表达，但底层只保存这一套周期规则。

## 自动浇水状态

自动浇水总控是一个状态：

```text
开启
关闭
暂停到指定时间后自动开启
```

自动浇水关闭或暂停期间，自动计划到点直接跳过，不排队、不补跑。手动浇水不受自动浇水状态影响。

## 冲突和中断

自动计划冲突时排队顺延，不并发、不抢占。计划等待超过设定时间后跳过。默认最长等待时间后续在详细设计中确定。

当前版本不做运行中暂停/继续，也不做重启后恢复未完成计划。

用户停止、故障停机、设备重启或掉电都会终止当前运行。上电后泵和阀保持关闭，后续自动计划按正常时间继续；已经错过的当前运行不补跑。

## 故障策略

推荐默认规则：

```text
无水：必须停机，默认锁定当前 Zone，可配置不锁定
高流量：默认停机并锁定当前 Zone，可配置只报警
低流量：默认只报警，可配置停机和锁定当前 Zone
低液位：必须停机，不锁定 Zone，液位恢复后可再试
待机漏水：全局锁定，必须手工清除
流量计异常：全局锁定，必须手工清除
```

锁定只阻止后续启动，不删除历史记录。清除锁定必须由用户确认。

## 记录和日志

浇水记录是主记录，实际开阀运行后写入，尽量多保留。每条记录包含：

```text
zoneId
source：手动 / 自动计划
planId
startedAt
endedAt
plannedDurationSec
actualDurationSec
estimatedVolumeMl
avgFlowMlPerMin
minFlowMlPerMin
maxFlowMlPerMin
stopReason
faultCode
```

事件日志保持轻量，只记录故障、警告和重要状态变化。正常计划开始、正常完成、正常手动浇水不重复写事件日志。

事件日志默认使用基础库能力，目标配置 512 条，最大可配 1024 条，环形覆盖。

## 交互边界

Web 页面负责完整配置和记录查看。

API 负责状态查询、手动启动当前 Zone、停止当前运行、设置自动浇水状态、设置计划、清除故障、读取记录等原子操作。

本地按键只做现场常用操作和应急操作：

```text
选择 Zone
启动/停止当前 Zone
信息页切换
```

本地屏幕只显示状态和少量确认，不做复杂配置菜单。
