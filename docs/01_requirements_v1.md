# ESP32 灌溉控制器当前需求 V2

本文是当前实现依据。`old-docs/` 只保留为背景材料；当旧文档与本文冲突时，以本文和根目录 [灌溉系统核心逻辑说明](../灌溉系统核心逻辑说明.md) 为准。

## 1. 产品目标

本项目实现一台适合家庭阳台、楼顶花园等场景的 ESP32 智能灌溉控制器。目标按优先级排序：

1. 安全可靠：启动、异常、停止、恢复出厂和重启路径都默认关阀。
2. 清晰可用：Web/API 覆盖业务功能，本地按键只承载高频安全动作。
3. 边界明确：应用层只实现灌溉业务，系统维护、日志、OTA、认证等基础能力优先使用 `Esp32Base`。
4. 不过度设计：首版不内置天气联动、复杂统计或云端服务，通过局域网 API 留出扩展边界。

## 2. 首版范围

### 必做能力

- 固定 4 路水路板级配置，默认启用第 1、2 路；实际焊接几路，就在灌溉设置中启用对应水路。
- 每路独立控制阀门、计时、流量统计、水流异常判断和历史记录。
- 手动浇水和计划浇水统一为“路级浇水任务”；同一路互斥，不同路可并行。
- 所有浇水任务必须有目标时长，不允许永久开启。
- 支持每路流量计脉冲计数、启动阶段补偿水量和稳定脉冲 P/L 配置。
- 支持待机漏水检测。
- 支持局域网 Web 页面和 HTTP API。
- 支持 WiFi 配网、NTP/mDNS、Web OTA、基础日志和健康诊断，均使用 `Esp32Base`。
- 支持本地 GPIO0/BOOT 长按恢复出厂；业务 Web 不提供恢复出厂入口。

### 延后能力

- 天气联动、智能家居联动和复杂自动化由外部系统调用 API 实现。
- 大型前端 SPA、复杂报表、云端访问服务不在首版范围。
- 极限低功耗优化在实机稳定后再评估。

## 3. 核心业务规则

### 水路模型

- 固定 `MaxRoads = 4`。
- 启用状态使用 4 bit mask，而不是“启用数量”。
- 默认启用 mask 为 `0b0011`，即第 1、2 路启用。
- 每路固定一个阀门 PWM 输出引脚和一个流量计脉冲输入引脚；页面只展示，不开放修改。

### 手动浇水

- 触发来源包括 Web 页面、HTTP API、本地按键。
- 每路手动启动独立判断：停用、忙碌、时长非法时只影响该路，不阻塞其他可启动水路。
- Web 页面触发必须记录为 `web_page`；外部 HTTP API 触发必须记录为 `http_api`。
- 本地按键只操作当前选中的已启用水路，使用系统级默认手动时长；本地不启动全部水路。
- 本地屏幕和按键选择列表只展示已启用水路；停用水路不展示、不允许本地启动。
- 本地启动、单路停止和停止全部都必须二次确认；第一版采用同一个按键 5 秒内再次按下确认。
- 停止全部和停止单路只结束已经运行的任务，并记录停止来源和停止范围。

### 计划浇水

- 每路最多 6 条计划，总计 24 个计划槽。
- 每个计划槽只属于一路，使用 `uint32_t planId` 标识，保存启用状态、启动时间、目标时长和循环规则；最近处理状态不写入计划定义，由 PlanExecutionTracker 按 `planId + 日期 + 分钟` 记录。
- 不再配置“同时/顺序模式”。同时浇水通过多路相同启动时间表达，顺序浇水通过不同启动时间表达。
- 计划触发只锁定目标水路，不锁定整个系统。
- 如果目标水路停用、忙碌、配置无效或被跳过，不生成浇水历史记录，只保存计划触发结果。

### 记录与结果

- 每条历史记录对应一路一次已经实际启动过的浇水任务。
- 浇水执行结果包括：正常完成、用户停止、水流异常停止、漏水保护停止、恢复出厂保护停止。
- 计划触发结果独立保存，用于解释计划到点后为什么没有形成浇水记录。
- 业务事件使用 `Esp32BaseAppEventLog` 记录，用于解释计划跳过、保护触发、运行异常和告警清除等灌溉业务行为；设备系统日志由 `Esp32Base` 负责。

## 4. 异常与安全

- 启动初始化时必须先关闭所有阀门。
- 停用水路不能启动任务。
- 同一路不能同时运行两个任务。
- 水流异常只关闭异常水路。
- 待机漏水检测只在所有水路未运行、阀门应关闭时启用；触发后关闭全部阀门并保留告警。
- 校准采集或等待录入实际水量期间，不允许启动普通手动/计划浇水；停止全部会中止校准并关阀。
- 恢复出厂待处理时仍允许停止类安全操作。
- WiFi 断开不影响本地安全逻辑。

## 5. 页面与 API

- 首页：设备状态、已启用水路当前浇水状态、手动启动、单路停止、停止全部、当前告警；停用水路在首页隐藏。
- 当前计划页面：按 24 个计划槽配置每路计划；单次跳过通过 `/api/v1/schedule/skip` 和 `/api/v1/schedule/unskip` 支持。
- 后续目标：如确认需要，再实现今天、明天、后天的近期计划展开视图和页面跳过操作。
- 历史记录：只显示浇水任务记录。
- 计划配置：按 24 个计划槽配置每路计划。
- 灌溉设置：水路启用、名称、固定引脚、流量参数和安全阈值；默认手动时长由系统级配置提供。
- API 路径使用 `/api/v1/*`，危险动作必须使用 POST。

## 6. 当前已知硬件边界

- 本地按键不按固定水路编号分配快捷键；通过上一个/下一个选择任意已启用水路，再启动或停止当前选中水路。
- Stop All 必须始终可用，用于现场快速停水。
- GPIO35/36/39 是输入脚，没有内部上拉；GPIO32 在本设计中也按板级上拉处理。流量计必须外接或板载上拉。
- 阀门 PWM 输出不能直接驱动电磁阀，必须通过 MOSFET、继电器模块或专用驱动器。

## 7. 验收标准

- 编译：`pio run` 通过。
- 静态结构：`node scripts/check-web-structure.mjs` 通过。
- Web/API：所有业务页面和 API 返回合法内容，页面操作与 JSON 结果一致。
- 台架：四路 PWM、四路流量脉冲、停止保护和记录写入均通过无水测试。
- 真实水路：单路、多路、计划、跳过、水流异常、漏水和恢复出厂保护均通过。
- 稳定性：最终交付前完成 72 小时连续运行验证。
