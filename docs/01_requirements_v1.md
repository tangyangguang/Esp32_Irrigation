# ESP32 灌溉控制器需求优化稿 V1

本文基于 `old-docs/` 重新整理需求，作为后续实现的当前依据。旧文档仍保留作背景，但实现时优先遵守本文和 `AGENTS.md`。

## 1. 产品目标

本项目实现一台适合家庭阳台、楼顶花园等场景的 ESP32 智能灌溉控制器。设计目标按优先级排序：

1. 安全可靠：任何异常、紧急停止、重启路径都默认关闭电磁阀。
2. 方便使用：常用操作少步骤，Web/API 覆盖全部功能，本地 LCD/按键适合现场高频操作。
3. 合理可维护：业务逻辑清晰，基础能力优先使用 `Esp32Base`，不在应用项目里重复造基础设施。
4. 不过度设计：首版只实现本机必需功能，复杂自动化通过局域网 API 扩展。

## 2. 首版范围

### 2.1 必做能力

- 支持 1-2 路 12V 常闭电磁阀，硬件最多 2 路，默认启用 1 路，软件可配置实际启用路数。
- 支持手动浇水、定时浇水、紧急停止。
- 所有浇水任务必须有倒计时，不允许永久开启。
- 紧急停止优先级最高，任何状态下一键关闭所有阀门。
- 支持每路 YF-S201 流量计脉冲计数和水量换算。
- 支持每路校准系数，默认 1.0。
- 支持 LCD1602、独立按键、蓝色状态 LED。取消 EC11 编码器，降低误触、机械寿命和防水结构复杂度。
- 支持局域网 Web 页面和 HTTP API。
- 支持 WiFi 配网、NTP/mDNS、Web OTA、基础日志和健康诊断，均使用 `Esp32Base`。
- 支持恢复出厂：仅配置、配置加记录。

### 2.2 延后能力

- 设备自身以局域网 Web/API 为边界，不包含设备固件以外的访问服务。
- 天气联动：首版不内置，可由局域网内其他系统调用 API 实现。
- 大型前端 SPA：不做，使用轻量服务端页面。
- 复杂统计报表：不做，只保留记录查询和导出基础能力。
- 极限低功耗指标：先实现可验证的 WiFi power save 策略，实机验证后再决定是否继续优化。

## 3. 核心业务规则

### 3.1 状态优先级

优先级固定为：

```text
紧急停止 > 手动浇水 > 定时浇水 > 待机
```

- 紧急停止会立即关闭所有阀门，结束当前所有浇水会话。
- 手动浇水可以打断定时浇水。
- 定时浇水遇到手动浇水时，本阶段默认跳过本次并记录原因。
- 任何状态机异常、参数非法、时间异常、存储异常，都不得导致阀门保持开启。

### 3.2 手动浇水

- 触发来源：本地 `START/OK` 键、Web、API。
- 每路独立设置浇水时长，范围 1-240 分钟。
- 未参与本次浇水的路，其时长记为 0。
- 只能设置已启用路；默认只启用第 1 路时，本地和 Web 都不显示第 2 路控制项。
- 支持两种执行模式：
  - 同时浇水：所有参与路同时启动，各自按独立倒计时关闭。
  - 顺序浇水：按路号从小到大依次执行；第 1 路完成后再启动第 2 路，时长为 0 的路跳过。
- 默认执行模式为同时浇水。
- 同时浇水示例：第 1 路 5 分钟、第 2 路 12 分钟，则两路同时开阀；第 1 路 5 分钟到时关闭，第 2 路继续运行到 12 分钟关闭。
- 顺序浇水示例：第 1 路 5 分钟、第 2 路 12 分钟，则先开第 1 路 5 分钟，关闭后再开第 2 路 12 分钟，总会话约 17 分钟。
- 所有参与路都关闭后，本次浇水会话结束。
- 手动停止或紧急停止后必须写入结束原因。

### 3.3 浇花计划

浇花计划是定时浇水的产品化表达，首版按“少而清楚”设计，不做复杂自动化规则。

- 最多 8 个计划。
- 每个计划包含：启用状态、开始时间、每路浇水时长、执行模式、重复方式。
- 每路浇水时长独立设置，范围 0-240 分钟；0 表示该路不参与本计划。
- 至少需要一条已启用路的时长大于 0，计划才有效。
- 计划触发后按计划的执行模式运行：
  - 同时浇水：所有时长大于 0 的路同时启动，并按各自时长独立关闭。
  - 顺序浇水：按路号从小到大依次执行所有时长大于 0 的路。
- 如果需要两路不同开始时间，创建两个计划，不在单个计划内设计多段时间轴。
- 重复方式只保留两类：
  - 按星期：选择周一到周日中的任意天；每天浇水就是全选。
  - 按间隔：每 N 天一次，N 范围 1-30 天，用上次执行日期作为下一次计算依据。
- 时间精度：1 分钟。
- 首版只支持“跳过今天”和“取消跳过今天”。天气、智能家居等外部系统如需联动，也通过该能力实现。
- 冲突处理保持简单：如果计划触发时已有手动浇水或其他计划正在运行，则跳过本次并记录原因，不等待、不叠加、不自动延长。

### 3.4 浇水会话规则

- 手动浇水和浇花计划统一进入同一种浇水会话。
- 会话保存执行模式、两路的目标时长、开始时间、关闭时间、实际流量和结束原因。
- 每一路都是独立子任务：可运行、已完成、手动停止、紧急停止、水流异常。
- 某一路到时或异常关闭，不影响另一路继续完成，除非触发紧急停止。
- 同时模式下，多路可以并行运行；顺序模式下，同一时刻最多只开一路。
- 红色 `STOP` 永远关闭所有路，并结束整个会话。
- 同一时间最多只有一个浇水会话，避免多个手动/计划会话并发。

## 4. 异常与安全

- 启动初始化时，先配置阀门引脚并关闭所有阀门，再启动业务逻辑。
- 浇水启动后，每路开阀独立计时；默认 10 秒内无流量脉冲判定为水流异常，关闭对应阀门并记录异常。
- 水流异常检测窗口可配置，范围 1-60 秒，配置键建议为 `flow_no_pulse_timeout_s`；真实硬件建议不低于 5 秒。启动初期应至少保留短暂机械/管路响应缓冲，避免 1 秒配置在刚开阀时误判。
- 待机状态下 10 秒内检测到 3 个及以上流量脉冲，判定为疑似漏水或阀门粘连，进入告警状态。
- WiFi 断开不主动推送告警，只在本地和 Web 状态显示。
- 过流检测需要明确硬件检测输入后再实现；没有硬件输入前，只在文档中保留，不写假逻辑。

## 5. 数据与配置

配置使用 `Esp32BaseConfig` 的 NVS 能力。应用 namespace 不使用 `eb_` 前缀。

系统重要参数都应可配置，首版至少包含：

- 路数与路配置：启用路数，范围 1-2，默认 1；每路名称；每路是否显示。
- 流量计量：每路校准系数，默认 1.0；每路脉冲当量，默认 450 脉冲/L。
- 浇水默认值：默认执行模式；本地快速手动浇水的每路默认时长。
- 安全阈值：水流异常检测窗口，默认 10 秒，范围 1-60 秒；待机漏水检测窗口和脉冲阈值。
- 浇花计划：最多 8 个计划，每个计划的启用状态、开始时间、每路时长、执行模式和重复方式。
- 交互设置：按键锁定状态、LCD 背光/显示偏好、状态 LED 启用状态。
- Web/API 设置：鉴权信息、API token、设备名称、hostname。
- 维护设置：记录保留策略、恢复出厂选项、OTA 相关开关。

记录数据以“可分析的原始数据”为目标，优先保存事实，不只保存汇总结论：

- 浇水会话记录：会话 ID、触发来源、执行模式、触发时间、会话开始/结束时间、结束原因。
- 每路原始记录：目标时长、实际开始/结束时间、实际运行秒数、原始脉冲数、校准系数快照、脉冲当量快照、估算水量、该路结束原因。
- 异常记录：异常时间、路号、异常类型、当时脉冲数、阀门状态、会话 ID。
- 系统事件记录：配置变更、启动、重启、OTA、恢复出厂、WiFi 状态变化、按键操作。
- 导出能力：Web 端支持 CSV/JSON 导出，导出应包含原始字段，方便后续在表格或分析工具中处理。
- 容量策略：记录容量目标先以现场诊断和导出够用为准；容量和存储格式在实测后锁定，不在首个闭环阶段强求固定条数。

## 6. 交互

### 6.1 本地交互

- LCD1602 只显示英文和数字，单行不超过 16 字符。
- 本地控制使用 6 个面板按键，常规操作全部短按完成：
  - `STOP ALL`：红色，一键关闭所有路，任何状态始终有效。
  - `R1/UP`：浇水中短按停止第 1 路；菜单中作为上一项或数值增加；手动设置界面中用于选择/取消第 1 路。
  - `R2/DOWN`：浇水中短按停止第 2 路；菜单中作为下一项或数值减少；手动设置界面中用于选择/取消第 2 路。第 2 路未启用时此键无业务动作。
  - `START/OK`：待机时启动本地快速手动浇水；菜单中确认。
  - `MENU/BACK`：进入菜单或返回上一级。
  - `LOCK`：短按锁定/解锁普通按键。
- 锁定状态下，`STOP ALL`、`R1/UP`、`R2/DOWN` 的停止功能仍然有效；其他普通按键无效。
- 本地恢复出厂使用 GPIO0/BOOT 键长按 3 秒触发确认流程；当前无 LCD 菜单阶段仅记录请求，并在 Web 设置页显示，最终执行仍需在 Web 中输入确认文本。GPIO0 是 ESP32 启动脚，按住上电或复位会进入下载模式；硬件和说明书必须明确“仅在设备正常运行后长按用于恢复出厂”。
- 恢复出厂必须二次确认，支持“仅恢复配置”和“恢复配置并清空记录”。

### 6.2 Web/API

- Web 页面必须覆盖设备全部功能，包括状态查看、手动控制、浇花计划、所有配置、流量校准、记录查询/导出、日志、恢复出厂、重启和 OTA。
- API 必须覆盖 Web 端核心操作，用于局域网自动化集成。
- 首个阶段复用 `Esp32BaseWeb` Basic Auth；业务 API token 在业务 API 完整设计时再加入。
- 登录账号和密码由 `Esp32Base` Web Auth 管理，认证优先级为已保存认证、应用默认认证、库默认认证；业务页面链接 `/esp32base/auth`，本项目不保存基础库鉴权密码。
- API 路径使用 `/api/v1/*`。
- 危险动作必须使用 POST。
- 根路径 `/` 进入灌溉业务首页；`/esp32base` 保留为基础库系统工具入口。

## 7. 验收标准

阶段性验收按实现进度拆分：

- 基础环境：`pio run` 通过，`/irrigation` 和 `/api/v1/status` 可访问。
- 硬件安全：上电后所有阀门默认关闭，紧急停止任何状态有效。
- 手动闭环：指定路按指定时长开启，到时关闭并记录。
- 流量闭环：流量脉冲可计数，校准后计量误差目标小于等于 5%。
- 计划闭环：浇花计划按分钟触发，跳过今天规则生效。
- 稳定性：最终交付前完成 72 小时连续运行验证。
