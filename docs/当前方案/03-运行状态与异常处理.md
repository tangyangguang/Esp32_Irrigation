# 03 运行状态与异常处理

## 运行并发

已确认规则：

- 同一水源同一时刻只允许 1 条分支水路运行。
- 不同水源可以并行运行。
- 同一水源只有一个流量计，当前流量和流速归属于正在运行的那条水路。
- 手动运行当前按不排队、不抢占处理；后续 Web 界面方案只验证页面表达是否清楚。
- 自动计划正常到点执行；执行不了或执行中发现异常，就报告异常事实。
- 主控下发给从站的是当前水路任务，必须包含运行时长。
- 从站收到任务后本地计时，到点关闭分支阀、主阀和水泵/自吸泵输出。

理由：

- 同一水源串行运行，流量才能归因到具体水路。
- 不同水源由不同从站执行，可以并行。
- 不提前设计复杂队列、抢占和阻断状态机，先把正常运行和异常报告说清楚。

## 启停顺序

默认启动顺序：

1. 检查水源状态和保护状态。
2. 打开主阀；未启用则跳过。
3. 打开分支阀。
4. 启动水泵；未启用则跳过。
5. 进入流量检测宽限。

默认停止顺序：

1. 停止水泵；未启用则跳过。
2. 关闭分支阀。
3. 关闭主阀；未启用则跳过。

保留少量启停延时参数，不扩展更多动作阶段。

默认值：

| 参数 | 默认值 | 说明 |
| --- | ---: | --- |
| 主阀提前开启时间 | 1 秒 | 主阀先于分支阀打开 |
| 主阀延后关闭时间 | 1 秒 | 分支阀关闭后再关主阀 |
| 水泵/自吸泵启动延后时间 | 1 秒 | 分支阀打开后再启动泵 |
| 水泵/自吸泵停止提前时间 | 1 秒 | 停止时先停泵再关阀 |

这些参数都是非负秒数。不要用正负数表达提前或延后。

## 从站当前任务闭环

从站不保存自动计划，也不做未来调度。

从站只执行当前任务闭环：

1. 主控下发水源、水路、运行时长和必要运行参数。
2. 从站按启停顺序打开对应输出。
3. 从站本地计时。
4. 到达时长后，从站本地关闭水泵/自吸泵、分支阀和主阀。
5. 如果主控提前下发停止命令，从站立即按停止顺序关闭输出。
6. 如果运行中出现从站可判断的异常，从站停止输出并等待主控后续同步。

这样做的目的不是让从站独立运行完整计划，而是保证当前任务不完全依赖主控持续在线。

## 启动前检查

启动前只做必要检查，不建立复杂阻断体系。

检查：

- 水源是否启用。
- 目标水路是否启用。
- 同一水源是否已经有水路在运行。
- 从站是否能通信。
- 如果启用了自吸泵且启用了低液位传感器，检查低液位是否有效。
- 流量计、自吸泵、低液位等已启用能力的配置是否完整。

检查不通过时，不启动本次任务，记录或显示原因即可。

## 运行中停止

运行中遇到以下情况，应停止当前任务：

- 用户停止。
- 启用了自吸泵且启用了低液位传感器时，低液位有效。
- 无流量。
- 低流量或过流量，且系统参数设置为停机。

低流量和过流量默认只提示，不停机；可以配置为停机。

## 自吸泵和低液位

低液位传感器不是必须配置项。

规则：

- 未启用自吸泵时，低液位不作为必需条件。
- 启用自吸泵但未启用低液位传感器时，系统正常启停自吸泵，缺水主要靠无流量保护发现。
- 启用自吸泵且启用低液位传感器时，启动前低液位有效则不启动。
- 启用自吸泵且启用低液位传感器时，运行中低液位有效则停止当前水路和自吸泵。

紧急停机仍然依赖现场电源开关；软件方案处理正常任务闭环和可检测异常，不做复杂告警确认流程。

## 通信异常边界

主控和从站通信异常时：

- 未启动的任务不应继续启动。
- 已下发到从站的当前任务，由从站按本地时长到点关闭。
- 主控恢复通信后，同步从站当前状态和最近任务结果。
- 不补跑通信异常期间错过的计划。

## 流量保护

保留三类流量保护：

- 待机漏水：待机时检测到持续流量，报告异常。
- 运行无流量：启动后超过宽限仍无脉冲，停止本次任务。
- 低/过流量：有标准流量时检测偏离，默认提示，不停机，可配置为停机。

默认参数：

| 参数 | 默认值 | 说明 |
| --- | ---: | --- |
| 启动流量宽限时间 | 10 秒 | 启动后等待管路建立流量 |
| 无流量确认时间 | 10 秒 | 宽限后仍无脉冲则无流量 |
| 待机漏水检测窗口 | 30 秒 | 待机时统计流量脉冲 |
| 待机漏水脉冲数 | 3 个 | 窗口内达到则报告疑似漏水 |
| 低/过流量持续时间 | 20 秒 | 持续偏离才报告 |
| 低流量比例 | 30% | 低于标准流量比例 |
| 过流量比例 | 200% | 高于标准流量比例 |
| 低/过流量默认动作 | 提示 | 可配置为停机 |

不建议当前加入：

- 目标水量自动停止。
- 累计水量异常保护。
- 每个水源单独一套复杂保护阈值。

## 我的建议

- 运行状态不要做成一个大枚举。建议拆成几个维度：在线状态、运行状态、可启动状态、保护状态、配置状态。
- 启停顺序要简单。太多提前/延后参数会让用户很难判断现场动作是否正常。
- 不要把异常都做成“阻断”。多数异常只需要解释本次为什么没执行或为什么停了。
- “低/过流量默认停机”没有必要，默认提示更稳妥。
- 从站本地闭环只做到当前任务级别，不扩展成从站计划系统。

## 后续细化

后续只需要确认这些参数的默认值和单位，不再扩大启停动作模型。
