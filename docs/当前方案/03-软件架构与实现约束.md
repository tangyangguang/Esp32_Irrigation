# 03 软件架构与实现约束

## 总体结构

固件采用“`Esp32Base` 通用平台能力 + 本项目浇水业务”的单向依赖结构。

`Esp32Base` 负责：

- 启动生命周期。
- WiFi、DNS、mDNS 和配网。
- Web Server、认证和基础 UI。
- OTA / Web OTA。
- LittleFS 封装和系统日志。
- App Events。
- 健康、看门狗和系统状态。
- NTP、RTC 和统一时间。
- 少量适合基础配置页的 App Config 字段。

本项目负责：

- 定稿 PCB 引脚语义和安全态。
- Zone、计划和业务配置。
- 阀、流量计和自吸泵信号。
- 浇水与校准状态机。
- 每日计划调度。
- 流量保护。
- 浇水记录。
- 面向用户的浇水 Web 页面。

如果基础库能力缺失或存在缺陷，本项目只能记录问题并给基础库维护者提出修改建议；不能修改 `../Esp32Base`，也不能在本项目复制一套通用框架。

## 基础库构建约束

- 使用 `ESP32BASE_PROFILE_FULL`。
- 显式启用 `ESP32BASE_ENABLE_APP_EVENTS=1`。
- 显式启用 RTC：`ESP32BASE_ENABLE_RTC=1`。
- RTC 驱动：`ESP32BASE_RTC_DRIVER_DS3231`。
- I2C：`GPIO21 / GPIO22`。
- 应用在 `Esp32Base::begin()` 前初始化 `Wire` 并调用 `Esp32BaseRtc::configure(Wire)`，或明确选择基础库自动初始化；只能保留一种方式。
- 使用 classic ESP32 4MB 双 OTA 分区表 `esp32-4mb-ota-balanced.csv`。
- 业务文件必须位于 `/app/irrigation/`，不能写入 `/esp32base/` 命名空间。
- 业务文件通过 `Esp32BaseFs` 访问，不直接依赖 `LittleFS.h`。

时间统一使用：

- `Esp32BaseTime::snapshot()`。
- `Esp32BaseTime::formatTime()` / `formatEpoch()`。
- `Esp32BaseTime::resolveCurrentBootEvent()`。

业务代码不直接读取 DS3231 寄存器，不直接依赖 NTP 状态，也不使用 RTC INT/SQW 做计划触发。

## 启动与首次使用

启动顺序：

```text
Serial
  -> BoardHardware::safeOff()
  -> Wire / RTC binding
  -> register business Web routes
  -> Esp32Base::begin()
  -> ConfigStore::load()
  -> RunRecordStore::begin()
  -> domain services begin
  -> Scheduler armed after leaving current minute
```

规则：

- `safeOff()` 必须早于任何可能阻塞或失败的初始化。
- Web 认证和首次配网使用 `Esp32Base` 现有流程，项目不另做账号系统。
- 首次启动且业务配置不存在时，创建默认配置。
- 已存在的配置损坏时，不静默覆盖为默认值；保留损坏文件，进入 `config_invalid` 安全状态。
- `config_invalid` 状态禁止所有出水任务，允许用户通过 Web 保存新的有效配置。
- RTC 无效但 NTP 有效时可自动调度；RTC 和 NTP 都无可信时间时按业务规则只禁止自动计划。
- 重启发现未完成任务时，不恢复输出，补写 `interrupted_by_restart` 浇水结果和业务事件。

## 分层和依赖方向

```text
IrrigationWeb
  -> ZoneService / PlanService / RunController / Scheduler / RunRecordStore
      -> ConfigStore / EventAdapter / BoardHardware
          -> Esp32Base / Arduino GPIO
```

禁止：

- Web 直接操作 GPIO、文件或 NVS。
- Scheduler 直接打开阀。
- BoardHardware 理解计划、历史和页面。
- ConfigStore 做业务调度决策。
- FlowSafetyService 自己关闭 GPIO；它只能向 RunController 报告保护决策。
- 校准另写一套阻塞式硬件流程。

## 核心服务

| 服务 | 单一职责 |
| --- | --- |
| `BoardHardware` | 固定引脚、安全态、阀 PWM、流量脉冲快照和泵干接点信号 |
| `ConfigStore` | 业务配置读写、版本和完整校验 |
| `ZoneService` | Zone 启用、名称和标准流量 |
| `PlanService` | 计划、启动时间、Zone 时长和重复时间校验 |
| `RunController` | 唯一的出水任务状态机和运行快照 |
| `FlowSafetyService` | 缺水、漏水、低流量和过流量判定 |
| `Scheduler` | 每日计划判定和当天触发位图 |
| `RunRecordStore` | 本项目浇水记录环形存储和查询 |
| `EventAdapter` | 将业务事件转换为 `Esp32BaseAppEventLog::Event` |
| `IrrigationWeb` | 页面、表单、业务 API 和状态展示 |

当前不创建 `LocalButtons`、`LcdDisplay` 或本地菜单模块。未来实现时只能依赖上述业务服务。

## 固定产品常量

| 常量 | 值 |
| --- | ---: |
| Zone 数量 | 6 |
| 默认启用 Zone 数量 | 6 |
| 最大计划数 | 8 |
| 每计划最大启动时间数 | 4 |
| 同时活动 Zone 数 | 1 |
| 普通单 Zone 最大时长 | 120 分钟 |
| 校准最大时长 | 60 分钟 |

代码只保留 `kMaxZones = 6`，不保留可修改的 `valveIndex`。物理输出由 Zone 数组索引固定推导。

## 业务配置

内存配置聚合为：

```cpp
struct IrrigationConfig {
    uint16_t version;
    SupplyConfig supply;
    FlowConfig flow;
    ValveConfig valve;
    ZoneConfig zones[6];
    WateringPlan plans[8];
};
```

默认值：

| 配置 | 默认值 |
| --- | ---: |
| Zone 1..6 | 全部启用 |
| 计划 | 无 |
| 自吸泵 | 禁用 |
| 流量计脉冲/L | 250 |
| 阀吸合时间 | 3 秒 |
| 阀维持占空比 | 60% |
| 泵启动延时 | 0 秒 |
| 停泵后关阀延时 | 1 秒 |
| 流量建立宽限 | 10 秒 |
| 连续无流量确认 | 10 秒 |
| 待机漏水窗口/脉冲 | 30 秒 / 3 个 |
| 低/过流量持续确认 | 20 秒 |
| 低流量阈值 | 标准流量的 30% |
| 过流量阈值 | 标准流量的 200% |
| 低/过流量动作 | 提示 |

所有字段必须定义后端范围校验，HTML 输入限制不能替代后端校验。至少校验：

- 名称 UTF-8 长度。
- 启动时间 0..1439 且计划内不重复。
- 启用计划之间启动时间不重复。
- 时长范围。
- 脉冲/L 非零且在传感器合理范围。
- 比例、窗口、延时和 PWM 占空比范围。
- 至少一个有效 Zone 步骤才能启动，但允许保存暂时没有步骤的禁用计划。

## 配置存储

业务配置文件：

```text
/app/irrigation/config.json
```

格式包含：

- `version`
- `updatedAt`，无可信时间时为 0
- `data`

写入流程：

1. 将完整新配置写入临时文件。
2. 重新读取并做结构、范围和交叉校验。
3. 原子替换正式文件。
4. 失败时保留原正式配置和当前运行配置。
5. 成功后通知服务刷新；正在运行任务继续使用原快照。

当前只支持最终方案的配置结构，不读取旧 V6、旧原型或旧字段。`version` 用于拒绝不支持的结构，并为未来正式发布后的 OTA 升级保留明确入口，不在当前代码中加入未知历史迁移分支。

## WateringRun 数据边界

任务快照至少包含：

- 运行 ID、来源、计划 ID 和计划名称快照。
- 最多 6 个步骤。
- 每个步骤的 Zone ID、名称快照、目标时长和标准流量快照。
- 阀、泵和流量保护配置快照。
- 当前状态、步骤、结果和原因码。
- 开始、结束、状态进入时间。
- 当前和累计脉冲、估算水量及有效运行时间。

原因码至少包括：

- `user_stop`
- `no_flow`
- `low_flow`
- `high_flow`
- `config_invalid`
- `hardware_safety_error`
- `interrupted_by_restart`

设备忙、没有有效步骤、计划禁用、时间无效等发生在任务接受前，属于请求结果或计划事件，不进入运行原因码。

## RunController 状态机

```text
Idle
  -> Precheck
  -> OpenValve
  -> PumpSignalOn           (pump enabled)
  -> PumpStartDelay         (pump enabled)
  -> FlowGrace
  -> Running
  -> PumpSignalOff          (pump enabled)
  -> PumpStopDelay          (pump enabled)
  -> CloseValve
  -> AdvanceStep
  -> Finalize
  -> Idle
```

状态规则：

- `Precheck` 只验证已经接受的任务快照和最终硬件条件，不再次创建任务。
- `OpenValve` 只提交开阀动作并立即进入后续状态；`BoardHardware` 独立使用时间戳完成 100% 吸合到维持 PWM 的切换，不阻塞 RunController，也不延迟泵信号和流量宽限计时。
- `FlowGrace` 计数脉冲但不计目标浇水时长、不判断低/过流量。
- `Running` 执行目标计时和全部适用保护。
- 普通完成、用户停止和保护停止最终都进入同一安全关闭链。
- 用户停止在 `OpenValve` 到 `AdvanceStep` 的任何状态都必须生效。
- 缺水以及配置为停止的低/过流量不进入下一步骤，直接安全关闭并结束批次。
- `Finalize` 写小型固定浇水记录和事件请求，不执行长循环、不等待网络。

统一安全关闭：

```text
stop requested / fault detected
  -> pump signal off if asserted
  -> pump stop delay if required
  -> active valve off
  -> DRV_SD shutdown
  -> finalize current step and run
  -> Idle
```

所有状态机等待使用无符号时间差处理 `millis()` 回卷。流量脉冲由中断或轻量驱动累计，业务层只读取原子快照。

## 调度实现约束

Scheduler 只读取 `Esp32BaseTime` 和 `PlanService` 快照，并调用 `RunController::requestStart()`。

- 不直接操作硬件。
- 不直接写浇水记录。
- 计划跳过通过 `EventAdapter` 写事件。
- 当天触发位图存入独立 NVS key，不混入 JSON 配置。
- 同一槽位在判断到期时先设置触发位，再尝试启动。
- 时间来源从 RTC 升级到 NTP 时，沿用同一本地日期位图，不重复运行。

## 浇水记录存储

文件：

```text
/app/irrigation/runs.bin
```

采用固定大小环形记录，当前固定保留最近 512 个批次。容量不是用户设置。

每条记录包含：

- magic、结构版本、记录序号和 CRC。
- 批次级元数据和最多 6 个固定步骤结果。
- 完整提交标志，避免掉电后把半条记录当作有效数据。

写入必须是可恢复的两阶段提交；启动时扫描头信息和必要记录，损坏单条应跳过并产生业务事件，不能格式化整个文件系统。

NVS 保存最小未完成任务标记。开始任务、进入新 Zone 和正常结束时更新；重启后据此生成中断结果。不得按每个流量脉冲或每秒写 NVS。

App Events 不保存完整浇水记录，`RunRecordStore` 也不复制事件的筛选、导出和清除功能。

## Web 信息结构

业务顶层只保留：

1. 总览
2. 运行
3. 计划
4. Zone
5. 浇水记录
6. 浇水设置

### 总览

- 当前时间及来源。
- 空闲/运行/配置无效状态。
- 当前任务和当前 Zone。
- 下一次计划。
- 今日浇水量；时间无效时显示不可用。
- 活动流量告警和最近业务事件摘要。
- 自吸泵启用时显示“控制信号”状态。

### 运行

- 已启用 Zone 的时长表，0 表示跳过。
- 启动顺序浇水。
- 选择已启用计划立即执行。
- 当前步骤、目标/已运行时间、流量、脉冲和用水量。
- 停止按钮和不能启动的明确原因。

### 计划

- 最多 8 个计划。
- 名称、启用状态、最多 4 个每日时间。
- 只显示当前已启用 Zone 的时长输入，存储仍保留全部 6 个时长。
- 禁用计划可编辑、启用、删除，但不能立即执行。

### Zone

- 固定显示 6 个 Zone。
- 名称、启用状态、固定输出、标准流量和最近浇水。
- Zone 标准流量校准入口。

### 浇水记录

- 只展示本项目 `RunRecordStore` 的浇水记录。
- 支持按来源、Zone、结果和时间查看。
- 业务事件使用基础库 App Events 页面，浇水记录页只提供入口，不重新实现事件管理。

### 浇水设置

- 自吸泵启用、启动延时和停止延时。
- 流量计每升脉冲数和流量计校准入口。
- 流量建立、缺水、漏水、低流量和过流量参数。
- 阀吸合时间和维持占空比。
- 不提供事件保留数量、输入输出映射、调试 GPIO 和本地按钮/LCD 设置。

WiFi、认证、OTA、系统日志、App Events、文件系统和系统状态使用 `Esp32Base` 页面，不在业务页面重做。

## 路由原则

页面路径建议：

```text
/index
/run
/plans
/zones
/history
/settings
/calibration/flow-meter
/calibration/zone
```

校准页面不是顶层导航，从 Zone 或浇水设置进入。

写操作使用 POST，后端重新做认证、同源和参数校验。表单提交优先使用 POST + 重定向；只有实时运行状态使用轻量 JSON API。Web 路由在 `Esp32Base::begin()` 前注册，并核对不超过基础库应用路由容量。

## 实现顺序

1. 清理并核对定稿引脚、`safeOff()` 和板级硬件测试。
2. 固定类型、默认配置、后端校验和 ConfigStore。
3. 实现 ZoneService、PlanService 和运行快照生成。
4. 实现手动 RunController、安全关闭和用户停止。
5. 加入全程缺水保护、漏水、低/过流量和自吸泵状态。
6. 实现 RunRecordStore 和未完成任务恢复记录。
7. 实现 Scheduler、当天触发位图和时间跳变测试。
8. 实现校准流程。
9. 完成 Web 页面并接入业务快照。
10. 做实阀、流量计、自吸泵信号、断电、重启、OTA 和存储故障联调。

LCD2004 和 4 个本地按钮不在以上阶段内。当前 Web 与业务功能验收完成后，再单独设计本地交互方案。

## 代码开工验收合同

以下条件全部满足才能认为当前方案正确落地：

- 引脚与 2026-07-11 网表逐项一致。
- 上电、重启、配置损坏和 OTA 期间所有输出安全关闭。
- 默认启用 6 个 Zone，无默认计划，自吸泵禁用。
- 手动、计划和立即执行共用一个状态机。
- 任何出水任务，包括校准，运行全过程都有缺水保护。
- 低/过流量默认提示，配置为停止时终止整个批次。
- 忙时手动请求拒绝、计划跳过，均不创建伪浇水记录。
- 自动计划不补跑、不重复、不排队。
- 浇水记录和 App Events 分开保存、分开展示。
- 当前运行配置使用不可变快照。
- 重启不恢复浇水，并能解释上次任务为何中断。
- Web 不直接碰 GPIO、NVS 或文件。
- 当前代码不包含旧 V6、旧原型、RS485、主阀、映射、本地按键或 LCD 兼容分支。
