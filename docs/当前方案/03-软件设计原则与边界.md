# 03 软件设计

## 设计结论

旧固件实现已经清除。新软件只依据当前产品方案、定稿 PCB 和已经核实的 `Esp32Base` 公共能力从空白设计，不继承旧代码的默认值、命名、对象拆分、配置、状态机或页面结构。

设计保持四个边界：

1. 硬件适配：安全控制 PCB 固定输入输出。
2. 浇水业务：执行、流量保护、调度、校准和学习。
3. 数据存储：业务配置、调度防重复标记和浇水记录。
4. Web 接口：页面展示、输入校验和业务命令入口。

业务规则只在业务层实现一次。Web、自动调度、校准和将来的本地按钮都调用同一套能力，不能直接控制 GPIO 或读写业务文件。

## 模块职责

| 模块 | 唯一职责 |
| --- | --- |
| `BoardHardware` | 引脚初始化、阀驱动、泵控制信号、流量脉冲计数和全部输出安全关闭 |
| `IrrigationConfig` / `IrrigationConfigStore` | 当前配置结构、默认值、完整校验、原子保存和运行时配置快照 |
| `WateringController` | 当前唯一活动操作、区域顺序、非阻塞执行、停止和结果汇总 |
| `FlowMonitor` | 流量建立、无流量保护、流速与水量计算、低/高流量和非计划流量判断 |
| `WateringScheduler` | 可信时间下检查计划、避免重复、提交自动浇水请求 |
| `FlowCalibrationService` | 管理流量计校准样本和区域基准流量学习，出水仍交给 `WateringController` |
| `WateringRecordStore` | 保存、分页、导出和清理浇水记录 |
| `IrrigationEvents` | 将业务事件统一写入 `Esp32Base` App Events |
| `IrrigationWeb` | 注册业务页面和接口，将请求转换为业务命令 |

不单独设置只做简单转发的区域服务、计划服务或通用仓储层。配置修改由 `IrrigationConfigStore` 的明确业务方法完成。

## 核心数据

### 持久配置

- `ValveDriveConfig`：全功率吸合时间、PWM 频率、维持占空比。
- `PumpConfig`：是否启用、启动延时、停泵后关阀延时。
- `FlowMeterConfig`：当前脉冲/L。
- `FlowProtectionConfig`：无流量、低/高流量和非计划流量参数及动作。
- `ZoneConfig`：固定编号、启用状态、名称、基准流量；基准流量为 0 表示未学习。
- `WateringPlan`：编号、名称、启用状态、每日启动时间和各区域运行时长。

固定硬件引脚、区域数量、计划数量、单区域最长运行时间和 UTC+8 时区是产品常量，不作为用户配置。

### 运行数据

- `WateringRequest`：一次请求的来源、用途和区域时长。
- `WateringStep`：一个区域及其本次目标运行时长。
- `WateringSession`：请求开始时生成的配置与步骤快照、当前区域、计时、脉冲和结果。
- `WateringStatus`：供 Web 读取的统一只读状态，不暴露可修改的内部对象。
- `WateringRecord`：一次实际出水的最终业务记录。
- `ZoneWateringRecord`：一次会话中单个区域的计划值、实际值和结果。

`WateringSource` 表示手动区域、自动计划或手动立即执行计划；`WateringPurpose` 表示普通浇水、流量计校准或区域流量学习。结果使用 `WateringResult`，停止原因使用 `WateringStopReason`。这些概念保持分开，避免使用脱离上下文后含义不清的通用 `Operation`、`Result` 或 `Reason` 名称。

## 配置模型与存储

- 全部浇水业务配置使用一个权威文件 `/app/irrigation/config.json`，不拆成 NVS、App Config 和多个业务文件。
- `Esp32Base` App Config 适合少量标量字段，不适合同时原子管理 6 个区域和 8 个计划的嵌套数据，因此本项目使用 `Esp32BaseFs` 保存领域配置，不重复实现通用文件系统能力。
- 配置包含一个当前结构版本。只接受当前版本，不读取或迁移旧结构。
- 首次启动且文件不存在时生成默认配置；已有文件解析失败、版本不符或校验失败时不能用默认值覆盖原文件。
- 配置不可用时所有输出保持关闭，页面显示具体错误，并允许用户明确确认后重新生成配置。
- 保存时先构造完整副本，校验所有字段和跨字段关系，再写临时文件、回读校验并原子替换。掉电恢复只处理当前结构的临时文件或备份文件。
- 保存成功后一次性替换内存配置。已经开始的操作继续使用自身快照，修改只影响之后的操作。
- PWM 频率只在空闲时重新配置全部阀通道；运行中保存的新频率从下一次出水开始生效。

必须校验的关系包括：区域与固定编号一致、名称有效、PWM 为 1–25 kHz、占空比不为 0、计划启动时间有效且不重复、区域时长不超过 120 分钟、低流量阈值小于高流量阈值，以及启用计划至少有一个启动时间和一个有效区域时长。

## 硬件安全控制

- 固件进入业务初始化后的第一项硬件动作是将 `DRV_SD` 置为关断、泵信号关闭、6 路阀 PWM 为 0。
- 阀与 GPIO 的对应关系严格使用定稿网表：区域 1 至区域 6 依次为 GPIO 33、32、26、25、14、27。
- `BoardHardware` 是唯一能够写阀、`DRV_SD` 和泵信号的模块，并强制同一时刻最多一个阀打开。
- 阀打开后立即输出 100%；独立计时达到吸合时间后切换为配置的维持占空比，不阻塞泵启动和业务循环。
- PWM 使用固定 8 位分辨率，所有阀通道使用相同频率。空闲时频率变更必须重新配置全部通道并再次确认输出为 0。
- 流量脉冲使用中断累计单调递增计数，业务通过无符号差值取得区间脉冲，不在运行切换时清零中断计数器。
- 任意内部错误、配置失效或保护停机都调用同一个立即安全关闭入口。

## 浇水执行模型

`WateringController` 同一时刻只持有一个 `WateringSession`，其内部状态为：

1. `Idle`：没有活动操作，输出保持关闭。
2. `Starting`：打开当前区域阀，按配置输出泵控制信号并执行泵启动延时。
3. `WaitingForFlow`：等待流量建立并执行启动阶段无流量保护。
4. `Watering`：计算区域运行时长、流量和水量并执行运行中保护。
5. `Stopping`：正常结束时先关闭泵信号，按配置延时后关阀，然后进入下一区域或结束。

状态名只用于软件内部，页面显示用户可理解的中文状态。所有计时使用 `millis()` 差值，不使用阻塞延时。

执行规则：

- 请求开始时复制区域顺序、区域时长、泵参数、阀参数和流量保护参数，后续配置修改不改变当前会话。
- 阀全功率吸合与泵启动延时并行，不额外增加区域准备时间。
- 区域运行时长从首次检测到有效流量后开始；没有建立流量时不消耗用户设置的浇水时长。
- 正常完成一个区域时使用停泵后关阀延时；未启用泵时直接关阀。
- 用户停止时立即关闭泵；启用泵时完成停泵后关阀延时，重力供水时立即关阀。
- 无流量、配置失效和内部故障不等待正常停机延时，立即关闭全部输出。
- 当前区域安全关闭后才能启动下一区域。
- 重启后不恢复会话；如果上次会话已经实际出水但未正常结束，应在启动后补记中断结果或相应事件。

## 流量监测

- 所有出水用途共用同一个 `FlowMonitor`，无流量判断只依赖原始脉冲，不依赖校准系数。
- 泵启动延时结束后进入流量建立宽限；宽限结束仍没有脉冲时开始无流量确认计时，达到确认时间立即保护停机。
- 流量已经建立后，任意连续无脉冲时间达到确认值都立即保护停机。
- 实时流量使用最近 5 秒脉冲窗口和当前脉冲/L 计算。
- 低流量和高流量只有在目标区域已经学习基准流量时启用；未学习不影响普通浇水、无流量保护、流量和水量估算。同类异常一次区域运行只记录一次，配置动作决定只报警还是停止整次浇水。
- 全部输出关闭后等待 30 秒，再按窗口检测非计划流量；报警持续期间不重复写事件，恢复一个完整窗口后解除。
- 区域切换、正常停止和故障停止都要重新建立相应监测窗口，不能沿用上一阶段计时。

## 自动调度

- 调度只使用 `Esp32BaseTime` 提供的可信 RTC/NTP 时间，不直接读写 RTC。
- 启动或时间刚变为可信时，以当前分钟初始化检查点，不触发当前分钟的计划。
- 正常运行只检查新进入的分钟，不扫描或补执行跳过的时间范围。
- 到点后先持久化“已处理”标记，再提交请求；设备忙或请求失败时仍视为本次已处理，并记录跳过事件。
- 当天已处理标记使用项目自己的 NVS 小状态保存，按本地日期和 `(计划编号, 启动分钟)` 识别，最多保存当天 32 个启动项。
- 时间向后调整时查询持久标记，保证同一启动项当天只处理一次；日期变化后清理上一天标记。
- 下一次计划时间由当前有效配置实时计算，不保存为配置。

## 校准与区域流量学习

- 校准会话只在内存保存未确认样本；页面刷新不丢失，重启后放弃未保存结果。
- 每次采样通过 `WateringController` 请求一个校准用途的单区域操作，使用原始脉冲和普通无流量保护。
- 用户输入实测水量后由 `FlowCalibrationService` 校验样本并计算脉冲/L，用户确认后更新当前流量系数。
- 区域流量学习同样提交单区域操作，使用 5 秒窗口判断稳定；建议值经用户确认后才保存为区域基准流量。
- 校准和学习不写普通浇水记录，只写必要的开始、保存、失败和放弃事件。

## 浇水记录与事件

### Flash 使用原则

- 当前 LittleFS 分区为 896 KiB，浇水记录、App Events、系统文件日志、业务配置和文件系统运行空间共用该分区，容量设计必须保留足够空闲空间，不能把分区写满。
- App Events 容量调整为 512 条，约占 94 KiB；系统文件日志默认只记录错误，使用 2 个 32 KiB 分段，最多占 64 KiB。
- 浇水记录使用不超过 576 KiB 的固定空间预算，其余空间留给配置、文件系统元数据、临时文件和安全余量。
- 不按秒、流量窗口或页面刷新写 Flash。流量计数、实时流量、运行计时和未确认校准样本只保存在内存。
- 配置只在用户确认保存时写入；浇水记录在一次会话结束时追加一次；调度防重复和中断识别只保存无法从其它数据恢复的最小状态。
- 普通浇水的开始、过程和正常完成不重复写事件；浇水记录保存业务过程，App Events 只保存报警、跳过、配置变化、校准结果、存储失败和中断等必要信息。
- 所有长期数据采用顺序追加和分段轮换，避免频繁重写整个文件；达到空间预算后覆盖最早记录，使擦写分散到记录区域。

### 浇水记录

- 记录使用 `/app/irrigation/records/` 下的紧凑二进制分段文件，不使用 JSON 长期保存；页面展示和 CSV 导出时再转换格式。
- 每条记录包含版本、长度、编号和 CRC，尾部再次保存记录长度，支持掉电后识别不完整记录和从最新记录反向分页。
- 分段按顺序写满后创建下一段；总空间达到 576 KiB 时删除最早完整分段，不进行全量文件重写。
- 只要会话至少一个区域检测到有效流量，就在结束时写一条记录；从未建立流量的失败尝试只记录事件。
- 记录包含编号、来源、计划信息、开始和结束时间、总结果、结束原因，以及各区域的计划时长、实际时长、脉冲数、估算水量和结果。
- 打开第一个阀前在项目 NVS 中保存精简的活动会话标记，首次检测到流量时只更新一次，正常写完记录后清除；不按流量脉冲或计时进度持续写入。
- 启动时发现活动会话标记，必须记录重启中断事件；标记表明已经出水时补写一条中断记录。无法从掉电现场恢复的时长或脉冲明确记为未知，不能伪造数值。
- 页面提供倒序分页、CSV 导出和确认后清空。记录保存失败必须写错误事件并在状态页提示存储异常。

紧凑记录预计约为 `32 + 16 × 实际区域数` 字节。在 576 KiB 预算内，理论容量约为 4500 条六区域记录、7300 条三区域记录或 12000 条单区域记录，实际数量还会受分段尾部空余和文件系统开销影响。按每天 1 条约可保留 12–32 年，按每天 3 条约可保留 4–11 年。当前内部 Flash 不能诚实保证所有使用方式都保存 1–2 万条；如果必须永久保留更多年限，应定期导出 CSV 到外部存储。

### 事件

- 事件统一使用 `Esp32BaseAppEventLog`，本项目不实现事件文件、分页、筛选、导出或清除。
- `source` 固定使用 `watering`、`schedule`、`flow`、`calibration`、`config`；`type` 使用稳定的英文动作或报警名；页面负责中文显示。
- `reason` 使用集中定义的原因代码，`object` 标识控制器、计划或区域，三个数值字段只保存该类事件约定的关键数值。
- 所有事件通过 `IrrigationEvents` 写入，避免不同模块使用不同字段表达同一件事。

## Web 设计

- 顶层页面固定为总览、运行、计划、区域、浇水记录和浇水设置。
- 页面使用 `Esp32BaseWeb` 的认证、导航、布局、表单和响应能力，以服务端渲染为主，只为运行状态使用小型 JSON 接口定时刷新，不建设单页应用框架。
- 所有修改请求使用 POST，先执行基础库认证和请求来源检查，再进行业务校验；成功后重定向到结果页面，避免刷新重复提交。
- Web 只能读取 `WateringStatus` 和查询模型，命令只能调用公开业务入口，不能持有配置引用或直接操作存储和硬件。
- 配置可以在浇水期间保存，但当前会话继续使用开始时快照；页面必须明确提示新配置从下一次操作生效。
- 流量计校准从浇水设置进入，区域流量学习从区域详情进入，事件使用基础库入口。
- 当前不实现 LCD2004 和 4 个按钮；未来本地操作复用相同业务命令和状态快照。

## 维护操作责任

- 软件不拦截 `Esp32Base` 提供的 OTA、重启和 LittleFS 格式化操作。
- 用户负责先查看运行状态，只在没有浇水、校准或流量学习时执行维护操作。
- OTA 或重启会中断当前操作，设备重新启动后保持所有输出关闭且不恢复任务。
- 格式化 LittleFS 会删除业务配置、浇水记录和基础库文件；执行前应先导出需要保留的记录，格式化后按首次启动流程重新配置。

## 实现与验证顺序

1. 修正定稿引脚并完成上电、空闲、开阀、维持和安全关闭测试。
2. 实现当前配置结构、默认值、完整校验、原子保存和失效安全模式。
3. 实现 `WateringController`、`FlowMonitor` 和普通手动浇水，先覆盖停止与无流量保护。
4. 实现计划调度、防重复、事件和浇水记录。
5. 实现流量计校准、区域流量学习和低/高流量判断。
6. 按确定的信息结构完成 Web 页面，最后做完整异常场景与实机验证。

纯业务规则尽量使用不依赖 Arduino 的代码并做本机测试。每个代码阶段至少通过 `pio run -d firmware`；硬件相关验收必须记录所用电磁阀、流量计、供水方式、电压、电流、温升和测试结果。
