# 03 软件设计

## 设计结论

软件只依据当前产品方案、定稿 PCB 和已经核实的 `Esp32Base` 公共能力实现。

设计保持四个边界：

1. 硬件适配：安全控制 PCB 固定输入输出。
2. 浇水业务：执行、流量保护、调度、校准和学习。
3. 数据存储：业务配置、调度防重复标记和浇水记录。
4. Web 接口：页面展示、输入校验和业务命令入口。

业务规则只在业务层实现一次。Web、自动调度、校准和将来的本地按钮都调用同一套能力，不能直接控制 GPIO 或读写业务文件。

## 模块职责

| 模块 | 唯一职责 |
| --- | --- |
| `IrrigationApp` | 依赖装配、固定启动顺序、业务就绪状态和主循环编排 |
| `BoardHardware` | 引脚初始化、阀驱动、泵控制信号、流量脉冲计数和全部输出安全关闭 |
| `IrrigationConfig` / `IrrigationConfigStore` | 当前配置结构、默认值、完整校验、原子保存和运行时配置快照 |
| `WateringController` | 当前唯一活动操作、区域顺序、非阻塞执行、停止和结果汇总 |
| `FlowMonitor` | 流量建立、无流量保护、流速与水量计算、低/高流量和关阀后水流判断 |
| `WateringScheduler` | 管理自动浇水总控状态，在可信时间下检查计划、避免重复并提交自动浇水请求 |
| `FlowCalibrationService` | 管理流量计校准的内存样本和稳态直线拟合；出水与区域基准流量学习仍交给 `WateringController` |
| `WateringRecordStore` | 基于 `Esp32BaseRecordStore` 保存、最新优先分页和按 ID 读取浇水记录，并向 System 页登记当前版本 Store |
| `IrrigationEvents` | 将业务事件统一写入 `Esp32Base` App Events |
| `IrrigationWeb` | 注册业务页面和接口，将请求转换为业务命令 |

不单独设置只做简单转发的区域服务、计划服务或通用仓储层。配置修改由 `IrrigationConfigStore` 的明确业务方法完成。

## 工程结构

新工程保持一层清晰业务目录：

```text
firmware/
├── platformio.ini
├── src/
│   ├── main.cpp
│   └── irrigation/
│       ├── IrrigationApp.*
│       ├── IrrigationTypes.h
│       └── 各职责模块的同名 .h/.cpp
└── test/
    └── 按核心行为划分的本机测试
```

`main.cpp` 只提供 Arduino `setup()` / `loop()` 并调用 `IrrigationApp`。文件名与主要类型一致；不建立只有一个文件的层级目录，也不创建含义宽泛的 `common`、`manager`、`helper` 或通用仓储框架。测试按状态机、调度、配置和记录等行为组织，不为每个类机械复制目录。

## 启动顺序与执行上下文

业务采用 Arduino `loopTask` 单任务协作模型，不增加业务 FreeRTOS 任务。流量脉冲中断只递增计数器，不做日志、存储、业务判断或硬件切换。

启动顺序固定为：

1. `setup()` 第一项直接初始化 `BoardHardware` 并关闭泵信号、全部阀 PWM 和阀驱动总使能。
2. 初始化 I2C，并将定稿的 DS3231 总线交给 `Esp32BaseRtc`。
3. 从固件配置读取默认 Web 账号密码并调用 `Esp32BaseWeb::setDefaultAuth()`，注册业务 Web 路由和文件系统格式化后的业务回调。
4. 调用 `Esp32Base::begin()`，由基础库初始化时间、文件系统、事件、WiFi 和 Web 等公共能力。
5. 加载业务配置、自动调度小状态并初始化浇水记录 Store。
6. 只有所需配置校验通过后才进入业务就绪状态；否则保持全部输出关闭，只开放诊断和重新生成配置入口。

正常循环每轮推进一次业务状态机并调用一次 `Esp32Base::handle()`。停止命令在业务入口立即关闭输出；所有 handler 必须快速返回，不在 handler 中执行校准、等待出水或其它长时间流程。

基础库构建以 FULL profile 作为已验证的 Web、OTA、文件系统和运行时能力组合，显式启用 DS3231 RTC、Record Store 和 App Events。App Events 使用基础库默认 100 KiB 预算；系统文件日志保持基础库默认的 4 个 32 KiB 分段并将默认模式设为 WARN，时区为 UTC+8 且无夏令时，并使用基础库的 classic ESP32 4MB OTA balanced 分区表。实际建工程时再根据链接结果裁掉没有被本项目使用的基础能力，不能改变这些已确认的存储和时间参数。

## 核心数据

### 持久配置

- `ValveDriveConfig`：全功率吸合时间、水路切换间隔、PWM 频率、维持占空比。
- `PumpConfig`：是否启用、启动延时、停泵后关阀延时。
- `FlowMeterConfig`：保存启动脉冲、估算启动水量和稳态流量系数。系数以放大 100 倍的定点整数保存。
- `FlowProtectionConfig`：无流量、低/高流量和关阀后水流参数及动作。
- `TimeSafetyConfig`：RTC 时间倒退阈值和存活检查点间隔。
- `ZoneConfig`：固定编号、启用状态、名称和基准脉冲速率；基准以 `P/s × 10000` 的整数保存，0 表示未设置，页面使用当前稳态流量系数动态换算为三位小数的 `L/min`。
- `WateringPlan`：固定编号、`configured`、名称、`scheduleEnabled`、每日启动时间和各区域运行时长；`configured` 明确表示计划槽位是否存在，`scheduleEnabled` 只表示是否自动执行。

固定硬件引脚、区域数量、计划数量、单区域最长运行时间和 UTC+8 时区是产品常量，不作为用户配置。

当前灌溉配置结构版本为 4。版本 4 增加必填的水路切换间隔，版本 3 及更早结构不读取、不转换；检测到旧配置时保持全部输出关闭并明确要求重新配置，不能静默补默认值覆盖已有现场配置。

### 运行数据

- `WateringRequest`：一次请求的来源、用途和区域时长。
- `WateringStep`：一个区域及其本次目标运行时长。
- `WateringSession`：请求开始时生成的配置与步骤快照、当前区域、计时、脉冲和结果。
- `IrrigationStatus`：供 Web 和未来本地界面读取的统一只读状态，包含业务就绪状态、当前会话、流量、自动浇水总控和存储故障，不暴露可修改的内部对象。
- `AutomaticWateringState`：自动浇水总控模式和可选的恢复时间。
- `DeviceAliveCheckpoint`：最近一次周期确认在线的可信时间；只有长时间没有其它带可信时间的业务记录或事件时才写入。
- `IrrigationCommandResult`：业务命令结果，包含稳定错误码和用户可读说明，避免用一个 `bool` 混淆未就绪、忙、参数错误、对象不存在、时间不可用和存储失败。
- `WateringRecord`：一次已经进入执行流程的普通浇水最终业务记录，包括尚未建立水流即失败的尝试。
- `ZoneWateringRecord`：一次会话中单个区域的计划值、实际值和结果。

`WateringSource` 表示手动区域、自动计划或手动立即执行计划；`WateringPurpose` 表示普通浇水、流量计校准或区域流量学习。结果使用 `WateringResult`，停止原因使用 `WateringStopReason`。`AutomaticWateringMode` 只使用 `Enabled`、`PausedIndefinitely` 和 `PausedUntil` 三种明确状态，`PausedUntil` 以 `resumeAtEpoch` 保存绝对恢复时刻。这些概念保持分开，避免使用脱离上下文后含义不清的通用 `Operation`、`Result` 或 `Reason` 名称。

## 配置模型与存储

- 水路和浇水计划等嵌套业务对象使用权威文件 `/app/irrigation/config.json`；阀驱动、水泵、流量保护、时间和存储等标量系统参数使用 `Esp32Base` App Config 和其 NVS 存储。
- 自动浇水总控、调度防重复标记和 `DeviceAliveCheckpoint` 是项目 NVS 小状态，不混入计划定义或业务记录。Web 认证由 `Esp32Base` 管理。
- `Esp32Base` App Config 管理可独立校验的标量系统参数；它不管理 6 条水路和 8 个计划的嵌套数据。业务对象继续使用 `Esp32BaseFs`，两类配置不提供重复保存入口。
- 配置包含结构版本和单调递增的内容修订号。结构版本不匹配时进入配置不可用状态，不做自动转换。
- 加载时同时检查当前文件、临时文件和备份文件，选择能够完整解析并通过结构、业务校验且修订号最高的当前版本副本。
- 三个文件都不存在时生成默认配置；只要其中存在文件但没有有效副本，就不能用默认值覆盖现场内容。
- 配置不可用时所有输出保持关闭，页面显示具体错误，并允许用户明确确认后重新生成配置。
- 保存时先构造完整副本，校验所有字段和跨字段关系，再写临时文件、回读校验并原子替换。掉电恢复只处理当前结构的临时文件或备份文件。
- Web 表单携带读取时的配置修订号；修订号已经变化时拒绝旧页面覆盖并提示刷新。保存成功后修订号加一，保存失败时运行配置和修订号都不改变。
- 保存成功后一次性替换内存配置。已经开始的操作继续使用自身快照，修改只影响之后的操作。
- PWM 频率只在空闲时重新配置全部阀通道；运行中保存的新频率从下一次出水开始生效。

每个字段先按 `02-业务规则与安全.md` 的有效范围校验，再校验跨字段关系：区域和计划编号与固定槽位一致，名称是有效 UTF-8，所有开启自动执行的启动项没有全局时间冲突，区域时长为 0 或 1–120 分钟，低流量阈值低于 100% 且高流量阈值高于 100%，开启自动执行的计划至少有一个启动时间和一个当前启用区域的有效时长。

## 硬件安全控制

- 固件进入业务初始化后的第一项硬件动作是将 `DRV_SD` 置为关断、泵信号关闭、6 路阀 PWM 为 0。
- 阀与 GPIO 的对应关系严格使用定稿网表：区域 1 至区域 6 依次为 GPIO 33、32、26、25、14、27。
- `BoardHardware` 是唯一能够写阀、`DRV_SD` 和泵信号的模块，并强制同一时刻最多一个阀打开。
- 多水路切换采用先断后通：先停止泵并关闭上一水路，全部输出关闭达到水路切换间隔后，才打开下一水路并启动泵，避免两个阀的启动电流重叠。
- 阀打开后立即输出 100%；独立计时达到吸合时间后切换为配置的维持占空比，不阻塞泵启动和业务循环。
- PWM 使用固定 8 位分辨率，所有阀通道使用相同频率。空闲时频率变更必须重新配置全部通道并再次确认输出为 0。
- 流量脉冲使用中断累计单调递增计数，业务通过无符号差值取得区间脉冲，不在运行切换时清零中断计数器。
- 任意内部错误、配置失效或保护停机都调用同一个立即安全关闭入口。

## 浇水执行模型

`WateringController` 同一时刻只持有一个 `WateringSession`，其内部状态为：

1. `Idle`：没有活动操作，输出保持关闭。
2. `StartingZone`：打开当前区域阀，按配置输出泵控制信号并执行泵启动延时。
3. `WaitingForFlow`：等待流量建立并执行启动阶段无流量保护。
4. `WateringZone`：计算区域运行时长、流量和水量并执行运行中保护。
5. `StoppingZone`：正常结束时先关闭泵信号，按配置延时后关阀，然后进入下一区域或结束。
6. `SwitchingZone`：多水路运行时保持泵和全部阀关闭，等待水路切换间隔后再启动下一水路。

状态名只用于软件内部，页面显示用户可理解的中文状态。所有计时使用 `millis()` 差值，不使用阻塞延时。

正常状态迁移为 `Idle -> StartingZone -> WaitingForFlow -> WateringZone -> StoppingZone`；多区域操作从 `StoppingZone -> SwitchingZone -> StartingZone` 进入下一水路。用户停止进入安全停机路径；保护、配置失效和内部故障从任意活动状态先立即关闭全部输出，再形成结果、记录和事件，不能为了写 Flash 或生成页面而推迟关阀。

执行规则：

- 请求开始时复制区域顺序、区域时长、泵参数、阀参数和流量保护参数，后续配置修改不改变当前会话。
- 阀全功率吸合与泵启动延时并行，不额外增加区域准备时间。
- 区域运行时长从首次检测到有效流量后开始；没有建立流量时不消耗用户设置的浇水时长。
- 正常完成一个区域时使用停泵后关阀延时；未启用泵时直接关阀。
- 非最后一个区域关阀后进入水路切换间隔，间隔期间不建立下一水路的流量基线，也不把余流脉冲计入下一水路。
- 用户停止时立即关闭泵；启用泵时完成停泵后关阀延时，重力供水时立即关阀。
- 无流量、配置失效和内部故障不等待正常停机延时，立即关闭全部输出。
- 当前区域安全关闭后才能启动下一区域。
- 重启后不恢复会话。突然断电可能使尚未结束的浇水没有最终业务记录；系统保留基础库启动日志和掉电时间估算，但不为无法确认的区域进度、脉冲或水量补造记录。
- 对外启动入口统一校验业务已就绪、当前空闲、区域仍启用、至少一个有效时长和本次快照完整。自动调度、Web 以及未来本地按钮不能绕过这些校验。
- 停止命令停止当前整次会话而不是只跳过当前区域；空闲时收到停止命令返回“没有正在进行的浇水”，不产生停止事件。
- 当前区域的原始脉冲从开阀前计数到实际关阀或安全关闭，覆盖流量建立和停泵后关阀阶段；实际浇水秒数只计算从流量建立到正常浇水结束、用户停止或故障发生的时间。
- 会话结束后形成一次不可变结果摘要，包含请求中的全部区域；尚未运行的后续区域明确标记为 `NotStarted`。记录或事件处理完成前不接受下一次请求，避免结果被覆盖。
- 估算水量使用本次会话开始时复制的脉冲/L 系数和原始脉冲计算并保存为 mL。计算使用 64 位整数完成舍入，溢出时标记结果受限，不写入伪造数值。

## 流量监测

- 所有出水用途共用同一个 `FlowMonitor`，无流量判断只依赖原始脉冲，不依赖校准系数。
- 泵启动延时结束后开始一个完整的流量建立超时；超时仍没有有效脉冲时立即保护停机，不再拆分宽限和确认两个串联计时器。
- 流量已经建立后，任意连续无脉冲时间达到运行中无流量超时都立即保护停机。
- 实时流量使用最近 5 秒脉冲窗口和当前脉冲/L 计算。
- 低流量和高流量只有在目标水路已经学习有效基准脉冲速率时启用；当前窗口和基准直接按原始脉冲速率比较，校准系数只用于页面流量和水量换算。连续异常按窗口实际时长累计，任一正常窗口清零；异常确认后立即记录，连续正常达到恢复确认时间后重新布防，同次运行再次确认异常时形成新的事件，未恢复期间不重复记录。配置动作决定只报警还是停止整次浇水。
- 全部输出关闭后按配置等待关阀后检测延时（默认 30 秒），再按窗口检测关阀后水流；完成第一个有效窗口前状态未知，报警持续期间不重复写事件，恢复一个完整窗口后解除并记录恢复事件。一次业务循环内完成关阀、复位监测器和观察状态时必须复用同一个 `millis()` 快照，避免无符号时间差回绕而提前开始检测。
- 区域切换、正常停止和故障停止都要重新建立相应监测窗口，不能沿用上一阶段计时。

## 自动调度

- 调度只使用 `Esp32BaseTime` 提供的可信 RTC/NTP 时间，不直接读写 RTC。
- 启动时间按本地固定 UTC+8 的 `0..1439` 分钟保存；暂停恢复时间按 UTC epoch 保存。运行时统一从同一个时间快照计算本地日期和分钟，不能混用本地 epoch 与 UTC epoch。
- 应用层要求本地日期不早于 2026-01-01。RTC 状态异常或早于该日期且没有 NTP 时间时，自动调度不可用。
- RTC 来源的当前时间比本次启动内已经观察到的最大可信时间，或上次可信浇水记录、事件、检查点时间倒退超过配置阈值时，进入“RTC 时间倒退”状态；默认阈值 5 分钟。NTP 成功校时后接受 NTP 时间并解除该状态。
- 每次检查计划前先判断 `AutomaticWateringState`。`PausedIndefinitely` 不自动恢复；`PausedUntil` 只有在时间可信且达到 `resumeAtEpoch` 后才切换为 `Enabled`。
- 定时暂停只在当前时间可信且没有 RTC 倒退时保存最终的 `resumeAtEpoch`；小时数、12/24/48 小时和明后天早上 6 点都只是页面生成恢复时间的快捷输入，不提交或保存为独立状态。暂停、手动恢复和到期自动恢复各最多写一次 NVS 并记录一次事件。
- 暂停期间遇到的到点项仍先写入当天已处理标记，然后跳过，不提交浇水请求且不逐项写跳过事件。恢复后不扫描暂停期间的计划。
- 启动或时间刚变为可信时，以当前分钟初始化检查点，不触发当前分钟的计划。
- 正常运行只检查新进入的分钟，不扫描或补执行跳过的时间范围。
- 到点后先持久化“已处理”标记，再提交请求；标记写入失败时禁止启动本次自动浇水，避免重启后重复执行。设备忙或请求失败时仍视为本次已处理，并按启动结果及当前任务类型记录用户可理解的具体跳过原因。
- 每个计划的 4 个启动时间使用固定槽位；一个本地日期的已处理状态用 `(计划编号, 启动槽位)` 对应的 32 位掩码表示，不把可修改的具体分钟作为身份。
- 调度只保存当前本地日期和前一日期的处理掩码。超过 RTC 倒退阈值的回拨已经由时间安全状态阻止自动调度，两天数据足以处理阈值内的同日和跨午夜回拨。该状态连同版本和校验信息保持在基础库 256 字节 blob 限制内。
- 当前分钟只有在本地分钟按正常连续顺序进入时才参与匹配；启动、时间首次可信或分钟发生不连续跳变时，仅以目标分钟重新建立检查点，不运行恰好落入该分钟的计划。
- 修改计划、开启自动执行或恢复自动浇水时，以当前分钟重新建立检查点，不因为配置恰好包含当前分钟而立即启动。
- 下一次计划时间由调度层的只读 `NextAutomaticWatering` 查询模型根据当前有效配置、UTC+8 时间和暂停状态实时计算，不保存为配置。正常状态从当前分钟之后查找；定时暂停从恢复分钟之后查找，恢复所在分钟不触发；每日计划最多向后检查一个完整周期。
- 计划自身的 `scheduleEnabled` 与自动浇水总控相互独立：只有总控为 `Enabled` 且计划开启自动执行时才可由调度启动；手动执行计划不检查这两个条件。

## 校准与区域流量学习

- 校准会话只在内存保存样本、待填写状态和结果预览；页面刷新不丢失，重启后清空本次会话，但已经应用的设备参数继续保留。
- 每次采样通过 `WateringController` 请求一个校准用途的单区域操作，使用原始脉冲和普通无流量保护。
- `CalibrationStabilityDetector` 使用可配置的原始脉冲速率窗口识别稳态；`WateringController` 记录启动和稳态阶段，并只在停止命令后仍有脉冲时保留关闭尾水诊断，保证阶段脉冲与完整脉冲一致。
- `FlowCalibrationService` 在 RAM 中保存最多 10 个样本。只有已识别稳态的正常样本填写实测总水量并参与拟合；自动生成启动脉冲、估算启动水量和稳态流量系数三个单值，用户确认后统一保存，三项也可手工修改。水路不拆分参数。
- 区域流量学习同样提交单区域操作，从开阀开始执行 10 分钟硬上限。流量建立后每 5 秒生成一个原始脉冲速率窗口，页面保留最近 10 个、判定只使用最近 5 个；所有判定窗口非零，且最大最小差不超过 10% 或一个脉冲量化误差时确认稳定。建议值使用五个窗口的总脉冲和总时间计算，保存为 `baselinePulseRateX10000`。手工输入也换算到同一字段并要求三位小数 `L/min` 精确回算，不保存显示专用副本。
- 校准和学习不写普通浇水记录；只有用户确认并成功保存最终结果时才写信息事件，未得到有效结果或用户放弃时只在当前页面提示。

## 浇水记录与事件

### Flash 使用原则

- 当前 LittleFS 分区为 896 KiB，浇水记录、App Events、系统文件日志、业务配置和文件系统运行空间共用该分区，容量设计必须保留足够空闲空间，不能把分区写满。
- App Events 使用基础库默认 100 KiB 预算，估算容量约 2029–2113 条；系统文件日志使用基础库默认的 4 个 32 KiB 分段，最多占 128 KiB，本项目默认模式为 WARN。
- 浇水记录使用 512 KiB 固定逻辑预算，并要求追加后至少保留 32 KiB 文件系统剩余空间；浇水记录、App Events 和系统文件日志的现有容量预算均保持不变。
- 不按秒、流量窗口或页面刷新写 Flash。流量计数、实时流量、运行计时和未确认校准样本只保存在内存。
- 配置只在用户确认保存时写入；浇水记录在一次会话结束时追加一次；调度防重复状态只在自动计划到点时保存。
- 自动浇水总控只在用户改变状态或定时暂停到期时写一次，不按秒更新 Flash。
- 存活检查点默认 12 小时且可配置，0 表示关闭。任一带可信时间的浇水记录或 App Event 都会重新计算间隔，期间不再额外写检查点；安静设备最多约 730 次/年，正常产生业务数据的设备更少。
- 普通浇水的开始、过程和正常完成不重复写事件；浇水记录保存业务过程，App Events 只保存报警、跳过、配置变化、校准结果、存储失败和中断等必要信息。
- 浇水记录和 App Events 使用基础库的顺序追加、完整分段轮换和损坏恢复，不在项目内重复实现文件管理。

### 浇水记录

- 每种完整业务事实使用独立 Store；当前只有一个 `recordTypeName=watering`、`storeVersion=3` 的浇水 Store，路径和分段由 `Esp32BaseRecordStore` 管理。
- Store 是 `IrrigationApp` 单例的长期成员。在 `Esp32Base::begin()` 成功后先用当前 v3 定义调用 `begin()`，无论初始化是否因已有存储损坏而失败，都继续尝试向 `Esp32BaseWeb` 登记同一个底层对象，并分别检查两个结果。只登记当前使用的 v3，不登记历史版本或 App Events。v1、v2 与 v3 使用不同目录，升级不会覆盖或迁移旧记录；旧记录不再由业务页面读取，是否删除由用户另行决定。
- 同一 Store 的 `captureStartTime`、追加、读取、System 清空和格式化后 reload 全部串行发生在 Arduino loop/system task；其它任务不得直接访问。未来若其它任务产生记录，只能通过业务队列交给该任务保存。
- 浇水控制器在任务启动时复制本次水路时长、阀门、水泵、流量计、保护参数和各水路基准；运行中保存计划、水路或系统参数只影响后续任务，不改变当前任务。PWM 频率若在浇水中修改，保存后延迟到当前任务安全结束再应用。
- 基础库保存记录 ID、完成时间、boot ID、完成 uptime、整次历时和 CRC。本项目只使用显式小端偏移编码固定 208 字节业务 payload，不写 JSON、字段名、指针、可变对象或 C++ 结构体内存。
- `maximumStoreBytes` 为 512 KiB，达到预算后由基础库淘汰最早完整分段；不提供单条修改或删除。
- 普通浇水请求一旦被控制器接受并进入执行流程，就在会话结束时写一条记录，不以是否检测到有效流量或脉冲数是否大于零作为保存条件。启动前因参数、忙碌或状态不允许而拒绝的请求不写浇水记录；自动计划未启动由计划跳过事件表达。校准和区域流量学习继续排除在普通浇水记录之外。
- payload 前 4 字节依次保存来源、计划 ID、总结果和结束原因。随后按区域 1 至区域 6 的固定位置各保存 34 字节：区域结果、flags、计划秒数、实际浇水秒数、总脉冲数、历史估算水量、有效浇水阶段整段平均流量、当时基准流量、末段平均/最小/最大流量，流量单位均为 mL/min；flags 同时记录水量封顶、低流量、高流量、基准可用、末段可用和末段稳定。位置已经表达区域 ID，未包含区域写全零。
- 整次历时读取基础库元数据；计划总时间、实际浇水总时间、总脉冲数、历史估算总水量和按实际浇水时长加权的平均流量在读取时从区域明细计算，不重复保存。脉冲和水量使用 64 位整数汇总。
- 计划名称和区域名称不进入历史记录；页面按稳定 ID 读取当前名称，计划已删除时显示固定编号和“已删除”。
- 页面可以使用实际脉冲数和当前稳态流量系数给出“按当前系数重算”的参考水量，但不能修改已经保存的历史估算水量。
- 手动浇水允许在时间不可信时使用。此时记录保存 `bootId + uptimeSec` 和单调运行时长，不伪造日期；同一次启动内后来获得可信时间时可以通过 `Esp32BaseTime` 解析，无法解析的历史记录明确显示相对时间。
- 浇水页面负责业务字段解码、倒序分页、列表和详情；基础库不理解浇水 payload。本产品不提供浇水记录导出或业务页面清空，清空只使用 System 页的 `Clear Business Records`。成功清空后沿用原 Store 对象，只重新读取状态、计数和分页数据，不重新 `begin()`、重建对象或重置记录 ID。记录保存失败必须写错误事件并在状态页提示存储异常。

固定 payload 为 `4 + 34 × 6 = 208` 字节，加上基础库 24 字节公共元数据后每条槽位为 232 字节。按基础库当前 512 KiB 预算和分段算法计算的定义容量为 2254 条；由于只淘汰完整分段，轮换期间预计保留约 2184～2254 条。目标设备仍应读取 `StoreStatus` 复核，页面始终以设备实时状态为准。

### 事件

- 事件统一使用 `Esp32BaseAppEvents`，本项目不实现事件文件、轮换、导出或清除。
- 本项目集中定义稳定的 `eventCode`、`reasonCode`、`objectId`、两个数值和 flags，并在业务页面映射中文含义；不把文字或字段名写入事件。
- 当前离散事件只覆盖普通浇水异常停止、确认后的低/高流量、自动浇水暂停与恢复、计划跳过、重要设置修改、校准结果保存、水路基准流量保存或清除、一次具体浇水记录保存失败和运行中的调度状态保存失败。普通浇水异常停止和低/高流量事件在 flags 中保存手动/自动来源及自动计划 ID，页面结合当前配置显示计划和水路名称；事件详情提供浇水记录入口，完整执行过程仍以浇水记录为准。低/高流量达到连续确认时间后先在 RAM 锁存检测流量和当时基准，控制器确认输出关闭后再写入 App Events，避免文件写入阻塞运行中的安全控制；同一水路同一类型每次浇水最多写一条，是否因此停止用 flag 表达。普通完成和用户主动停止由浇水记录表达，不重复写事件；校准和水路学习没有得到有效结果时只在当前页面提示。
- 所有事件通过 `IrrigationEvents` 写入，避免不同模块使用不同字段表达同一件事。
- `IrrigationEvents` 是 `IrrigationApp` 长期持有的单个对象，内部长期持有硬件时钟不可用、设备时间不可用、设备时间倒退和关阀后水流异常四个 `ConditionStateTracker`。条件状态只调用 `observeConditionState()`，离散事实只调用 `appendDiscreteEvent()`，并完整处理基础库返回结果。
- RTC 由业务每 60 秒实际刷新一次，必须根据 `Esp32BaseRtc::status()` 或 `isAvailable()` 判断结果，不能把 `refresh()` 返回值当成硬件可用。硬件时钟连续失败或成功约 60 秒后确认；设备连续 5 分钟没有可信时间后确认不可用，连续恢复 30 秒后解除；RTC 倒退在调度器进入和离开对应时间状态时形成事件。
- 关阀后水流的延时、窗口和阈值继续由 `UnexpectedFlowMonitor` 判断，事件条件本身不重复延时。浇水、校准、学习以及监测器尚未完成首个有效窗口时上报未知，不能误报恢复。
- 浇水记录 Store 初始化、登记、重载，调度状态启动读取，检查点、配置和 App Events 自身故障只进入当前状态和系统日志，不建立持续事件。格式化文件系统后基础库先重载已登记的业务 Store；若浇水记录 Store 状态不可读或不可写，业务层必须用当前定义重新 `begin()` 并核验最终状态，不能把一次重载失败永久保留为故障。一次实际浇水记录丢失和运行中的调度状态保存失败仍记录离散错误事件。
- 事件写入失败不回滚配置、调度命令或安全动作；只设置事件存储故障，并在故障状态变化时写一次系统日志，不在循环中自动 reload，也不尝试用 App Events 记录自身故障。
- 完整格式化 LittleFS 并成功重建 App Events 后，业务层调用基础库公开接口清除全部持续条件的已确认状态，将 4 项页面状态恢复为未知并立即重新观察。仍然存在的条件按原确认时间写入新事件，使新事件历史能够解释当前异常。单独清空 App Events 历史时不重置条件状态。
- 当前使用 App Events v2，不读取或迁移旧版事件；安装前明确清空旧 App Events 或经用户确认格式化 LittleFS。条件 ID 1～4 是跨重启持久化协议，后续不能在未清理原条件状态时改成其它含义。
- 每次启动及实际复位原因使用基础库系统日志，不重复写入 App Events。应用读取检查点并结合本次启动时间估算可能断电范围；软件重启、OTA、看门狗等不能描述成确定停电。
- 时间不可信时不生成停电时间估算。

## Web 设计

- 顶层入口固定为首页、计划、水路、记录、事件和设置。首页同时承担空闲总览和普通浇水运行控制，不设置独立浇水页。
- 页面使用 `Esp32BaseWeb` 的认证、导航、布局、表单和响应能力，以服务端渲染为主，只为运行状态使用小型 JSON 接口定时刷新，不建设单页应用框架。空闲首页约每 5 秒局部更新关阀后水流监测，页面隐藏时降为约 15 秒；普通浇水约每秒更新计时和状态，流量曲线仍只追加真实的 5 秒采样点。只有空闲与运行布局互相切换时允许整页重载。
- 固件配置提供默认 Web 账号密码，应用在 `Esp32Base::begin()` 前调用 `Esp32BaseWeb::setDefaultAuth()`。基础库已保存的认证优先于固件默认值；用户登录后直接使用基础库 Web Auth 页面修改。本项目不生成或保存另一套密码，也不实现认证页面。
- 所有修改请求使用 POST，先执行基础库认证和请求来源检查，再进行业务校验；成功后重定向到结果页面，避免刷新重复提交。
- Web 只能读取 `IrrigationStatus` 和查询模型，命令只能调用公开业务入口，不能持有配置引用或直接操作存储和硬件。
- 配置可以在浇水期间保存，但当前会话继续使用开始时快照；页面必须明确提示新配置从下一次操作生效。
- 流量计校准和区域流量学习使用独立隐藏路由并从水路设置进入；校准页确认结果后把启动脉冲、估算启动水量和稳态流量系数三项写入 NVS，其中稳态流量系数进入通用系统参数并参与流量与水量换算，另外两项只由校准页统一编辑和回显。水路页只显示紧凑维护信息和校准入口，不提供系统参数页跳转。业务事件页面使用基础库存储与读取接口，由项目统一映射简洁中文标题、说明和详情；基础库原始数字页面只保留在系统工具区。
- 空闲首页读取 `Esp32BaseTime::Snapshot`、`AutomaticWateringState`、`NextAutomaticWatering` 和最新浇水记录，主状态区以轻量两行信息显示秒级设备时间、自然日期及 RTC/NTP 来源，不重复显示固定的 UTC+8 时区标签，并以当前业务状态、关阀后水流监测、下一次自动浇水和最近一次浇水为主要信息；不增加第三张报告卡或完成弹窗。下一次卡片突出计划名称，并从计划快照计算本次启用水路数、预计总时长和最多三项水路时长摘要。最近一次卡片使用记录的来源、计划 ID、开始与完成时间、整次结果、停止原因和逐路结果生成面向用户的说明，展示计划浇水、实际浇水和历史估算水量；完成但带高低流量报警使用警告状态，零水流失败显示尝试水路、计划浇水、零实际浇水和后续未执行水路。状态标签已经表达结果时，正文只补充水路和原因，不重复相同状态。普通浇水运行页检测到会话结束时用 `sessionStorage` 标记本次页面切换，空闲页只对最新记录卡执行一次短暂强调，不建立需要关闭或持久确认的新业务状态。页面只用设备 epoch 加页面已打开时长驱动时钟，不依赖浏览器时区。正常关阀后水流只显示稳定的“监测已开启”或首窗阶段的“监测中”，不显示“准备中”和内部窗口倒计时；报警时状态接口提供滚动观察时长、脉冲数和按实际观察时长换算的平均流量，首页约每 5 秒更新。RTC 已确认不可用但 NTP 正常时只在时间来源下显示紧凑警告标签，不改变主状态背景，也不增加独立警告面板。容量和检查点不进入日常首页。普通浇水运行时改为完整运行控制台，顶部以紧凑摘要显示一次任务名称，并用当前水路和阶段替代重复的来源、名称和“正在浇水”堆叠；流量趋势根据 5 秒样本数动态显示实际时间范围，最多 10 分钟，单位写在图表标题中，使用相对时间横轴和四等分动态流量纵轴；存在基准流量时绘制轻量基准虚线。没有基准时显示“水流已建立”，存在基准且尚未触发报警时显示“流量监测中”，不能把尚未触发报警概括为“流量正常”。校准或流量学习运行时只显示专属页面入口。业务异常只在发生时突出显示。
- 自动浇水暂停与恢复集中在计划页，只有暂停至恢复时间和无限期暂停两个结果；小时数和常用时间只在客户端联动填写最终恢复时间。POST 由服务端校验日历、未来时间和时间可信状态。
- 手动浇水只提交最终区域时长。所有已配置计划直接以卡片列出，点击卡片立即在客户端填入当前启用区域的时长；没有计划时显示明确空状态。模板计划编号不提交，后续手工修改只改变本次时长，不改变计划和浇水记录来源。
- 浇水记录详情统一左对齐，概览突出实际浇水、历史估算水量、进入执行与计划水路数、高低流量报警数和基准覆盖数；区域明细以同一行“实际 / 计划”对照展示时长，零水流失败保留失败水路和零值，未开始的后续水路明确标记为“未开始”。当时基准流量使用独立列；流量表现展示整段平均、末段流量与稳定性，并在有基准时分别紧跟相对差值。没有基准只在基准列说明本次不进行高低流量判定；没有触发报警不能描述为“正常”。脉冲数和完整指标定义降到默认折叠的“指标与技术说明”。事件页顶部显示四项持续条件的当前状态和未恢复数量，事件业务详情优先显示用户能直接判断的数据，内部码和值折叠为技术信息。
- 记录列表始终分页读取，内部记录 ID 只用于详情读取和深链接，不作为用户可见编号；列表时间使用不含年份的短格式，按浇水任务、执行水路、执行结果、实际 / 计划和估算用水量汇总。System 页的业务记录清空、OTA、重启和 LittleFS 格式化属于已经确认由用户负责的维护操作，业务不增加平行入口或阻止接口。
- 当前只注册必要的业务页面、系统参数跳转入口和运行状态接口；详情和修改使用移动端可用的弹层，简单命令使用 `action` 区分。
- 所有业务 POST 使用 `Esp32BaseWeb::checkPostAllowed()`，采用 `POST -> 303 -> GET`；状态 API 只读。页面需要同时处理 GET/POST 时使用 `METHOD_ANY` 注册并通过 `addNavItem()` 加入导航。
- 当前不实现 LCD2004 和 4 个按钮；未来本地操作复用相同业务命令和状态快照，后续设计输入见 `01-产品与硬件边界.md`。

## 维护操作责任

- 软件不拦截 `Esp32Base` 提供的 OTA、重启和 LittleFS 格式化操作。
- 用户负责先查看运行状态，只在没有浇水、校准或流量学习时执行维护操作。
- OTA 或重启会中断当前操作，设备重新启动后保持所有输出关闭且不恢复任务。
- 格式化 LittleFS 会删除业务配置、浇水记录和基础库文件；系统不自动格式化，也不扫描整个分区判断是否首次使用。新设备首次烧录后若文件系统无法挂载，首页明确说明可能需要初始化；用户确认没有需要保留的数据后，仍通过 System 页主动格式化。已经使用过的设备挂载失败时不得仅凭该提示直接格式化，应先保留可能仍可恢复的数据。格式化成功后按首次启动流程自动创建默认配置和当前版本存储。
- System 页格式化成功后，基础库自动 reload 已登记的原 Store 对象；业务回调不再次 `begin()` 或 `reload()`。回调立即关闭输出，依据原对象的新状态同步存储故障标记，并清除项目 NVS 中的自动浇水状态、调度标记和 `DeviceAliveCheckpoint`，同时丢弃业务派生状态并重新进入首次启动流程。这些状态引用已经删除的配置和记录，继续保留会产生错误关联；WiFi 和 Web Auth 由基础库管理，不属于这组业务状态。不在格式化前增加阻止接口。

## 失败处理原则

- 启动时配置不可用：所有浇水命令和自动调度不可用，输出保持关闭。Web 必须区分文件系统不可用、已有配置无有效副本、配置写入失败和其它启动检查失败；只有文件系统不可用时说明新设备首次烧录后可能需要初始化，所有情况都保留人工判断，不自动格式化。
- 运行中配置保存失败：继续使用保存前配置，不形成半更新状态。
- 调度防重复状态无法读取或写入：停止自动调度，手动浇水仍可使用。
- 自动浇水暂停或恢复状态写入失败：命令失败并保持原状态；到期自动恢复写入失败时继续暂停。
- 存活检查点写入失败：只失去相应掉电估算精度并提示存储警告，不影响手动浇水或自动调度。
- 浇水记录或事件写入失败：先保证输出已经安全关闭，再显示持久存储故障；不得伪造“已保存”。运行中已有的有效配置仍可用于之后的手动浇水，自动调度是否继续取决于其防重复状态是否正常。
- 所有错误同时保留稳定代码和中文说明；页面文案不能把“控制信号已输出”描述成外部泵已经真实运行。

## 实现与验证顺序

1. 核对并固定定稿引脚，完成上电、空闲、开阀、维持和安全关闭测试。
2. 实现当前配置结构、默认值、完整校验、原子保存和失效安全模式。
3. 实现 `WateringController`、`FlowMonitor` 和普通手动浇水，先覆盖停止与无流量保护。
4. 实现计划调度、防重复、事件和浇水记录。
5. 实现流量计校准、区域流量学习和低/高流量判断。
6. 按确定的信息结构完成 Web 页面，最后做完整异常场景与实机验证。

纯业务规则尽量使用不依赖 Arduino 的代码并做本机测试。每个代码阶段都要在 `../../firmware/README.md` 记录并通过当时实际可用的构建和测试命令；硬件相关验收必须记录所用电磁阀、流量计、供水方式、电压、电流、温升和测试结果。

代码进入实机前至少覆盖以下自动测试：

- 正常多区域执行，以及泵启用和重力供水两种时序。
- 在每个活动状态收到用户停止、流量建立超时、运行中无流量和内部故障。
- `millis()` 回绕附近的吸合、泵延时、浇水时长和保护计时。
- 自动计划正常到点、忙时跳过、重启防重复、同日和跨日回拨、时间前跳、三种暂停模式及 NVS 写入失败。
- 时间早于 2026-01-01、RTC 倒退阈值、NTP 向后校正、检查点关闭、12 小时间隔和已有业务活动不重复写检查点。
- 不同复位原因的基础库系统日志，以及可信时间不足时不生成停电估算。
- 配置旧修订提交、字段和跨字段非法、临时文件/备份恢复及模拟写入失败。
- 校准稳态检测配置边界、连续窗口判定、跨采样状态清理、阶段脉冲守恒、未识别稳态自动无效、样本编辑删除和 10 条上限、跨水路采样、三项校准结果、残差和异常结果处理，以及区域流量学习的原始脉冲稳定完成、单脉冲量化误差、零脉冲窗口、手工停止和开阀后 10 分钟上限。
- 记录提交前后断电、记录尾部断写、CRC 错误、分段轮换、容量覆盖、未知真实时间和存储失败。
- 固件默认认证与基础库已保存认证的优先级，以及 Web 未认证、非 POST、跨源、重复提交、忙状态和参数边界。
