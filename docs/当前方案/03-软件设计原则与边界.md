# 03 软件设计原则与边界

## 文档定位

本文件只约束下一阶段的软件设计方向，不提前规定类名、对象名、状态枚举、文件格式、接口路径或代码目录。

正式设计代码结构前，需要结合当前产品规则、定稿网表、`Esp32Base` 的实际接口和现有代码重新评审，选择最简单且足够可靠的实现。

## 设计原则

### 以产品概念为中心

- 软件对象必须能够清楚对应控制器、区域、浇水计划、启动时间、区域运行时长、流量报警、浇水记录和事件。
- 代码名称优先参考成熟灌溉控制器和通行工程术语。
- 用户口头名称只用于表达需求，不自动成为最终产品或代码术语。
- 同一个概念在文档、页面、API 和代码中保持一致。

### 保持简单

- 只为当前产品真实存在的业务建模。
- 一个规则只保留一个权威实现，不在 Web、调度、本地操作和校准中分别复制。
- 不为了将来可能出现的功能预先增加抽象层、插件机制、通用映射或复杂扩展点。
- 当直接、清晰的实现已经足够时，不增加额外服务、状态或数据转换层。

### 安全优先

- 上电第一时间关闭全部阀和泵控制信号。
- 没有活动区域时保持阀驱动关断。
- 任何出水过程都必须支持立即停止和无流量保护。
- 校准和流量学习必须复用正常的硬件安全控制，不能直接写一套临时 GPIO 流程。
- 所有等待都使用非阻塞计时，不能影响 Web、流量计数、停止操作和看门狗。
- 重启、异常和 OTA 后不自动恢复未完成浇水。

### 分清数据职责

- 业务配置保存用户设置。
- 运行中的浇水使用开始时确定的配置，配置修改只影响后续浇水。
- 浇水记录由本项目保存，结构应满足业务查询，不依赖事件日志拼装。
- 事件使用 `Esp32Base` App Events，本项目只提供有业务意义的事件内容。
- 系统日志、业务事件和浇水记录不能互相代替。

## `Esp32Base` 使用边界

固件基于 `../Esp32Base` 搭建，优先使用其已有能力：

- 启动生命周期。
- WiFi、Web 和认证。
- OTA。
- 文件系统。
- 系统日志和 App Events。
- 健康状态和看门狗。
- NTP、RTC 和统一时间。

本项目实现浇水领域功能：

- 固定 PCB 引脚和安全硬件控制。
- 区域和浇水计划。
- 手动浇水和自动调度。
- 流量监测、报警、校准和学习。
- 浇水记录。
- 浇水业务页面。

项目不能修改 `../Esp32Base`。发现基础库问题时，记录问题和建议，交由基础库单独处理。

## 配置保留原则

- 首次启动时生成默认配置。
- 修改代码中的默认值不能覆盖设备已经保存的有效配置。
- 重新编译烧录或 OTA 升级后，只要当前配置结构仍然有效，就继续使用设备已有配置。
- 读取配置后必须重新校验所有字段和字段之间的关系，不能因为文件能解析就认为配置有效。
- 配置无法安全使用时，保持所有输出关闭，并通过页面明确提示用户处理。
- 当前开发阶段只维护当前确认的配置结构，不为不同结构同时保留多套读取和运行分支。

具体配置格式、版本字段、原子写入方式和结构调整后的处理方式，在软件设计阶段结合 `Esp32Base` 文件系统能力确定。

## 时间与调度原则

- RTC 和 NTP 统一通过 `Esp32Base` 时间能力使用。
- 自动调度只使用可信时间。
- 手动浇水不依赖可信日历时间。
- 必须处理重启、时间向前跳变、时间向后跳变和当天重复触发。
- 调度只决定是否启动浇水，不能直接操作硬件。

当天执行标记放在内存、NVS 还是其它位置，等软件设计时根据可靠性和写入寿命统一决定。

## Web 与本地操作

- Web 页面调用业务能力，不直接操作 GPIO、配置文件或浇水记录文件。
- 页面提交的数据必须由后端重新校验。
- 运行状态由统一的只读状态数据提供，不能由页面自行推断。
- OTA、重启和文件系统维护在浇水或校准期间必须阻止，用户先停止当前操作。
- 将来增加 LCD2004 和 4 个按钮时，与 Web 调用相同业务能力，不复制规则或硬件控制流程。

## 下一阶段需要完成的软件设计

进入代码修改前，需要单独完成以下设计并评审：

1. 核心业务对象及专业命名。
2. 一次浇水的执行模型和非阻塞状态设计。
3. 手动、计划、校准和流量学习如何共用安全控制。
4. 流量监测和报警的职责边界。
5. 配置结构、校验、保存和刷新方式。
6. 浇水记录的数据结构、容量、掉电恢复和查询方式。
7. 事件名称和字段约定。
8. 自动调度的防重复实现。
9. Web 页面、业务接口和状态数据边界。
10. 单元测试、硬件测试和异常场景验收清单。

这些设计必须以当前产品规模为准，先证明需要，再增加结构；不能直接沿用现有代码中的类名或拆分方式。

## 软件验收底线

- 引脚与定稿网表一致。
- 默认只启用区域 1、区域 2。
- 阀默认全功率吸合 3 秒，随后以 75% 占空比维持。
- 上电和空闲时所有输出处于安全状态。
- 普通浇水、校准和流量学习都有全程无流量保护。
- 手动和自动操作遵守同一套业务与安全规则。
- 自动计划不重复、不补执行。
- 配置更新不覆盖已有有效用户配置。
- 浇水记录和事件职责分离。
- Web 不直接控制硬件和存储。
- 当前代码中不加入 LCD2004 和按钮交互实现。
