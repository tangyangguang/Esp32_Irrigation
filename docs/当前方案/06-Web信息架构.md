# 06 Web 信息架构

## 原则

Web 页面按对象和任务组织，不按技术难度组织。

页面必须回答明确问题：

- 当前是否正常。
- 现在能不能浇水。
- 自动计划怎么安排。
- 有哪些水源和水路。
- 实际浇水、异常、配置和校准历史是什么。
- 为什么没启动、为什么停止、哪里异常。
- 全局参数在哪里改。

## 顶层页面

当前建议顶层页面：

1. 总览
2. 运行
3. 计划
4. 水源
5. 历史
6. 系统参数

暂不建议为了以下内容新增顶层页面：

- 诊断。
- 维护。
- 统计。
- 天气。
- 通知。
- OTA。
- LCD。

这些内容如果将来需要，应先证明它们不能被现有页面承载。反过来，如果当前 6 个页面仍显冗余，也可以继续合并。

## 页面职责

### 总览

回答：现在系统是否正常，是否有正在运行或需要处理的问题。

显示：

- 主控时间和状态。
- 从站和水源摘要。
- 当前运行摘要。
- 当前异常和提示摘要。
- 今日浇水摘要。

不做：

- 完整手动启动表单。
- 配置表单。
- 完整历史列表。

### 运行

回答：现在能不能浇水，如何手动启动或停止。

显示：

- 当前任务。
- 水源可用性。
- 手动启动入口。
- 不能启动或启动失败原因。
- 运行保护状态。

不做：

- 编辑计划。
- 修改硬件映射。
- 修改系统参数。

### 计划

回答：什么时候自动浇水，浇哪些水路，浇多久。

显示：

- 计划列表。
- 启用状态。
- 下一次触发。
- 步骤摘要。
- 计划编辑视图。

不做：

- 手动启动普通浇水。
- 展示完整历史。

### 水源

回答：有哪些水源和水路，它们是否可用。

显示：

- 水源列表。
- 从站地址、在线状态和配置状态。
- 主阀、水泵、流量计、低液位摘要。
- 水路列表。
- 水源配置和水路配置入口。
- 校准入口。

不做：

- 普通手动浇水。
- 在列表中直接铺开复杂配置表单。
- 展示完整历史。

### 历史

回答：哪些水路真实运行过，为什么没启动或停止，用户和系统发生过什么变化。

显示：

- 历史列表。
- 类型筛选：浇水、异常、配置、校准、操作。
- 筛选和分页。
- 历史详情。
- 采用稳定流速为参考流量的操作。

不做：

- 发起浇水。
- 复杂告警确认或清除流程。

### 系统参数

回答：影响全局运行、通信和保护的参数是什么。

显示：

- 时间和 RTC。
- 全局运行参数。
- 通信参数。
- 流量保护参数。
- 输入映射：流量计输入、低液位输入。

不做：

- 水源级配置。
- 水路级配置。
- 通用诊断后台。

## 我的建议

- 页面数量不要先定死。先看用户要完成哪些任务，再决定合并或拆分。
- 水源页不做普通手动浇水入口。否则运行页职责会被削弱。
- 系统参数页只放全局参数。不要再叫“设置”，否则水源配置和水路配置容易被混进去。
- 先用统一历史页面，不拆记录和事件。历史页通过类型筛选解决复杂度。
- 对象管理页不要把列表和完整编辑表单并排铺在同一个主页面。列表页主区只放列表、筛选、状态摘要和操作入口；新增、编辑、详情、校准进入弹窗、抽屉或独立详情视图。

## 后续细化

当前按桌面 Web 管理界面设计信息架构，移动端适配后置。后续 Web 界面方案继续验证 6 个顶层页面是否足够清楚。
