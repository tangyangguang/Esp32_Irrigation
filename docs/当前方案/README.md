# 当前方案

本目录定义当前唯一有效的 12V 智能浇水控制器方案，是后续代码实现和验收的依据。

当前硬件事实只以以下 2026-07-11 文件为准：

- `pcb_irrigation/Netlist_Schematic1_2026-07-11.tel`
- `pcb_irrigation/BOM_Board1_Schematic1_2026-07-11.xlsx`

旧工程、旧 PCB、旧原型和 `上一版/V6/` 只可用于理解背景，不能继承其中的引脚、对象、页面、接口或兼容逻辑。

## 文档结构

只保留三份正文，按以下顺序阅读：

1. `01-产品与硬件边界.md`：产品做什么、硬件固定能力和当前不做什么。
2. `02-业务规则与安全.md`：计划、运行、流量保护、校准、浇水记录和业务事件。
3. `03-软件架构与实现约束.md`：基础库边界、数据模型、状态机、存储、Web 和验收合同。

本目录不再保留 HTML 功能清单或交互原型。Web 页面已经进入实现阶段，页面行为必须回到这三份方案文档核对。

## 已固定的核心结论

- 产品是单板 12V DC 智能浇水控制器。
- 6 路输出全部是固定 Zone 电磁阀输出，不做主阀和输出映射。
- 同一时刻只打开一个 Zone。
- 只有一套供水系统，不做多水源并行。
- 1 路流量计固定接入且必配；运行和校准全过程都有缺水保护。
- 1 路可选自吸泵干接点输出，只表达控制命令，不反馈泵的真实运行状态。
- 最多 8 个每日计划，每个计划最多 4 个每日启动时间。
- 手动浇水、自动计划和立即执行计划共用一套顺序运行状态机。
- 自动计划忙时跳过，不排队、不抢占、不补跑。
- 浇水记录由本项目保存；异常、保护、跳过、配置和校准等事件使用 `Esp32Base` App Events。
- 启动、WiFi、NTP、OTA、文件系统等纯系统事实由 `Esp32Base` 系统日志和状态页负责，业务事件不机械复制。
- 固件必须基于 `../Esp32Base`，本项目不重复实现通用基础能力，也不修改基础库。

## 暂缓但保留的硬件能力

PCB 已引出 4 个按钮输入，并预留 I2C 扩展能力。未来计划增加 LCD2004 和本地按钮操作，但当前阶段不设计页面、不定义按键行为、不读取按钮，也不实现 LCD 驱动。

未来本地操作只能抽取已经由 Web 和业务服务验证成熟的常用能力，不得形成另一套业务规则、状态机或配置模型。

## 进入代码前的约束

- 所有引脚必须逐项对照定稿网表，不允许从旧代码或旧文档复制。
- 任何出水流程都必须经过统一安全状态机，校准不能绕过缺水保护。
- 配置损坏、时间无效、重启、OTA 和异常恢复的行为必须符合本文档。
- 业务配置、运行快照、浇水记录和事件必须分清职责，不能把 App Events 当作浇水记录数据库。
- 当前没有旧配置、旧 API、旧页面和旧流程兼容要求。
