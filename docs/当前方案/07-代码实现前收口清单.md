# 07 代码实现前收口清单

## 目的

这份清单用于约束正式编码顺序，避免先写页面、先写调度或先写临时硬件动作导致返工。

当前实现只面向这块 12V 单板智能浇水控制器。旧 RS485、主从站、主阀、多水源和输入输出映射方案不进入代码。

## 开工前固定规则

### 产品常量

| 常量 | 值 | 说明 |
| --- | ---: | --- |
| 最大 Zone 数 | 6 | 固定 6 路阀输出 |
| 默认启用 Zone 数 | 1 | 初始化只启用 Zone 1 |
| 最大计划数 | 8 | 产品复杂度限制，不是硬件限制 |
| 每个计划最大启动时间 | 4 | 同一组 Zone 时长可一天执行多次 |
| 同时运行 Zone 数 | 1 | 不支持多 Zone 并行 |

### 必须固定的业务规则

- 计划只支持每天策略。
- 启用计划才参与自动调度，禁用计划只保留配置。
- 禁用计划不触发、不补跑、不显示为下一次计划，也不能直接立即执行。
- 手动浇水、自动计划、立即执行计划都生成 `WateringRun`。
- 手动浇水不是选择单个 Zone，而是给已启用 Zone 设置时长后顺序执行。
- 自动计划忙时跳过并记录，不排队、不抢占、不补跑。
- 流量计必配，不提供启用/禁用。
- 无流量是停机保护。
- 低/过流量第一阶段只提示，不停机。
- 低液位只在自吸泵启用且低液位启用时参与保护。
- 自吸泵输出只是干接点控制信号，不直接驱动泵或继电器线圈。
- 任何上电、重启、OTA、配置损坏场景都不能自动恢复浇水。

## 实现顺序

### 阶段 1：类型、默认配置、配置校验

目标：让所有后续模块共享同一套结构，不各自发明字段。

交付：

- `IrrigationTypes.h`
- `IrrigationConfig.h`
- 默认配置生成。
- 基础配置校验。
- 计划、Zone、启动时间的基础 helper。

验收：

- 固件能编译。
- 默认配置满足 Zone 1 启用、Zone 2..6 停用、无计划、自吸泵禁用、低液位禁用。
- 常量值和文档一致。

### 阶段 2：板级硬件抽象

目标：先把硬件安全态和固定引脚语义封装起来。

交付：

- `BoardHardware`
- `safeOff()`
- 6 路阀输出接口。
- 流量计脉冲计数接口。
- 低液位输入接口。
- 自吸泵干接点输出接口。

验收：

- 上电所有输出关闭。
- 驱动关断脚进入安全态。
- 单独打开/关闭每个 Zone 不影响其它 Zone。
- 自吸泵输出默认关闭。

### 阶段 3：ConfigStore

目标：业务配置统一读写，后续服务不直接碰文件。

交付：

- `/irrigation/config.json`
- `version / updatedAt / data`
- `supply / flow / valve / zones / plans` 分区。
- 临时文件写入、校验、替换。
- 加载失败时使用默认配置并记录事件。

验收：

- 首次启动能生成默认配置。
- 保存后重启能读回配置。
- 配置损坏时不启动任务，并保留安全态。

### 阶段 4：ZoneService 和 PlanService

目标：把 Zone 和计划的业务规则集中，Web 和调度都只调用服务。

交付：

- Zone 启用/停用、改名、默认时长、标准流量。
- 计划新增、保存、删除、启用/禁用。
- 启用计划过滤。
- 有效步骤生成。

验收：

- 未启用 Zone 不进入手动和计划步骤。
- 禁用计划不进入调度和立即执行选择。
- 删除计划释放计划槽位。

### 阶段 5：RunController 手动闭环

目标：先跑通最核心的顺序浇水，不接自动计划。

交付：

- `WateringRun` 状态机。
- 手动任务生成。
- 顺序执行多个 Zone。
- 用户停止。
- 安全关闭路径。

验收：

- 手动顺序浇水能执行。
- 当前 Zone 到时后关闭，再进入下一 Zone。
- 用户停止后所有输出关闭。
- 状态机非阻塞。

### 阶段 6：流量、自吸泵、低液位保护

目标：在手动闭环稳定后增加保护，不改变任务入口。

交付：

- 无流量停机。
- 自吸泵启动/停止延时。
- 低液位启动前阻止和运行中停机。
- 待机漏水提示。
- 低/过流量提示。

验收：

- 无流量触发异常停机。
- 自吸泵启用时先开 Zone 阀，再输出泵信号。
- 自吸泵禁用时流程不包含泵动作。
- 自吸泵禁用时低液位不影响浇水。

### 阶段 7：Scheduler

目标：自动计划只负责触发，不直接碰硬件。

交付：

- 按 `Esp32BaseTime` 检查当前时间。
- 启用计划和启用 StartTime 才参与扫描。
- 每个启动时间每天最多触发一次。
- 忙时跳过并记录。

验收：

- 启用计划到点执行。
- 禁用计划不触发。
- 设备忙时自动计划跳过，不排队。
- 重启后不补跑错过计划。

### 阶段 8：历史、事件和 Web

目标：业务闭环稳定后再做用户界面。

交付：

- 浇水历史。
- 停止、异常、跳过、校准、配置事件。
- 总览、运行、计划、Zone、校准、历史、设置页面。

验收：

- Web 不直接操作 GPIO。
- Web 不直接写配置文件。
- 页面状态来自业务快照。
- 用户能看懂为什么不能启动、为什么停止。

## 本轮开始编码范围

本轮只做阶段 1：

- 新增核心类型。
- 新增默认配置。
- 新增基础校验。
- 编译验证。

本轮不做：

- GPIO 动作重构。
- 配置文件读写。
- 状态机。
- Scheduler。
- Web 页面改造。
