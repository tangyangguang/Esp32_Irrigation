# 11 RS485 协议与工程骨架

## 目标

本文继续收口正式实现前必须确定的两件事：

- 主控和分站之间的 RS485 灌溉协议。
- 主控 ESP32 工程和分站 STC8H8K64U 工程的最小骨架。

原则：

- 协议只服务当前灌溉产品，不做 Modbus 寄存器平台。
- 主控轮询，分站应答，分站不主动抢占总线。
- 帧足够小，便于 STC8H 侧实现和排查。
- 工程结构清楚即可，不做复杂框架。

## 通信模型

通信角色：

- 主控是唯一主站。
- 分站是从站。
- 一个分站对应一个水源。
- 分站地址来自 4 位拨码，地址范围建议为 `1..15`。
- 地址 `0` 保留，不作为普通水源地址。

通信方式：

```text
主控发送请求
  ↓
目标分站处理
  ↓
目标分站返回响应
  ↓
主控更新状态或记录结果
```

分站不主动上报。所有状态变化由主控下一次轮询得到。

这样做的好处：

- 半双工总线简单。
- 不需要总线仲裁。
- 分站代码简单。
- 通信异常时容易解释。

## 帧格式

建议使用小二进制帧。

```text
offset  size  field
0       1     magic0 = 0xA5
1       1     magic1 = 0x5A
2       1     version = 1
3       1     addr
4       1     seq
5       1     cmd
6       1     flags
7       1     len
8       N     payload
8+N     2     crc16 little-endian
```

约定：

- 多字节整数统一 little-endian。
- `len` 最大建议先限制为 `64` 字节。
- `crc16` 使用 CRC16 Modbus 算法。
- CRC 覆盖 `version` 到 `payload`，不覆盖两个 magic 字节。
- 响应必须复用请求的 `seq` 和 `cmd`。
- 主控按 `seq` 判断响应是否属于当前请求。

`flags` 建议：

```text
bit0 response
bit1 error
bit2 busy
bit3 reserved
bit4..7 reserved
```

错误响应：

```text
flags.response = 1
flags.error = 1
payload[0] = errorCode
payload[1..] 可选错误上下文
```

不做：

- 字节转义。
- 分片传输。
- 大 payload。
- 广播启动任务。
- 从站主动上报。

如果解析中遇到坏帧，接收端丢弃并继续寻找下一组 magic。

## 硬件引脚前置条件

主控侧 RS485 可直接用 `Esp32BaseRs485Port` 配置：

```text
RX  = GPIO16
TX  = GPIO17
DIR = GPIO13
```

分站侧 RS485 使用：

```text
RX  = P3.6 / RxD_2
TX  = P3.7 / TxD_2
DIR = P4.1
```

分站工程使用 `STC8H_UART1_PIN_GROUP=1` 启用 UART1 第二组引脚。不要在本项目里直接手写 `P_SW1` 当补丁。

## 时序与重试

主控策略：

- 同一时间只发一个请求。
- 请求后等待响应。
- 超时后重试有限次数。
- 多次失败后把该分站标记为离线。
- 通信恢复后重新同步状态和最近结果。

建议初始参数：

```text
请求响应超时：200 ms
单请求重试：2 次
轮询周期：1..3 秒
离线判定：连续失败 3 轮
```

这些值先作为实现默认，后续可通过系统参数调整。

分站策略：

- 只响应发给自己地址的请求。
- 地址不匹配直接忽略。
- CRC 错误直接忽略。
- 正在执行当前任务时仍要响应 `GET_STATUS` 和 `STOP_TASK`。
- 正在忙于状态切换时，对不能立即执行的命令返回 `busy` 或错误码。

## 命令集

命令保持小集合。

```text
0x01 PING
0x02 GET_STATUS
0x03 START_TASK
0x04 STOP_TASK
0x05 GET_LAST_RESULT
0x06 GET_INPUTS
0x07 GET_INFO
```

不做：

- 通用读写寄存器。
- 任意 GPIO/PWM/ADC 读写。
- 分站计划配置。
- 分站历史查询。
- 分站文件传输。

### PING

用途：

- 判断指定地址是否有分站响应。
- 轻量连通性检测。

请求 payload：

```text
empty
```

响应 payload：

```text
uptimeSec u32
```

### GET_INFO

用途：

- 读取分站静态能力。
- 主控发现和页面诊断使用。

响应 payload：

```text
deviceType        u8   固定为 irrigation slave
protocolVersion   u8
valveCount        u8   当前为 8
inputCount        u8   当前为 4
capabilityFlags   u16
firmwareMajor     u8
firmwareMinor     u8
firmwarePatch     u8
```

`capabilityFlags` 建议：

```text
bit0 hasPumpOutput
bit1 hasFlowInput
bit2 hasLowLevelInput
bit3 hasVoltageAdc
bit4 hasPwmValveDrive
```

### GET_STATUS

用途：

- 主控周期轮询分站当前状态。
- 页面显示当前运行、异常、输入和电压摘要。

响应 payload：

```text
stationState      u8
taskState         u8
faultCode         u8
activeTaskId      u16
activeZoneId      u8
activeValveNo     u8
elapsedSec        u32
remainingSec      u32
flowPulseCount    u32
flowPulseRate     u16
inputBits         u8
voltageMv         u16
lastResultSeq     u16
```

说明：

- `flowPulseRate` 可以先定义为最近窗口脉冲数或脉冲/分钟，具体单位在实现前最终确认。
- `lastResultSeq` 用于主控判断最近任务结果是否已经读取过。

### START_TASK

用途：

- 主控下发当前水路任务。

请求 payload 建议：

```text
taskId                  u16
sourceId                u8
zoneId                  u8
branchValveNo           u8
durationSec             u32

mainValveEnabled        u8
mainValveNo             u8
pumpEnabled             u8

mainValveOpenLeadSec    u8
mainValveCloseDelaySec  u8
pumpStartDelaySec       u8
pumpStopLeadSec         u8

flowMeterEnabled        u8
flowInputNo             u8
lowLevelEnabled         u8
lowLevelInputNo         u8
lowLevelActiveMode      u8
lowLevelDebounceMs      u16

noFlowGraceSec          u16
noFlowConfirmSec        u16
lowFlowAction           u8
overFlowAction          u8

mainValveFullPowerMs    u16
mainValveHoldDutyPermil u16
branchFullPowerMs       u16
branchHoldDutyPermil    u16
```

约定：

- 阀号范围为 `1..8`。
- 输入号范围为 `1..4`。
- duty 使用 `0..1000` 表示 `0%..100%`。
- `durationSec` 必须大于 0，且不超过系统最大单次运行时长。
- 分站只接受一个当前任务；运行中再次 `START_TASK` 返回 busy。

响应 payload：

```text
acceptedTaskId u16
```

### STOP_TASK

用途：

- 用户停止。
- 主控主动终止当前任务。

请求 payload：

```text
taskId     u16   0 表示停止当前任务
stopReason u8
```

响应 payload：

```text
stoppedTaskId u16
taskState     u8
```

分站收到后按停止顺序关闭泵、分支阀和主阀。

### GET_LAST_RESULT

用途：

- 主控读取最近一次任务结束事实。
- 通信恢复后补同步从站最近结果。

响应 payload：

```text
resultSeq          u16
taskId             u16
zoneId             u8
resultCode         u8
faultCode          u8
plannedDurationSec u32
actualDurationSec  u32
flowPulseCount     u32
stablePulseRate    u16
inputBitsAtEnd     u8
voltageMvAtEnd     u16
```

说明：

- 分站只保存最近结果。
- 主控负责转成完整历史。
- 如果没有结果，返回 `resultCode = none`。

### GET_INPUTS

用途：

- 读取输入和电压的当前值。
- 页面诊断和启动前检查使用。

响应 payload：

```text
inputBits       u8
flowPulseCount  u32
flowPulseRate   u16
voltageMv       u16
```

## 主控工程骨架

主控工程建议保持简单。

```text
master-esp32/
  platformio.ini
  include/
    IrrigationConfig.h
    IrrigationTypes.h
    BoardPins.h
  src/
    main.cpp
    app/
      RuntimeController.cpp
      Scheduler.cpp
      CalibrationController.cpp
    comm/
      Rs485Protocol.cpp
      Rs485Master.cpp
    storage/
      ConfigStore.cpp
      HistoryStore.cpp
    web/
      WebPages.cpp
      WebApi.cpp
    board/
      Board.cpp
```

说明：

- `main.cpp` 只负责初始化和主循环调度。
- `BoardPins.h` 固定主控板引脚，例如 RS485、I2C。
- `Rs485Protocol` 只做帧编码/解码和 CRC。
- `Rs485Master` 负责请求、响应、超时、重试和轮询节奏。
- `RuntimeController` 负责能否启动、启动后状态和停止原因。
- `WebPages` 不直接操作 RS485，通过运行层调用。

不要增加：

- 复杂依赖注入。
- 多级事件总线。
- 跨 task 配置写入。
- 通用设备抽象表。

## 分站工程骨架

分站工程建议也保持小。

```text
slave-stc8h/
  platformio.ini
  board/
    board_config.h
    board_pins.h
    board_init.c
  src/
    main.c
    app/
      task_runner.c
      protection.c
      status_reporter.c
    comm/
      irrigation_frame.c
      command_handler.c
    drivers/
      output_driver.c
      input_monitor.c
    base/
      stc8h_gpio_wrap.c
      stc8h_uart_wrap.c
      drv_rs485_uart_wrap.c
      stc8h_pwm_wrap.c
      stc8h_timer_wrap.c
      stc8h_adc_wrap.c
      stc8h_wdt_wrap.c
      util_crc_wrap.c
      util_soft_timer_wrap.c
```

说明：

- `base/` 只放 PlatformIO wrapper include，不复制基础库实现。
- `board_init.c` 只初始化当前使用的板级资源，不做“一键初始化全部外设”。
- `output_driver` 负责所有输出安全态和 PWM duty。
- `task_runner` 只处理当前任务，不理解计划。
- `command_handler` 把协议命令映射到业务动作。

不要增加：

- EEPROM 参数系统。
- 从站计划系统。
- 通用 IO 命令解释器。
- 大型日志系统。

## 分站主循环

建议形态：

```text
main()
  board_outputs_safe_off()
  board_init_used_peripherals()
  protocol_init()
  task_runner_init()
  enable_timer_tick()
  enable_wdt()

  while (1)
    wdt_feed()
    protocol_poll_rx()
    command_handler_handle()
    input_monitor_handle()
    task_runner_handle()
    status_reporter_handle()
```

要求：

- 运行中不使用长阻塞 delay。
- 定时动作基于 1ms tick 或 soft timer。
- 任何异常路径优先关闭输出。
- 协议收包不能阻塞当前任务计时。

## 首轮开工边界

第一批代码只做这些：

主控：

- FULL profile 工程能编译。
- RTC 初始化路径存在。
- RS485 port 初始化路径存在。
- `Rs485Protocol` 能编码/解码帧。
- `Rs485Master` 能 PING / GET_INFO / GET_STATUS。
- 页面可显示“未发现从站”或真实发现结果。

分站：

- 工程能编译。
- 上电安全关闭所有输出。
- RS485 能响应 PING / GET_INFO / GET_STATUS。
- 8 路 PWM 初始化但默认关闭。
- WDT 和 1ms tick 正常。

不在首轮做：

- 自动计划。
- 校准。
- 历史文件。
- 完整 Web 页面。
- 低/过流量判断。
- 分站 OTA。

这不是产品拆版本，而是降低同时开工的风险。首轮完成后继续把当前任务闭环补齐。

## 当前风险

仍需核实：

- 主控 `485_DIR`、`485_RX`、`485_TX` 到 ESP32 GPIO 的准确映射。
- 从站 `PWM1..PWM8` 到 STC8H PWM pin select 的准确映射。
- `IN1..IN4` 是否都适合流量脉冲计数，是否需要外部中断。
- 电压 ADC 换算比例。

如果流量计脉冲必须依赖中断计数，而基础库现有能力不足，应先回 `Stc8hBase` 补可复用脉冲计数能力。
