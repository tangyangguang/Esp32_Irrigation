# 00 设计原则与边界

## 目标

当前方案要把灌溉系统的边界、逻辑和模型重新说清楚。

本阶段成功标准：

- 用户能看懂系统由哪些对象组成。
- 每个对象负责什么、不负责什么。
- 运行、停止、异常、配置和记录之间的关系清楚。
- 页面职责能从方案自然推出，而不是靠界面临时补逻辑。

## 当前阶段边界

现在只做方案设计和 Web 界面方案验证。

不做：

- 固件代码。
- API、存储、通信协议。
- 工程目录、类名、任务调度实现。
- LCD 菜单和本地按键完整流程。

这些内容必须等方案和界面方案确认后再做。

## 设计原则

1. 先看硬件事实，再定产品能力。
2. 先定对象和状态，再定页面。
3. 安全动作本地闭环优先。
4. 能用简单规则解决的，不做通用平台。
5. 默认值要合理，可配置项要分组清楚。
6. 不按场景标签拆功能，例如家庭、菜地、温室、农场。
7. 不把历史文档里的结论直接继承为当前结论。

## 最小闭环

当前方案的最小闭环是：

1. 主控发现并管理从站。
2. 一个从站对应一个水源。
3. 水源下有若干分支水路。
4. 用户可以手动运行水路。
5. 用户可以配置自动计划。
6. 系统能按规则启动、停止、跳过或异常停止任务。
7. 真实运行产生浇水记录。
8. 真实运行、异常、跳过和配置变化都能被追溯。
9. 从站执行当前有时长的水路任务，能本地到点关闭输出。
10. 从站不保存未来计划，但当前任务运行中遇到本地异常时能停止输出并上报。

如果某个能力不服务这个闭环，先不进入当前方案。

## 我的建议

- 当前不要追求“大而全”的智能灌溉平台。这个项目更适合做成一个可理解、可维护、可现场排查的本地灌溉控制器。
- 不建议引入脚本、复杂条件联动、天气自动调整、多用户权限、复杂统计和通用 IO。它们会显著增加状态和解释成本。
- 不建议把所有硬件资源都做进产品。PCB 有预留不等于当前产品必须启用。
- 建议先把“正常浇水”和“异常时为什么不能浇水”讲清楚，再考虑锦上添花的展示。

## 后续细化

- 是否继续以桌面 Web 管理界面为第一目标，移动端后置。
- LCD2004 和主控按键是否只在 Web 方案确认后抽取最小现场能力。
