# 06 软件架构与信息结构

## 设计结论

当前软件只服务这一块 12V 单板智能浇水控制器。

最终软件采用“基础库平台能力 + 本项目浇水业务”的结构。

当前优化后的结论：

- `Esp32Base` 负责通用能力：启动、WiFi、Web、认证、OTA、LittleFS、日志、事件、配置、健康、看门狗、NTP、RTC 和统一时间。
- 本项目只实现业务能力：Zone、计划、浇水任务、阀控制、流量安全、自吸泵信号、低液位保护、历史和业务 Web。
- 不新增 RTC 驱动，不新增计划队列，不做复杂流量闭环，不做通用 IO 平台。
- 所有手动浇水、自动计划和立即执行计划都统一进入 `WateringRun` 状态机。

软件模型采用成熟灌溉控制器常见结构：

```text
Controller
  -> Zone
  -> WateringPlan
  -> StartTime
  -> ZoneRuntime
  -> WateringRun
  -> WateringStep
  -> FlowSafety
  -> PumpStartSignal
  -> DryRunProtection
  -> EventLog
```

术语固定：

- `Zone`：一路阀对应的浇水区域。中文界面也优先显示 `Zone 1`、`Zone 2`。
- `WateringPlan`：浇水计划。
- `StartTime`：计划每天启动的时间。一个计划最多 4 个启动时间。
- `ZoneRuntime`：一个计划中某个 Zone 的运行时长。
- `WateringRun`：一次实际执行的浇水任务。
- `WateringStep`：一次任务中的单个 Zone 执行步骤。

不再使用“水路”作为软件核心对象名。文档中如出现“水路”，只作为中文解释，不作为代码模型名。

## 产品级规则

### Zone

- 硬件最大支持 6 个 Zone。
- 系统初始化默认只启用 `Zone 1`。
- `Zone 2` 到 `Zone 6` 默认停用，由用户启用。
- 每个 Zone 固定对应一个阀输出，不允许重新映射。
- 未启用 Zone：
  - 不出现在手动浇水时长表中。
  - 不出现在计划编辑的 Zone 时长表中。
  - 不参与自动计划执行。
  - 已有计划中保留的时长不删除，重新启用后可继续使用。

### WateringPlan

计划固定为每天策略。

一个计划包含：

- 名称。
- 启用状态。
- 最多 4 个 `StartTime`。
- 6 个 Zone 的时长配置。

显示和编辑时只显示已启用 Zone。存储时保留 6 个 Zone 的时长，避免用户停用 Zone 后丢失原配置。

多个启动时间表示同一组 Zone 时长在一天内执行多次：

```text
日常浇水
  StartTime: 08:00, 16:00
  Zone 1: 10 min
  Zone 2: 8 min
```

如果上午和下午每个 Zone 的时长不同，应创建两个计划：

```text
上午浇水
  StartTime: 08:00
  Zone 1: 10 min

下午补水
  StartTime: 16:00
  Zone 1: 5 min
```

### WateringRun

手动浇水、计划到点、立即执行计划，最终都生成 `WateringRun`，由同一个状态机执行。

任务来源：

- `manual`：用户在运行页设置已启用 Zone 时长后启动。
- `plan`：每日计划到点自动触发。
- `run_plan_now`：用户选择一个计划立即执行。
- `calibration`：校准流程创建的临时任务。

同一时刻只允许一个 Zone 打开。

## 软件分层

### 1. 基础层：Esp32Base

基础层由 `../Esp32Base` 提供。

使用能力：

- 启动生命周期。
- WiFi。
- Web。
- Web 认证。
- OTA / Web OTA。
- LittleFS。
- 系统日志。
- App Events。
- App Config。
- 健康状态。
- 看门狗。
- NTP。
- mDNS / DNS。
- RTC / NTP 统一时间。

本项目不重复实现这些通用能力。

如果基础库缺少可复用的通用能力，应优先修改 `../Esp32Base`，不要在本项目里写局部旁路补丁。

### 时间与 RTC 边界

`Esp32Base` 已经提供 RTC 和统一时间能力，本项目不再新增自己的 RTC 驱动、RTC 服务或 RTC 状态机。

本项目只做两件事：

- 在构建配置中启用 `ESP32BASE_ENABLE_RTC=1`，驱动选择 `ESP32BASE_RTC_DRIVER_DS3231`，I2C 使用 `GPIO21/GPIO22`。
- 业务调度、历史时间和页面显示统一使用 `Esp32BaseTime::snapshot()`、`Esp32BaseTime::formatTime()` 和 `Esp32BaseTime::resolveCurrentBootEvent()`。

业务代码不要直接依赖 NTP 状态，也不要直接读写 DS3231 寄存器。

当前也不使用 `RTC INT/SQW` 做闹钟唤醒或计划触发。计划调度由主循环按统一时间检查完成，避免引入第二套时间触发机制。

### 2. 应用编排层：IrrigationApp

职责：

- 设置上电安全态。
- 注册业务 Web 路由。
- 加载业务配置。
- 初始化业务服务。
- 将业务循环挂到 `Esp32Base::handle()` 之后。

启动顺序：

```text
Serial
  -> BoardHardware::safeOff()
  -> WebRoutes::register()
  -> Esp32Base::begin()
  -> ConfigStore::load()
  -> DomainServices::begin()
  -> idle
```

上电阶段任何配置加载失败，都不能自动启动浇水。

`IrrigationApp` 只做编排，不放业务判断，不直接执行浇水流程。

### 3. 板级硬件层：BoardHardware

职责：

- 固定当前 PCB 引脚。
- 统一上电安全态。
- 提供最小硬件访问接口。

内部能力：

- `ValveDriver`
- `FlowMeter`
- `LowLevelInput`
- `PumpDryOutput`
- `LocalButtons`

板级层只处理引脚和电平，不理解计划、Zone 名称、历史和 Web。

### 4. 业务服务层

核心服务：

| 服务 | 职责 |
| --- | --- |
| `ConfigStore` | 读取、保存和迁移业务配置 |
| `ZoneService` | 管理 6 个 Zone 的启用、名称、默认时长、标准流量 |
| `PlanService` | 管理计划、启动时间和 Zone 时长 |
| `RunController` | 执行当前 `WateringRun` 状态机 |
| `Scheduler` | 使用 `Esp32BaseTime` 触发每日计划 |
| `FlowSafetyService` | 处理流量计统计、无流量停机、漏水提示、低/过流量提示 |
| `HistoryService` | 保存浇水、异常、配置、校准和用户操作 |
| `EventService` | 适配 `Esp32BaseAppEventLog` |

业务服务层可以调用 `BoardHardware`，但 `BoardHardware` 不反向调用业务服务。

### 5. Web 应用层：IrrigationWeb

职责：

- 注册业务页面和 POST 路由。
- 渲染总览、运行、计划、Zone、校准、历史、设置。
- 处理用户提交。
- 调用业务服务。
- 显示状态、错误和历史。

Web 层不直接操作 GPIO，不直接写配置文件，不直接写历史文件。

## 依赖方向

代码依赖只能按这个方向走：

```text
IrrigationWeb
  -> ZoneService / PlanService / RunController / Scheduler / HistoryService
      -> ConfigStore / EventService / BoardHardware
          -> Esp32Base / Arduino / GPIO
```

禁止反向依赖：

- `BoardHardware` 不知道 Web、计划、历史。
- `RunController` 不渲染页面。
- `IrrigationWeb` 不直接 `digitalWrite()`。
- `Scheduler` 不直接打开阀，只能请求 `RunController` 启动任务。
- `ConfigStore` 不做业务决策，只负责读写和默认值。

## 核心数据结构

### 常量

```cpp
constexpr uint8_t kMaxZones = 6;
constexpr uint8_t kDefaultEnabledZones = 1;
constexpr uint8_t kMaxPlanStartTimes = 4;
```

计划数量建议先设为 8 个。这个不是硬件限制，是产品复杂度限制；如果后续确认用户需要更多，再调整为编译期常量。

```cpp
constexpr uint8_t kMaxPlans = 8;
```

### ZoneConfig

```cpp
struct ZoneConfig {
    uint8_t id;                  // 1..6
    bool enabled;
    char name[24];
    uint32_t defaultDurationSec;
    uint32_t standardFlowMlPerMin;
    uint8_t valveIndex;          // 固定 0..5，不允许用户改
};
```

默认值：

```text
Zone 1: enabled = true
Zone 2..6: enabled = false
```

### WateringPlan

```cpp
struct StartTime {
    bool enabled;
    uint16_t minuteOfDay;        // 0..1439
};

struct WateringPlan {
    uint8_t id;
    bool enabled;
    char name[32];
    StartTime startTimes[4];
    uint32_t zoneDurationSec[6]; // 保存全部 6 个 Zone，执行时过滤
};
```

执行过滤规则：

```text
Zone 未启用 -> skip
durationSec == 0 -> skip
其它 -> 生成 WateringStep
```

### WateringRun

```cpp
enum class RunSource : uint8_t {
    Manual,
    Plan,
    RunPlanNow,
    Calibration
};

enum class RunState : uint8_t {
    Idle,
    Precheck,
    OpenValve,
    PumpSignalOn,
    PumpStartDelay,
    FlowGrace,
    Running,
    PumpSignalOff,
    PumpStopDelay,
    CloseValve,
    AdvanceStep,
    Finished
};

enum class RunResult : uint8_t {
    None,
    Completed,
    UserStopped,
    FaultStopped,
    Skipped
};

struct WateringStep {
    uint8_t zoneId;
    uint32_t targetDurationSec;
};

struct WateringRun {
    uint32_t id;
    RunSource source;
    uint8_t planId;              // 非计划来源时为 0
    RunState state;
    RunResult result;
    uint16_t reasonCode;
    WateringStep steps[6];
    uint8_t stepCount;
    uint8_t currentStep;
    uint32_t startedAtEpoch;
    uint32_t finishedAtEpoch;
    uint32_t stateEnteredMs;
};
```

`reasonCode` 用于记录停止、异常或跳过原因。先用枚举，不用字符串做运行时判断。

```cpp
enum class RunReason : uint16_t {
    None = 0,
    ManualRequest,
    PlanStartTime,
    RunPlanNow,
    UserStop,
    NoEffectiveStep,
    ControllerBusy,
    ZoneDisabled,
    InvalidDuration,
    ConfigInvalid,
    NoFlow,
    LowLevel,
    RebootRecoveredSafe
};
```

### SupplyConfig

```cpp
enum class ContactType : uint8_t {
    NormallyOpen,
    NormallyClosed
};

struct SupplyConfig {
    bool pumpEnabled;
    uint32_t pumpStartDelayMs;
    uint32_t pumpStopDelayMs;
    bool lowLevelEnabled;
    ContactType lowLevelContactType;
    uint32_t lowLevelDebounceMs;
};
```

低液位只在 `pumpEnabled && lowLevelEnabled` 时参与保护。

状态机只表达硬件动作阶段；完成、用户停止、异常停止和跳过用 `RunResult` 表达，不再为每一种结束结果增加一套状态。

### FlowConfig

```cpp
struct FlowConfig {
    uint32_t pulsesPerLiter;
    uint32_t startupGraceSec;
    uint32_t noFlowConfirmSec;
    uint32_t leakWindowSec;
    uint32_t leakPulseThreshold;
    uint8_t lowFlowPercent;
    uint16_t highFlowPercent;
};
```

流量方案保持简单：

- 无流量是停机保护。
- 待机漏水是提示。
- 低流量和过流量第一阶段只做提示，不做可配置停机策略。
- 每个 Zone 的标准流量只用于提示和历史分析，不参与复杂闭环控制。

## 配置与存储设计

### 配置聚合

业务配置在内存中聚合为一个只由 `ConfigStore` 读写的结构：

```cpp
struct IrrigationConfig {
    uint16_t version;
    SupplyConfig supply;
    FlowConfig flow;
    ValveConfig valve;
    ZoneConfig zones[6];
    WateringPlan plans[8];
};
```

运行时服务只读自己需要的配置副本或 const 引用。配置保存后由 `ConfigStore` 统一通知相关服务刷新，不允许各模块分散写文件。

### 存储分工

| 数据 | 存储位置 | 原因 |
| --- | --- | --- |
| WiFi、Web Auth、基础库参数 | `Esp32Base` 自己管理 | 不在本项目重复 |
| 供水、流量、阀驱动、Zone、Plan | LittleFS JSON | 一份业务配置，便于整体校验和迁移 |
| 当前未完成任务标记 | NVS 小记录 | 重启后记录异常结束 |
| 业务历史摘要 | LittleFS 追加文件或环形文件 | 可控容量 |
| 业务事件 | `Esp32BaseAppEventLog` | 复用基础库事件能力 |

`Esp32BaseAppConfig` 只用于少量基础参数入口或维护项。Zone/Plan 这种数组型业务配置由本项目 Web 页面管理，底层使用 `Esp32Base` 的 LittleFS 能力。

### 配置文件

建议文件：

```text
/irrigation/config.json
/irrigation/history/runs.log
```

每个 JSON 文件必须包含：

- `version`
- `updatedAt`
- `data`

写入策略：

1. 写临时文件。
2. 校验可读。
3. 替换正式文件。
4. 失败时保留旧配置。

默认配置：

- `Zone 1` 启用。
- `Zone 2..6` 停用。
- 无计划。
- 自吸泵禁用。
- 低液位禁用。
- 流量计每升脉冲数未校准时允许手动录入，但自动计划启动前必须有有效值。

## 调度

- 自动计划触发时，如果设备空闲，立即生成并执行 `WateringRun`。
- 自动计划触发时，如果设备忙，本次触发跳过并记录历史。
- 手动浇水和立即执行计划在设备忙时拒绝并显示原因。
- 不做任务排队、抢占、补跑或堆叠。
- 每个计划的每个启动时间每天最多触发一次。
- 系统重启后不补跑重启期间错过的计划。

成熟控制器常见的 `Program Stacking` 暂不实现。当前产品先保持确定、可解释、容易验证；如果后续确认用户需要计划堆叠，再单独增加。

Scheduler 只负责判断“现在是否应该启动某个计划”。它不直接操作硬件，不直接写历史；触发结果统一交给 `RunController` 和 `HistoryService`。

## RunController 状态机

### 任务生成

```text
Manual / Plan / RunPlanNow
  -> 读取 ZoneConfig
  -> 过滤未启用 Zone
  -> 过滤 duration = 0
  -> 生成 WateringStep[]
  -> Precheck
```

没有有效步骤时，任务不启动，记录 `skipped:no_enabled_zone_or_duration`。

### 状态定义

| 状态 | 进入动作 | 退出条件 |
| --- | --- | --- |
| `Idle` | 所有输出保持关闭 | 收到有效启动请求 |
| `Precheck` | 检查配置、设备空闲、Zone、时长、低液位 | 通过进入 `OpenValve`，失败进入 `Finished` |
| `OpenValve` | 打开当前 Zone 阀，启动吸合/PWM 控制 | 阀控制动作提交后进入下一状态 |
| `PumpSignalOn` | 自吸泵启用时输出干接点信号 | 立即进入 `PumpStartDelay` |
| `PumpStartDelay` | 等待泵启动稳定 | 到时进入 `FlowGrace` |
| `FlowGrace` | 等待水流建立 | 到时进入 `Running`；缺水或停止请求进入安全关闭 |
| `Running` | 计时、统计流量、检查保护 | 到时进入停泵/关阀；故障进入安全关闭 |
| `PumpSignalOff` | 自吸泵启用时取消干接点信号 | 立即进入 `PumpStopDelay` |
| `PumpStopDelay` | 等待泵停止 | 到时进入 `CloseValve` |
| `CloseValve` | 关闭当前 Zone 阀 | 进入 `AdvanceStep` |
| `AdvanceStep` | 写当前步骤结果，选择下一步 | 有下一步进入 `OpenValve`，否则进入 `Finished` |
| `Finished` | 写任务结果和历史，输出保持关闭 | 回到 `Idle` |

### 单步执行

无自吸泵：

```text
OpenValve
  -> FlowGrace
  -> Running
  -> CloseValve
  -> AdvanceStep / Finished
```

启用自吸泵：

```text
OpenValve
  -> PumpSignalOn
  -> PumpStartDelay
  -> FlowGrace
  -> Running
  -> PumpSignalOff
  -> PumpStopDelay
  -> CloseValve
  -> AdvanceStep / Finished
```

先开 Zone 阀再启动泵，是为了避免泵启动时出口完全关闭。

### 异常停止

任何异常停止都必须进入统一安全关闭：

```text
FaultDetected
  -> PumpSignalOff if pump enabled
  -> PumpStopDelay if pump enabled
  -> CloseValve if valve open
  -> DriverSafeOff if no active valve
  -> WriteHistory
  -> Idle
```

异常原因：

- 无流量。
- 自吸泵启用时低液位缺水。
- 配置损坏。
- 系统重启恢复到安全态。

低流量和过流量第一阶段只记录提示，不进入异常停机路径。

### 非阻塞要求

状态机每次 `handle(nowMs)` 只做短动作：

- 不使用长 `delay()`。
- 不在状态机里做文件写入大循环。
- 不在状态机里渲染 Web。
- 不在状态机里等待网络。
- 流量计脉冲计数必须由中断或轻量轮询维护，状态机只读取快照。

状态机完整性检查结论：

- 上电安全态不属于 `RunController` 状态机，由 `BoardHardware::safeOff()` 保证。
- 运行中任意停止原因都收敛到同一条安全关闭路径。
- 自吸泵启用和未启用共用同一状态机，只跳过泵相关状态。
- Zone 之间切换必须经过关闭当前阀，再进入下一步的 `OpenValve`。
- `delay()` 不能用于状态等待；所有等待都用时间戳判断，保证 Web、流量计数和停机动作不被阻塞。

## Web 信息结构

Web 面向最终用户，不做工程调试后台。

页面数据只来自业务服务的只读快照，不直接读取硬件寄存器或配置文件。

### 总览

回答：

- 当前是否正常。
- 是否正在浇水。
- 下一次自动浇水是什么。
- 今日浇过多少。
- 有没有异常。

信息：

- 当前状态：空闲 / 运行中 / 故障 / 停止。
- 当前任务卡片。
- 下一次计划。
- 今日用水量。
- 启用 Zone 数。
- 最近事件。

操作：

- 进入手动浇水。
- 查看计划。
- 停止当前任务。

### 运行

回答：

- 现在如何启动一次浇水。
- 如何立即执行一个计划。
- 运行时实时数据是什么。

信息和表单：

- 已启用 Zone 的手动时长表。
- 启动顺序浇水按钮。
- 计划选择器。
- 立即执行按钮。
- 当前流量、脉冲、估算水量。
- 停止按钮。

不显示未启用 Zone。

### 计划

结构：

- 左侧：计划列表。
- 右侧：计划编辑表单。

计划列表显示：

- 名称。
- 启用状态。
- 每日启动时间。
- Zone 时长摘要。
- 下一次运行时间。

计划编辑：

- 名称。
- 启用状态。
- 4 个启动时间槽。
- 已启用 Zone 的时长表。

未启用 Zone 不显示，但存储中保留其时长。

### Zone

结构：

- Zone 列表。
- Zone 详情表单。

Zone 列表显示 1..6 全部 Zone：

- Zone 编号。
- 名称。
- 启用状态。
- 默认时长。
- 标准流量。
- 最近浇水。

Zone 详情：

- 名称。
- 启用状态。
- 默认时长。
- 标准流量。
- 固定输出位置。
- 校准入口。

### 校准

包含：

- 流量计接水量校准。
- Zone 标准流量校准。

校准产生：

- `pulsesPerLiter`
- `standardFlowMlPerMin`
- 校准历史。

### 历史

筛选：

- 全部。
- 浇水。
- 异常。
- 配置。
- 校准。

记录字段：

- 时间。
- 类型。
- 来源。
- Zone。
- 计划。
- 目标时长。
- 实际时长。
- 脉冲数。
- 估算水量。
- 结束原因。

### 设置

设置页只放全局参数：

- 自吸泵启用。
- 泵启动/停止延时。
- 低液位启用。
- 低液位常开/常闭。
- 低液位去抖。
- 阀吸合时间。
- 阀 PWM 维持占空比。
- 流量保护参数。
- 历史保留策略。

WiFi、OTA、系统日志等维护能力由 `Esp32Base` 页面提供入口，不在业务设置页重做。

## 信息快照

为了让 Web、历史和状态机解耦，业务层提供以下只读快照：

```cpp
struct StatusSnapshot {
    bool configValid;
    bool busy;
    RunState runState;
    RunResult runResult;
    uint8_t activeZoneId;
    uint32_t currentFlowMlPerMin;
    uint32_t currentRunVolumeMl;
    uint32_t todayVolumeMl;
    uint8_t enabledZoneCount;
    uint8_t enabledPlanCount;
    uint32_t nextRunEpoch;
};

struct ZoneSnapshot {
    uint8_t id;
    bool enabled;
    const char* name;
    uint32_t defaultDurationSec;
    uint32_t standardFlowMlPerMin;
    uint32_t lastRunEpoch;
};

struct PlanSnapshot {
    uint8_t id;
    bool enabled;
    const char* name;
    uint16_t startTimes[4];
    uint8_t startTimeCount;
    uint32_t zoneDurationSec[6];
    uint32_t nextRunEpoch;
};
```

这些快照用于页面渲染和 API 返回。快照不直接暴露内部可写指针，避免 Web 层绕过服务修改状态。

## Web 路由建议

页面：

```text

/plans                    计划列表和编辑
/zones                    Zone 列表和详情
```

接口：

```text
GET  /api/status

POST /api/run/plan-now
POST /api/run/stop
GET  /api/plans
POST /api/plans/save

GET  /api/zones
POST /api/zones/save
POST /api/calibration/start
```

## 业务事件

事件写入 `Esp32BaseAppEventLog`。

建议事件：

| source | type | reason 示例 |
| --- | --- | --- |
| `run` | `started` | `manual` / `plan` / `run_plan_now` |
| `run` | `completed` | `all_steps_done` |
| `run` | `stopped` | `user` |
| `run` | `fault_stopped` | `no_flow` / `low_level` / `low_flow` / `high_flow` |
| `plan` | `triggered` | `start_time` |
| `plan` | `skipped` | `controller_busy` / `no_enabled_zone` |
| `zone` | `changed` | `enabled` / `disabled` / `renamed` |
| `calibration` | `saved` | `flow_meter` / `zone_flow` |
| `config` | `changed` | `supply` / `safety` / `valve` |

历史页面显示用户能理解的文案，事件日志保留机器可检索字段。

## 实现顺序

1. 补齐 `IrrigationTypes`、默认配置和 `ConfigStore`。
2. 完成 `BoardHardware`：安全态、阀输出、流量计计数、低液位输入、自吸泵输出。
3. 完成 `ZoneService` 和 `PlanService`，先支持配置读写和快照。
4. 完成 `RunController`，先跑通手动顺序浇水。
5. 完成 `FlowSafetyService`：无流量停机、漏水提示、低/过流量提示。
6. 完成 `Scheduler`：每天计划触发，忙时跳过。
7. 完成 `HistoryService` 和 `EventService`。
8. 完成 Web 页面和表单提交。
9. 做硬件联调和保护验证。

不要先写页面后补业务规则。执行状态机、配置存储和硬件安全态必须先稳定。

第一阶段验收：

- 上电所有输出关闭。
- 默认只启用 `Zone 1`。
- 手动顺序浇水能按已启用 Zone 执行。
- 自动计划按最多 4 个启动时间触发。
- 忙时手动和立即执行被拒绝，忙时自动计划跳过并记录。
- 无流量停机。
- 自吸泵启用时按顺序输出干接点信号。
- 低液位只在自吸泵和低液位都启用时停机。
- 重启后不恢复未完成任务，只记录异常结束。
