# 06 软件架构与信息结构

## 设计结论

当前软件只服务这一块 12V 单板智能浇水控制器。

最终软件模型采用成熟灌溉控制器常见结构：

```text
Controller
  -> Zone
  -> WateringPlan
  -> StartTime
  -> ZoneRuntime
  -> WateringRun
  -> WateringStep
  -> FlowMonitor
  -> PumpStartSignal
  -> DryRunProtection
  -> EventLog
```

术语固定：

- `Zone`：一路阀对应的浇水区域。中文界面也优先显示 `Zone 1`、`Zone 2`。
- `WateringPlan`：浇水计划。
- `StartTime`：计划每天启动的时间。一个计划最多 4 个启动时间。
- `ZoneRuntime`：一个计划中某个 Zone 的运行时长。
- `WateringRun`：一次实际执行的浇水任务。
- `WateringStep`：一次任务中的单个 Zone 执行步骤。

不再使用“水路”作为软件核心对象名。文档中如出现“水路”，只作为中文解释，不作为代码模型名。

## 产品级规则

### Zone

- 硬件最大支持 6 个 Zone。
- 系统初始化默认只启用 `Zone 1`。
- `Zone 2` 到 `Zone 6` 默认停用，由用户启用。
- 每个 Zone 固定对应一个阀输出，不允许重新映射。
- 未启用 Zone：
  - 不出现在手动浇水时长表中。
  - 不出现在计划编辑的 Zone 时长表中。
  - 不参与自动计划执行。
  - 已有计划中保留的时长不删除，重新启用后可继续使用。

### WateringPlan

计划固定为每天策略。

一个计划包含：

- 名称。
- 启用状态。
- 最多 4 个 `StartTime`。
- 6 个 Zone 的时长配置。

显示和编辑时只显示已启用 Zone。存储时保留 6 个 Zone 的时长，避免用户停用 Zone 后丢失原配置。

多个启动时间表示同一组 Zone 时长在一天内执行多次：

```text
日常浇水
  StartTime: 08:00, 16:00
  Zone 1: 10 min
  Zone 2: 8 min
```

如果上午和下午每个 Zone 的时长不同，应创建两个计划：

```text
上午浇水
  StartTime: 08:00
  Zone 1: 10 min

下午补水
  StartTime: 16:00
  Zone 1: 5 min
```

### WateringRun

手动浇水、计划到点、立即执行计划，最终都生成 `WateringRun`，由同一个状态机执行。

任务来源：

- `manual`：用户在运行页设置已启用 Zone 时长后启动。
- `plan`：每日计划到点自动触发。
- `run_plan_now`：用户选择一个计划立即执行。
- `calibration`：校准流程创建的临时任务。

同一时刻只允许一个 Zone 打开。

## 软件分层

### 1. 基础层：Esp32Base

基础层由 `../Esp32Base` 提供。

使用能力：

- 启动生命周期。
- WiFi。
- Web。
- Web 认证。
- OTA / Web OTA。
- LittleFS。
- 系统日志。
- App Events。
- App Config。
- 健康状态。
- 看门狗。
- NTP。
- mDNS / DNS。
- RTC 时间。

本项目不重复实现这些通用能力。

如果基础库缺少可复用的通用能力，应优先修改 `../Esp32Base`，不要在本项目里写局部旁路补丁。

### 2. 应用启动层：IrrigationApp

职责：

- 设置上电安全态。
- 初始化板级硬件。
- 注册业务 Web 路由。
- 加载业务配置。
- 初始化业务服务。
- 将业务循环挂到 `Esp32Base::handle()` 之后。

启动顺序：

```text
Serial
  -> BoardHardware::safeOff()
  -> WebRoutes::register()
  -> Esp32Base::begin()
  -> ConfigStore::load()
  -> DomainServices::begin()
  -> idle
```

上电阶段任何配置加载失败，都不能自动启动浇水。

### 3. 板级硬件层：BoardHardware

职责：

- 固定当前 PCB 引脚。
- 统一上电安全态。
- 提供最小硬件访问接口。

模块：

- `ValveDriver`
- `FlowMeter`
- `LowLevelInput`
- `PumpDryOutput`
- `RtcInput`
- `LocalButtons`

板级层只处理引脚和电平，不理解计划、Zone 名称、历史和 Web。

### 4. 设备能力层：Device Drivers

职责：

- 阀控制：全功率吸合 + PWM 维持。
- 流量计：中断计数、窗口统计、累计脉冲。
- 自吸泵信号：干节点输出启停。
- 低液位输入：常开/常闭转换、去抖。

设备能力层不直接决定任务是否启动，只向业务层报告当前能力状态。

### 5. 领域服务层：Domain Services

核心服务：

| 服务 | 职责 |
| --- | --- |
| `ZoneService` | 管理 6 个 Zone 的启用、名称、默认时长、标准流量 |
| `PlanService` | 管理计划、启动时间、Zone 时长 |
| `RunController` | 执行当前 `WateringRun` 状态机 |
| `Scheduler` | 按 RTC/NTP 时间触发每日计划 |
| `RunQueue` | 处理计划重叠时的排队 |
| `SafetyMonitor` | 汇总无流量、低流量、过流量、漏水、低液位保护 |
| `FlowMonitor` | 计算当前流量、累计用水、标准流量偏差 |
| `HistoryService` | 保存浇水、异常、配置、校准和用户操作 |
| `EventService` | 适配 `Esp32BaseAppEventLog` |

业务规则必须集中在领域服务层，不写到 Web 页面或硬件驱动里。

### 6. Web 应用层：IrrigationWeb

职责：

- 组织最终用户页面。
- 处理表单提交。
- 调用领域服务。
- 显示状态、错误和历史。

Web 层不直接操作 GPIO。

## 核心数据结构

### 常量

```cpp
constexpr uint8_t kMaxZones = 6;
constexpr uint8_t kDefaultEnabledZones = 1;
constexpr uint8_t kMaxPlanStartTimes = 4;
constexpr uint8_t kMaxRunQueueDepth = 4;
```

计划数量建议先设为 8 个。这个不是硬件限制，是产品复杂度限制；如果后续确认用户需要更多，再调整为编译期常量。

```cpp
constexpr uint8_t kMaxPlans = 8;
```

### ZoneConfig

```cpp
struct ZoneConfig {
    uint8_t id;                  // 1..6
    bool enabled;
    char name[24];
    uint32_t defaultDurationSec;
    uint32_t standardFlowMlPerMin;
    uint8_t valveIndex;          // 固定 0..5，不允许用户改
};
```

默认值：

```text
Zone 1: enabled = true
Zone 2..6: enabled = false
```

### WateringPlan

```cpp
struct StartTime {
    bool enabled;
    uint16_t minuteOfDay;        // 0..1439
};

struct WateringPlan {
    uint8_t id;
    bool enabled;
    char name[32];
    StartTime startTimes[4];
    uint32_t zoneDurationSec[6]; // 保存全部 6 个 Zone，执行时过滤
};
```

执行过滤规则：

```text
Zone 未启用 -> skip
durationSec == 0 -> skip
其它 -> 生成 WateringStep
```

### WateringRun

```cpp
enum class RunSource : uint8_t {
    Manual,
    Plan,
    RunPlanNow,
    Calibration
};

enum class RunState : uint8_t {
    Idle,
    Precheck,
    OpenValve,
    PumpStartDelay,
    PumpSignalOn,
    FlowGrace,
    Running,
    StepStopping,
    PumpStopDelay,
    CloseValve,
    NextStep,
    Completed,
    UserStopped,
    FaultStopped,
    Skipped
};

struct WateringStep {
    uint8_t zoneId;
    uint32_t targetDurationSec;
};

struct WateringRun {
    uint32_t id;
    RunSource source;
    uint8_t planId;              // 非计划来源时为 0
    RunState state;
    WateringStep steps[6];
    uint8_t stepCount;
    uint8_t currentStep;
    uint32_t startedAtEpoch;
    uint32_t finishedAtEpoch;
};
```

### SupplyConfig

```cpp
enum class ContactType : uint8_t {
    NormallyOpen,
    NormallyClosed
};

struct SupplyConfig {
    bool pumpEnabled;
    uint32_t pumpStartDelayMs;
    uint32_t pumpStopDelayMs;
    bool lowLevelEnabled;
    ContactType lowLevelContactType;
    uint32_t lowLevelDebounceMs;
};
```

低液位只在 `pumpEnabled && lowLevelEnabled` 时参与保护。

### FlowConfig

```cpp
enum class FlowFaultAction : uint8_t {
    WarnOnly,
    StopRun
};

struct FlowConfig {
    uint32_t pulsesPerLiter;
    uint32_t startupGraceSec;
    uint32_t noFlowConfirmSec;
    uint32_t leakWindowSec;
    uint32_t leakPulseThreshold;
    uint8_t lowFlowPercent;
    uint16_t highFlowPercent;
    FlowFaultAction lowFlowAction;
    FlowFaultAction highFlowAction;
};
```

## 存储设计

### 存储分工

| 数据 | 存储位置 | 原因 |
| --- | --- | --- |
| WiFi、Web Auth、基础库参数 | `Esp32Base` 自己管理 | 不在本项目重复 |
| 供水、流量、阀驱动等少量全局参数 | NVS / `Esp32BaseConfig` | 结构简单 |
| Zone 配置 | LittleFS JSON | 数组结构，方便版本迁移 |
| Plan 配置 | LittleFS JSON | 计划和时长数组较多，不适合塞入 App Config 字段 |
| 当前未完成任务标记 | NVS 小记录 | 重启后记录异常结束 |
| 业务历史摘要 | LittleFS 追加文件或环形文件 | 可控容量 |
| 业务事件 | `Esp32BaseAppEventLog` | 复用基础库事件能力 |

`Esp32BaseAppConfig` 适合基础参数和少量业务参数；Zone/Plan 这种数组型业务配置由本项目 Web 页面管理，底层仍使用 `Esp32Base` 的 LittleFS 能力。

### 配置文件

建议文件：

```text
/irrigation/config/system.json
/irrigation/config/zones.json
/irrigation/config/plans.json
/irrigation/history/runs.log
```

每个 JSON 文件必须包含：

- `version`
- `updatedAt`
- `data`

写入策略：

1. 写临时文件。
2. 校验可读。
3. 替换正式文件。
4. 失败时保留旧配置。

## 调度与排队

成熟灌溉控制器常见做法是 `Program Stacking`：如果多个计划时间重叠，后触发的计划等待前一个完成。

本项目采用受限排队：

- 自动计划触发时，如果设备空闲，立即生成并执行 `WateringRun`。
- 自动计划触发时，如果设备忙，进入 `RunQueue`。
- 队列最大深度 4。
- 队列满时跳过本次计划并记录历史。
- 手动浇水和立即执行计划在设备忙时默认拒绝，不自动排队。
- 用户停止当前任务时，不自动清空队列；页面提供清空等待任务入口。

这样既符合成熟产品的计划堆叠习惯，又避免用户手动操作被系统延迟执行造成误解。

## RunController 状态机

### 任务生成

```text
Manual / Plan / RunPlanNow
  -> 读取 ZoneConfig
  -> 过滤未启用 Zone
  -> 过滤 duration = 0
  -> 生成 WateringStep[]
  -> Precheck
```

没有有效步骤时，任务不启动，记录 `skipped:no_enabled_zone_or_duration`。

### 单步执行

无自吸泵：

```text
OpenValve
  -> FlowGrace
  -> Running
  -> CloseValve
  -> NextStep / Completed
```

启用自吸泵：

```text
OpenValve
  -> PumpSignalOn
  -> PumpStartDelay
  -> FlowGrace
  -> Running
  -> PumpSignalOff
  -> PumpStopDelay
  -> CloseValve
  -> NextStep / Completed
```

先开 Zone 阀再启动泵，是为了避免泵启动时出口完全关闭。

### 异常停止

任何异常停止都必须进入统一安全关闭：

```text
FaultDetected
  -> PumpSignalOff
  -> CloseValve
  -> DriverSafeOff if no active valve
  -> WriteHistory
  -> Idle
```

异常原因：

- 无流量。
- 低流量且策略为停机。
- 过流量且策略为停机。
- 自吸泵启用时低液位缺水。
- 配置损坏。
- 系统重启恢复到安全态。

## Web 信息结构

Web 面向最终用户，不做工程调试后台。

### 总览

回答：

- 当前是否正常。
- 是否正在浇水。
- 下一次自动浇水是什么。
- 今日浇过多少。
- 有没有异常。

信息：

- 当前状态：空闲 / 运行中 / 故障 / 停止。
- 当前任务卡片。
- 下一次计划。
- 今日用水量。
- 启用 Zone 数。
- 最近事件。

操作：

- 进入手动浇水。
- 查看计划。
- 停止当前任务。

### 运行

回答：

- 现在如何启动一次浇水。
- 如何立即执行一个计划。
- 运行时实时数据是什么。

信息和表单：

- 已启用 Zone 的手动时长表。
- 启动顺序浇水按钮。
- 计划选择器。
- 立即执行按钮。
- 当前流量、脉冲、估算水量。
- 停止按钮。

不显示未启用 Zone。

### 计划

结构：

- 左侧：计划列表。
- 右侧：计划编辑表单。

计划列表显示：

- 名称。
- 启用状态。
- 每日启动时间。
- Zone 时长摘要。
- 下一次运行时间。

计划编辑：

- 名称。
- 启用状态。
- 4 个启动时间槽。
- 已启用 Zone 的时长表。

未启用 Zone 不显示，但存储中保留其时长。

### Zone

结构：

- Zone 列表。
- Zone 详情表单。

Zone 列表显示 1..6 全部 Zone：

- Zone 编号。
- 名称。
- 启用状态。
- 默认时长。
- 标准流量。
- 最近浇水。

Zone 详情：

- 名称。
- 启用状态。
- 默认时长。
- 标准流量。
- 固定输出位置。
- 校准入口。

### 校准

包含：

- 流量计接水量校准。
- Zone 标准流量校准。

校准产生：

- `pulsesPerLiter`
- `standardFlowMlPerMin`
- 校准历史。

### 历史

筛选：

- 全部。
- 浇水。
- 异常。
- 配置。
- 校准。

记录字段：

- 时间。
- 类型。
- 来源。
- Zone。
- 计划。
- 目标时长。
- 实际时长。
- 脉冲数。
- 估算水量。
- 结束原因。

### 设置

设置页只放全局参数：

- 自吸泵启用。
- 泵启动/停止延时。
- 低液位启用。
- 低液位常开/常闭。
- 低液位去抖。
- 阀吸合时间。
- 阀 PWM 维持占空比。
- 流量保护参数。
- 历史保留策略。

WiFi、OTA、系统日志等维护能力由 `Esp32Base` 页面提供入口，不在业务设置页重做。

## 业务事件

事件写入 `Esp32BaseAppEventLog`。

建议事件：

| source | type | reason 示例 |
| --- | --- | --- |
| `run` | `started` | `manual` / `plan` / `run_plan_now` |
| `run` | `completed` | `all_steps_done` |
| `run` | `stopped` | `user` |
| `run` | `fault_stopped` | `no_flow` / `low_level` / `low_flow` / `high_flow` |
| `plan` | `triggered` | `start_time` |
| `plan` | `queued` | `controller_busy` |
| `plan` | `skipped` | `queue_full` / `no_enabled_zone` |
| `zone` | `changed` | `enabled` / `disabled` / `renamed` |
| `calibration` | `saved` | `flow_meter` / `zone_flow` |
| `config` | `changed` | `supply` / `safety` / `valve` |

历史页面显示用户能理解的文案，事件日志保留机器可检索字段。

## 实现顺序

1. 补齐类型定义和配置存储。
2. 完成板级硬件抽象。
3. 完成 ZoneService 和 PlanService。
4. 完成 RunController 状态机。
5. 完成 FlowMonitor 和 SafetyMonitor。
6. 完成 Scheduler 和 RunQueue。
7. 完成 HistoryService 和 EventService。
8. 完成 Web 页面。
9. 做硬件联调和保护验证。

不要先写页面后补业务规则。执行状态机、配置存储和硬件安全态必须先稳定。
