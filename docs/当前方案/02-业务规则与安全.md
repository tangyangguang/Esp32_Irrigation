# 02 业务规则与安全

## 术语约定

方案和界面统一使用以下名称：

| 名称 | 含义 |
| --- | --- |
| 控制器 | 当前 ESP32 单板设备 |
| 区域 | 一路电磁阀控制的浇水范围；英文资料常称 Zone 或 Station |
| 浇水计划 | 一组自动浇水安排；对应成熟控制器中的 Program 或 Schedule |
| 启动时间 | 一个计划每天开始执行的时间 |
| 区域运行时长 | 计划中某个区域每次浇水的时间 |
| 基准流量 | 某个区域正常浇水时学习或人工确认的预期流量 |
| 手动浇水 | 用户立即启动一次浇水操作 |
| 流量报警 | 无流量、低流量、高流量或非计划流量异常 |
| 浇水记录 | 实际发生过的浇水业务数据 |
| 事件 | 需要追溯的重要动作、报警和异常信息 |

后续软件设计可以使用英文代码名称，但必须与这些产品概念一一对应，不能为了代码结构再创造一套用户难以理解的概念。

## 区域

- 控制器固定管理 6 个区域。
- 默认启用区域 1、区域 2；区域 3 至区域 6 默认停用。
- 用户可以修改区域名称、启用状态和基准流量。
- 停用区域不出现在手动浇水和计划时长表中。
- 已保存的区域时长在区域停用后继续保留，重新启用后可以继续使用。

## 浇水计划

- 最多保存 8 个计划。
- 一个计划包含名称、启用状态、最多 4 个每日启动时间和各区域运行时长。
- 当前周期固定为每天。
- 时长为 0 的区域不执行。
- 禁用计划保留配置，但不自动启动，也不提供“立即执行”。
- 已启用计划之间不能使用完全相同的启动时间，避免同一分钟产生不明确的执行顺序。
- 修改计划只影响之后的浇水，不改变已经开始的浇水操作。

多个启动时间表示同一组区域时长每天执行多次。如果不同时段需要不同的区域时长，应建立不同计划。

## 手动浇水

手动浇水支持两种方式：

- 为需要浇水的区域设置本次运行时长，然后按区域编号顺序执行。
- 选择一个已启用计划立即执行，使用该计划当前保存的区域时长。

只给一个区域设置非零时长，就是单区域手动浇水，不需要单独设计另一种运行模式。

## 运行规则

- 同一时刻只执行一次浇水操作。
- 一次浇水可以包含多个区域，但同一时刻只打开一个区域。
- 当前区域结束并安全关闭后，才启动下一区域。
- 设备正在浇水或校准时，新的手动请求直接拒绝并说明原因。
- 自动计划到点但设备正忙时，本次计划跳过并记录事件。
- 错过的计划不补执行。
- 用户可以随时停止当前浇水。
- 普通单区域最长运行 120 分钟。
- 区域运行时长从流量建立后的正常浇水阶段开始计算，不包含阀启动、泵启动延时和流量建立时间。

启用自吸泵时，单个区域按以下顺序运行：

1. 打开区域阀。
2. 输出自吸泵干接点控制信号。
3. 等待泵启动延时。
4. 等待流量建立。
5. 开始区域运行时长计时和流量监测。
6. 结束时先取消泵控制信号。
7. 等待停泵后关阀延时。
8. 关闭区域阀。

阀的全功率吸合和转入维持占空比由阀驱动独立计时，不额外推迟泵控制信号。

## 流量监测与报警

### 无流量报警

无流量报警是所有出水过程的基本保护，包括普通浇水、流量计校准和区域流量学习。

- 打开区域后先等待流量建立宽限时间。
- 宽限期结束后，连续没有有效脉冲达到确认时间，停止整次浇水。
- 流量已经建立后，运行途中连续没有有效脉冲达到确认时间，同样停止整次浇水。
- 停止时关闭泵控制信号和区域阀，并记录报警事件。

默认流量建立宽限为 10 秒，连续无流量确认时间为 10 秒。自吸泵实物测试时必须确认这个时间不会造成不可接受的干抽风险。

### 低流量和高流量报警

低流量和高流量使用区域基准流量判断。区域没有有效基准流量时，这两项监测不启用，无流量保护仍然有效。

- 默认低流量阈值：基准流量的 30%。
- 默认高流量阈值：基准流量的 200%。
- 默认持续确认时间：20 秒。
- 默认动作：只报警，当前浇水继续。
- 可选动作：报警并停止整次浇水。

系统只有一个总流量计，不能可靠判断异常来自当前末端、公共供水还是主管路，因此当前不设计自动隔离故障区域后继续浇水的复杂诊断流程。

同一区域一次运行中的同类报警只记录一次，避免事件重复刷屏。

### 非计划流量报警

所有区域阀和泵控制信号都关闭时检测到持续水流，称为非计划流量，通常意味着可能漏水或阀未完全关闭。

- 区域关闭后先等待 30 秒，再开始非计划流量检测，避免管路余流造成误报警。
- 默认检测窗口：30 秒。
- 默认阈值：窗口内 3 个脉冲。
- 达到阈值后显示报警并记录一次事件。
- 连续一个完整窗口恢复正常后解除报警。
- 报警没有恢复前不重复写入相同事件。
- 该报警只提示，不自动打开或关闭其它设备。

## 自动调度

- 产品时间按 UTC+8 使用，不使用夏令时。
- 自动计划只在时间可信时运行。
- 控制器启动时所在的当前分钟不触发计划，避免重启落在启动时间内造成重复浇水。
- 时间向前调整时不补执行已经跳过的时间。
- 时间向后调整时，同一个启动时间当天最多执行一次。
- 重启后仍需保证当天已经处理过的启动时间不重复执行。
- 时间无效时，手动浇水仍可使用；下一次计划和今日统计显示为不可用。

具体如何保存当天的执行标记，留到软件设计阶段决定。

## 流量校准与学习

### 流量计校准

目的：得到流量计每升脉冲数。

1. 选择一个已启用区域和最长出水时间。
2. 启动出水并累计原始脉冲。
3. 用户接水后停止，输入实测水量。
4. 至少采集 2 个有效样本，推荐 3 个。
5. 最大水量与最小水量至少相差 2 L。
6. 两个样本使用差分计算，三个样本使用线性拟合。
7. 拟合最大相对误差建议不超过 8%，超过时提示重新采样。
8. 保存计算得到的脉冲/L。

校准默认最长出水 10 分钟，最高 60 分钟。校准过程中保留无流量保护。

### 区域流量学习

目的：学习某个区域正常浇水时的基准流量，用于低流量和高流量报警。

1. 启动目标区域。
2. 等待流量建立；该阶段不参与计算。
3. 每秒读取流量，使用最近 5 秒窗口计算实时流量。
4. 至少取得 5 个有效窗口。
5. 最近窗口最大值与最小值之差不超过平均值的 10% 时认为稳定。
6. 将稳定窗口平均值作为建议基准流量，用户确认后保存。

流量学习保留无流量保护，不执行低流量和高流量报警。

## 默认配置

| 配置 | 默认值 |
| --- | ---: |
| 区域 1、区域 2 | 启用 |
| 区域 3 至区域 6 | 停用 |
| 浇水计划 | 无 |
| 自吸泵控制 | 禁用 |
| 流量计脉冲/L | 250 |
| 阀全功率吸合时间 | 3 秒 |
| 阀维持占空比 | 75% |
| 泵启动延时 | 0 秒 |
| 停泵后关阀延时 | 1 秒 |
| 流量建立宽限 | 10 秒 |
| 连续无流量确认 | 10 秒 |
| 区域关闭后检测延时 | 30 秒 |
| 非计划流量检测 | 30 秒内 3 个脉冲 |
| 低/高流量持续确认 | 20 秒 |
| 低流量阈值 | 基准流量的 30% |
| 高流量阈值 | 基准流量的 200% |
| 低/高流量动作 | 只报警 |

默认 250 脉冲/L 只是首次使用的初始系数，不能自动标记为“已校准”。用户完成流量计校准或明确确认参数后，页面才显示已校准状态。

## 浇水记录与事件

### 浇水记录

浇水记录是实际浇水的业务数据，由本项目保存和管理。

记录至少能够回答：

- 什么时间浇水。
- 来源是手动还是计划。
- 执行了哪些区域。
- 每个区域计划运行多久、实际运行多久。
- 每个区域记录了多少脉冲、估算用了多少水。
- 是否正常完成，为什么提前结束。

只要已经实际开始出水，就应产生浇水记录。校准和流量学习默认不写普通浇水记录。

### 事件

事件用于记录重要动作、报警和异常，使用 `Esp32Base` 的 App Events 能力。本项目只定义事件内容，不重新实现事件保存、分页、筛选、导出、清除和容量管理。

业务事件包括：

- 自动计划跳过。
- 手动请求被拒绝。
- 用户停止浇水。
- 无流量、低流量、高流量和非计划流量报警。
- 配置变化。
- 校准或流量学习保存、失败。
- 重启导致正在进行的浇水中断。
- 浇水记录保存失败。

设备启动、WiFi、时间同步、OTA 和文件系统等系统运行信息由 `Esp32Base` 负责。只有它们直接影响浇水业务时，本项目才记录对应的业务影响事件。

## Web 信息结构

业务页面保持为：

1. 总览
2. 运行
3. 计划
4. 区域
5. 浇水记录
6. 浇水设置

流量计校准从浇水设置进入，区域流量学习从区域详情进入，不单独占用顶层导航。

浇水记录页面只管理浇水记录。事件查看和管理使用 `Esp32Base` 提供的页面入口。

本地 LCD2004 和按钮操作在 Web 功能稳定后单独设计，届时从这些已经确定的业务功能中选择最常用操作。
