# 02 业务规则与安全

## 术语约定

方案和界面统一使用以下名称：

| 名称 | 含义 |
| --- | --- |
| 控制器 | 当前 ESP32 单板设备 |
| 区域 | 一路电磁阀控制的浇水范围；英文资料常称 Zone 或 Station |
| 浇水计划 | 一组区域运行时长，可手动执行，也可选择参加每日定时调度 |
| 自动执行 | 单个计划是否参加每日定时调度，不影响手动选择该计划 |
| 暂停自动浇水 | 暂时阻止全部计划自动启动的总控状态，不影响手动浇水 |
| 启动时间 | 一个计划每天开始执行的时间 |
| 区域运行时长 | 计划中某个区域每次浇水的时间 |
| 基准流量 | 某个区域通过流量学习并确认保存的正常预期流量 |
| 手动浇水 | 用户立即启动一次浇水操作 |
| 流量报警 | 无流量、低流量、高流量或非计划流量异常 |
| 浇水记录 | 实际发生过的浇水业务数据 |
| 事件 | 需要追溯的重要动作、报警和异常信息 |

后续软件设计可以使用英文代码名称，但必须与这些产品概念一一对应，不能为了代码结构再创造一套用户难以理解的概念。

## 区域

- 控制器固定管理 6 个区域。
- 默认启用区域 1、区域 2；区域 3 至区域 6 默认停用。
- 用户可以修改区域名称和启用状态；名称去除首尾空白后为 1–20 个 Unicode 字符且 UTF-8 编码不超过 63 字节，不接受 U+0000–U+001F 和 U+007F 控制字符。基准流量通过区域流量学习并确认后保存。
- 区域基准流量没有默认值，未学习时页面显示“未学习”。
- 停用区域不出现在手动浇水和计划时长表中。
- 已保存的区域时长在区域停用后继续保留，重新启用后可以继续使用。
- 如果停用区域会使某个开启自动执行的计划失去全部可运行区域，本次保存应拒绝并指出受影响计划；用户先关闭该计划的自动执行或调整计划后再停用区域。没有开启自动执行的计划可以保留为暂时不可运行状态。

## 浇水计划

- 最多保存 8 个计划。
- 一个计划包含名称、自动执行开关、最多 4 个每日启动时间和各区域运行时长；已配置计划的名称使用与区域名称相同的长度和编码规则。
- 当前周期固定为每天。
- 区域运行时长按整分钟配置；`0` 表示不执行，非零值为 1 至 120 分钟。
- 至少包含一个当前启用区域的非零时长，才是可运行的已配置计划；只用于手动运行时可以不设置启动时间。
- 关闭自动执行的计划保留全部配置且不参加定时调度，但仍可被用户手动选择并立即执行。
- 所有开启自动执行的启动项在全局范围内不能使用相同时间，包括同一计划的多个启动项，避免同一分钟产生不明确的执行顺序；开启开关时必须同时校验冲突。
- 修改计划只影响之后的浇水，不改变已经开始的浇水操作。

多个启动时间表示同一组区域时长每天执行多次。如果不同时段需要不同的区域时长，应建立不同计划。

## 手动浇水

手动浇水支持两种方式：

- 为需要浇水的区域设置本次运行时长，然后按区域编号顺序执行。
- 选择任意一个已经配置的计划立即执行，使用该计划当前保存的区域时长，不受其自动执行开关影响。

只给一个区域设置非零时长，就是单区域手动浇水，不需要单独设计另一种运行模式。

## 运行规则

- 同一时刻只执行一次浇水操作。
- 一次浇水可以包含多个区域，但同一时刻只打开一个区域。
- 当前区域结束并安全关闭后，才启动下一区域。
- 设备正在浇水或校准时，新的手动请求直接拒绝并说明原因。
- 自动计划到点但设备正忙时，本次计划跳过并记录事件。
- 错过的计划不补执行。
- 用户可以随时停止当前浇水。
- 普通单区域最长运行 120 分钟。
- 区域运行时长从流量建立后的正常浇水阶段开始计算，不包含阀启动、泵启动延时和流量建立时间。

启用自吸泵时，单个区域按以下顺序运行：

1. 打开区域阀。
2. 输出自吸泵干接点控制信号。
3. 等待泵启动延时。
4. 等待流量建立。
5. 开始区域运行时长计时和流量监测。
6. 结束时先取消泵控制信号。
7. 等待停泵后关阀延时。
8. 关闭区域阀。

阀的全功率吸合和转入维持占空比由阀驱动独立计时，不额外推迟泵控制信号。

## 流量监测与报警

### 无流量报警

无流量报警是所有出水过程的基本保护，包括普通浇水、流量计校准和区域流量学习。

- 打开区域并完成泵启动延时后开始等待流量建立；在流量建立超时内没有检测到有效脉冲，停止整次浇水。
- 流量已经建立后，运行途中连续没有有效脉冲达到运行中无流量超时，同样停止整次浇水。
- 停止时关闭泵控制信号和区域阀，并记录报警事件。

默认流量建立超时为 20 秒，运行中无流量超时为 10 秒。启动阶段直接使用一个总超时，不把同一件事拆成“宽限”和“确认”两个参数。自吸泵实物测试时必须确认启动超时不会造成不可接受的干抽风险。

### 低流量和高流量报警

低流量和高流量使用区域基准流量判断。区域没有有效基准流量时，这两项监测不启用，无流量保护仍然有效。

- 默认低流量阈值：基准流量的 30%。
- 默认高流量阈值：基准流量的 200%。
- 默认持续确认时间：20 秒。
- 默认动作：只报警，当前浇水继续。
- 可选动作：报警并停止整次浇水。

系统只有一个总流量计，不能可靠判断异常来自当前末端、公共供水还是主管路，因此当前不设计自动隔离故障区域后继续浇水的复杂诊断流程。

同一区域一次运行中的同类报警只记录一次，避免事件重复刷屏。

这些阈值是未取得现场基准数据前的保守初始值。完成区域流量学习后，应结合水塔水位变化、实际管路和阀门测试结果调整，不能把默认值视为通用行业标准。

### 非计划流量报警

所有区域阀和泵控制信号都关闭时检测到持续水流，称为非计划流量，通常意味着可能漏水或阀未完全关闭。

- 区域关闭后先等待 30 秒，再开始非计划流量检测，避免管路余流造成误报警。
- 默认检测窗口：30 秒。
- 默认阈值：窗口内 3 个脉冲。
- 达到阈值后显示报警并记录一次事件。
- 连续一个完整窗口恢复正常后解除报警。
- 报警没有恢复前不重复写入相同事件。
- 该报警只提示，不自动打开或关闭其它设备。

## 自动调度

- 自动浇水总控默认为正常运行。用户可以暂停全部计划的自动执行，并选择“暂停至手动恢复”或“暂停到指定日期和时间”。
- “暂停一段时间”是便捷输入方式，保存时统一换算为明确的恢复日期和时间，不另行维护易受重启影响的倒计时。
- 时间不可信时只能设置无限期暂停；暂停一段时间或暂停到指定时刻必须等待时间可信后设置。手动恢复不受时间状态影响。
- 设置定时恢复必须以可信时间为前提；时间失去可信状态时继续保持暂停，直到时间重新可信且已经到达恢复时刻。
- 暂停只阻止之后的自动计划，不停止已经开始的浇水，也不影响任何手动浇水方式。
- 暂停期间到点的计划直接跳过且不补执行。系统仍将相应启动项记为已处理，但不为每个跳过项重复写事件。
- 用户暂停、用户恢复和到期自动恢复记录事件。暂停原因属于用户决定，当前不设计天气等自动暂停来源。
- 产品时间按 UTC+8 使用，不使用夏令时。
- 自动计划只在时间可信时运行；本地日期早于 2026-01-01、RTC 状态异常且没有 NTP 时间时，均视为时间无效。
- 系统无法自行判断几分钟以内的实际误差，只显示当前时间来源是 RTC 还是 NTP。
- 只有 RTC 时间比设备最后一次可信时间倒退超过配置阈值时，才显示“RTC 时间倒退”并停止自动调度；默认阈值为 5 分钟，可以在系统参数中修改。
- NTP 成功校时后以 NTP 为准并解除“RTC 时间倒退”；用户主动暂停自动浇水的状态不受影响。
- 控制器启动时所在的当前分钟不触发计划，避免重启落在启动时间内造成重复浇水。
- 时间首次变为可信，或本地分钟不是按正常连续顺序进入时，只重新建立调度检查点，不触发跳变后落入的当前分钟。
- 时间向前调整时不补执行已经跳过的时间。
- 时间向后调整时，同一个启动时间当天最多执行一次。
- 重启后仍需保证当天已经处理过的启动时间不重复执行。
- 时间无效时，手动浇水仍可使用；下一次计划和今日统计显示为不可用。

当天已处理标记的保存和防重复方式见 `03-软件设计原则与边界.md`。

## 设备启动与掉电估算

- 每次设备启动及其复位原因使用 `Esp32Base` 系统日志记录，不在业务事件中重复保存。
- 当前硬件不能在突然断电后继续写入，因此不能记录精确断电时间，只能保存“最后一次确认在线”的时间并估算范围。
- 存活检查点默认间隔为 12 小时，可以按小时配置；`0` 表示关闭周期检查点。
- 浇水记录或事件已经提供了更新的可信时间时，重新计算检查点间隔，不额外写入检查点；浇水过程中也不周期写入。
- 新启动时结合上次确认在线时间、检查点间隔和本次启动时间显示可能断电时间及可能停电时长范围，不能显示成精确结论。
- 时间无效或启动原因不是可能掉电时，只显示已知事实，不估算停电时长。

## 流量校准与学习

### 流量计校准

目的：得到流量计每升脉冲数。

1. 选择一个已启用区域和最长出水时间。
2. 启动出水并累计原始脉冲。
3. 用户接水后停止，输入实测水量；单个样本水量至少为 1 L。
4. 一个有效样本即可计算，推荐采集 3 个样本降低人工量水误差。
5. 每个样本按“脉冲数 ÷ 实测升数”得到脉冲/L；多个样本使用“总脉冲数 ÷ 总实测升数”合并，水量较大的样本自然获得更高权重。
6. 页面显示各样本相对合并结果的偏差；最大偏差超过 8% 时提示样本不稳定，建议重新采样。
7. 用户确认后保存当前脉冲/L，不保存额外的“已校准”状态。

校准默认最长出水 10 分钟，最高 60 分钟。校准过程中保留无流量保护。

### 区域流量学习

目的：学习某个区域正常浇水时的基准流量，用于低流量和高流量报警。

1. 启动目标区域。
2. 等待流量建立；该阶段不参与计算。
3. 每秒读取流量，使用最近 5 秒窗口计算实时流量。
4. 至少取得 5 个有效窗口。
5. 最近窗口最大值与最小值之差不超过平均值的 10% 时认为稳定。
6. 将稳定窗口平均值作为建议基准流量，用户确认后保存。

流量学习保留无流量保护，不执行低流量和高流量报警。一次学习最长运行 10 分钟；达到上限仍未稳定时自动停止并提示重新检查供水条件，不能无限出水等待稳定。

## 默认配置与有效范围

| 配置 | 默认值 | 有效范围 |
| --- | ---: | --- |
| 区域 1、区域 2 | 启用 | 启用或停用 |
| 区域 3 至区域 6 | 停用 | 启用或停用 |
| 浇水计划 | 无 | 最多 8 个 |
| 自动浇水总控 | 正常运行 | 正常运行、无限期暂停、暂停到指定时刻 |
| RTC 时间倒退阈值 | 5 分钟 | 1–60 分钟 |
| 存活检查点间隔 | 12 小时 | 0 或 1–168 小时；0 表示关闭 |
| 自吸泵控制 | 禁用 | 启用或禁用 |
| 流量计脉冲/L | 250.00 | 0.01–100000.00 |
| 阀全功率吸合时间 | 3 秒 | 0.1–10.0 秒，步进 0.1 秒 |
| 阀 PWM 频率 | 20 kHz | 1–25 kHz |
| 阀维持占空比 | 75% | 1%–100% |
| 泵启动延时 | 0 秒 | 0–60 秒 |
| 停泵后关阀延时 | 1 秒 | 0–10 秒 |
| 流量建立超时 | 20 秒 | 1–120 秒 |
| 运行中无流量超时 | 10 秒 | 1–60 秒 |
| 区域关闭后检测延时 | 30 秒 | 0–300 秒 |
| 非计划流量检测 | 30 秒内 3 个脉冲 | 窗口 1–300 秒；阈值 1–65535 个脉冲 |
| 低/高流量持续确认 | 20 秒 | 1–300 秒 |
| 低流量阈值 | 基准流量的 30% | 1%–99% |
| 高流量阈值 | 基准流量的 200% | 101%–1000% |
| 低/高流量动作 | 只报警 | 只报警或停止整次浇水 |

250.00 脉冲/L 是首次使用的默认流量系数。页面保留两位小数，配置文件和内存使用放大 100 倍的整数保存，例如 `250.37` 保存为 `25037`。100 倍只是两位小数的精确表示，不是额外改变系数；当前没有三位小数需求，因此不使用 1000 倍。输入值直接按十进制文本转换，不先经过浮点数。用户可以直接修改，也可以通过流量计校准重新计算；系统只保存和使用当前系数，不维护额外的校准状态。

按脉冲数估算水量时使用 64 位整数计算：`估算水量 mL = 四舍五入(脉冲数 × 100000 ÷ 放大后的系数)`。这样只处理一次比例，既保留两位小数，也避免中间乘法溢出和浮点累计误差。

## 浇水记录与事件

### 浇水记录

浇水记录是实际浇水的业务数据。本项目定义其字段、业务校验和展示，底层保存、分页、轮换、损坏检查和清空使用 `Esp32BaseRecordStore`。

记录至少能够回答：

- 什么时间浇水。
- 来源是手动还是计划。
- 执行了哪些区域。
- 每个区域计划运行多久、实际运行多久。
- 每个区域记录了多少脉冲、估算用了多少水。
- 整次历时、计划浇水总时间、实际浇水总时间、总脉冲数和历史估算总水量。
- 是否正常完成，为什么提前结束。

只要已经实际开始出水，就应产生浇水记录。校准和流量学习默认不写普通浇水记录。

整次历时由记录基础元数据提供；其余总计在读取时从 6 个区域明细求和，不在 Flash 中重复保存。历史估算总水量汇总各区域当时已经保存的估算水量；按当前流量系数重算的水量只能作为明确标注的参考值。总脉冲和总水量使用 64 位整数汇总，避免相加溢出。

### 事件

事件用于记录重要动作、报警和异常，使用 `Esp32Base` 的 App Events 能力。本项目只定义事件内容，不重新实现事件保存、分页、筛选、导出、清除和容量管理。

业务事件包括：

- 自动计划因设备忙或请求失败而跳过。
- 自动浇水暂停和恢复。
- 无流量、低流量、高流量和非计划流量报警。
- 配置变化。
- 校准或流量学习保存、失败。
- 浇水记录保存失败。

设备启动、复位原因、WiFi、时间同步、OTA 和文件系统等技术运行信息由 `Esp32Base` 系统日志负责。只有这些系统行为直接影响浇水业务时，本项目才记录对应的业务影响事件，不复制系统日志。

## Web 信息结构

业务页面保持为：

1. 总览
2. 运行
3. 计划
4. 区域
5. 浇水记录
6. 浇水设置

流量计校准从浇水设置进入，区域流量学习从区域详情进入，不单独占用顶层导航。

浇水设置包含流量计、流量保护、自吸泵、全局阀驱动和系统参数。系统参数包含 RTC 时间倒退阈值和存活检查点间隔；阀驱动参数包括全功率吸合时间、PWM 频率和维持占空比。配置修改只影响之后启动的出水过程。

浇水记录页面只管理浇水记录。事件查看和管理使用 `Esp32Base` 提供的页面入口。

总览显示自动浇水总控状态并提供暂停或恢复入口；计划页面同时显示该状态。计划的“自动执行”开关只控制定时调度，手动运行入口可以选择任何已经配置的计划。

总览同时显示时间来源、时间无效、RTC 时间倒退和存储故障等自动调度不可用原因。RTC 时间倒退可以由 NTP 校时解除，这与用户主动设置的“暂停自动浇水”是两个独立状态。

固件配置提供默认 Web 账号密码。用户登录后使用 `Esp32Base` 的 Web Auth 页面修改，基础库保存的新认证优先于固件默认值；本项目不实现额外的密码生成、保存或首次认证流程。

本地 LCD2004 和按钮操作在 Web 功能稳定后单独设计；已经确认的后续设计输入见 `01-产品与硬件边界.md`。

OTA、重启和文件系统格式化属于用户主动维护操作。系统不自动判断是否适合执行，用户应先确认设备空闲；格式化前还应导出需要长期保留的浇水记录。
