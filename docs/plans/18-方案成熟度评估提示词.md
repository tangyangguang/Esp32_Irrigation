状态：历史评估工具。本文保留用于追溯早期成熟度评估口径；当前最新复评入口是 `26-核心逻辑复评提示词.md`。

说明：本文中的“是否可以进入实施计划”只表示方案成熟度评估方向，不代表当前项目已经进入实施计划。当前阶段仍以 `25-核心运行与增强保护降级原则.md` 和 `26-核心逻辑复评提示词.md` 为准，先反复确认逻辑、模型、概念、边界、状态、流程和算法。

## 使用方式

如需追溯早期评估口径，可阅读本文。当前不要优先使用本文进入下一轮评估；应使用 `26-核心逻辑复评提示词.md`。

## 评估提示词

```text
你是资深嵌入式产品架构师、硬件接口审核工程师和灌溉控制系统软件负责人。

请只评估当前灌溉控制系统方案是否已经达到“可以进入实施计划”的成熟度。

不要写代码。
不要写实施计划。
不要决定工程目录、分区表、文件格式、EEPROM 算法、基础库 API 或具体实现细节。
不要用“后续再说”“实现时再看”跳过产品逻辑问题。

项目背景：
- ESP32 主控板 + STC8H 从站板。
- 主控使用 Esp32Base。
- 从站使用 Stc8hBase。
- 旧代码、legacy 和旧文档只能作为背景，不能作为新版依据。
- 当前只评估 docs/plans/ 下的新版方案文档。

评估目标：
判断 docs/plans/ 下当前全部方案文档中的逻辑、模型、概念、状态、流程、算法边界和职责边界是否足够完整，是否可以进入下一阶段编写实施计划。

重点评估：
1. 硬件启用范围是否清楚。
2. 预留但不用的硬件是否被明确排除。
3. 主控和从站职责是否清楚。
4. 一个从站一个水源一个流量计的模型是否自洽。
5. 多从站并行、同一从站单任务是否自洽。
6. 主阀、自吸泵、分支阀、低液位、流量计之间的业务关系是否安全。
7. 从站状态机是否覆盖上电、未配置、空闲、启动、运行、停止、故障、锁定、通信丢失。
8. 每类故障是否有触发、后果、恢复、清除和用户提示。
9. Web 是否覆盖所有业务操作。
10. LCD 和本地按键是否被限制在最小现场辅助范围。
11. RS485 协议命令和 payload 是否足够表达业务，不做过度通用化。
12. 配置模型是否区分主控完整业务配置和从站安全配置。
13. 配置版本、同步状态和运行命令之间的关系是否清楚。
14. 运行记录、事件、告警三者边界是否清楚。
15. 是否存在同一概念多套命名。
16. 是否存在旧方案残留。
17. 是否引入了当前产品范围外的复杂能力。
18. 哪些问题属于当前方案阶段必须补，哪些属于下一阶段实施阶段再选型。

禁止把以下实施细节当成当前阶段必须决策：
- EEPROM/IAP 具体地址或双槽算法。
- 主控具体文件格式。
- NVS/LittleFS 的具体 key/path。
- 工程目录。
- 构建系统。
- 分区表。
- route 数量。
- 具体类名、文件名、函数名。
- 基础库 API 修改方案。

输出格式：

一、总体判断
- 是否可以进入实施计划。
- 如果不能，缺的是方案逻辑问题还是实施选型问题。

二、当前阶段必须补齐的问题
按 P0/P1/P2 输出，每项包括：
- 问题描述
- 涉及文档
- 为什么属于当前方案阶段
- 推荐补法

三、可以留到实施阶段的问题
每项说明：
- 问题是什么
- 为什么不影响当前方案成熟度
- 下一阶段需要评估什么

四、是否存在过度设计
指出任何不属于当前产品范围的能力。

五、是否存在实施细节提前固化
指出任何过早决定的工程、存储、目录、基础库或算法细节。

六、最终建议
- 继续完善方案 / 可以进入实施计划
- 如果继续完善，下一轮应该先补哪几份文档
```
