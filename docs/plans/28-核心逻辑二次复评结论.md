## 一、总体判断

当前方案已经把“核心运行最小可用”和“增强保护可降级”区分清楚。

本轮复评后的结论：

- 标准流速未学习：允许手动和计划运行，只是不启用低/高流量保护。
- 流量计计量未校准：允许普通运行，仍可记录脉冲和脉冲率；水量估算不可用或仅供参考。
- 电压检测未校准：允许普通运行，只是不启用电压告警和电压停机保护。
- 已配置并触发的低液位、无水、高流量、电压严重异常、通信超时、运行超时仍属于安全阻断或停机条件。
- 多从站可以并行运行；同一从站内部同一时间只运行一路。
- 不同计划同一从站冲突时，不排队、不隐藏延后，后触发计划项跳过并记录。

当前仍不进入实施计划。

原因：

- 用户已明确要求当前阶段继续做方案逻辑复评，不写实施计划。
- 当前结论只表示核心逻辑口径已经收敛，不等于实施层细节已经开始设计。
- 下一阶段是否进入实施计划，应由用户在复评完成后明确确认。

## 二、当前阶段必须修正的问题

本轮复评发现 2 个残留歧义，已在本轮修正。

### P1：计划 API 中“未校准目标跳过”的表述过宽

问题描述：

- `07-Web API与数据结构.md` 中曾写“被锁定、禁用、未校准、离线或配置未同步的目标应跳过”。
- “未校准”可能被误读为标准流速未学习、流量计计量未校准、电压检测未校准都会导致计划跳过。

为什么属于方案层问题：

- 这会破坏 `25` 已确认的原则：增强能力未就绪不得阻断核心运行。
- 这是 Web/API 的业务语义，不是实现细节。

修正方式：

- 改为“被锁定、禁用、离线、配置未同步或已触发明确安全阻断条件的目标应跳过”。
- 明确标准流速未学习、流量计计量未校准、电压检测未校准不应作为计划跳过原因。

当前判断：

- 已修正。

### P1：同站已有非本计划运行时不应再保留“等待”口径

问题描述：

- `19-运行调度与多从站并行模型.md` 中曾写“同从站已有非本计划运行：跳过或等待，按计划冲突规则定义”。
- 用户已确认不同计划同一从站冲突默认跳过并记录，不做排队。

为什么属于方案层问题：

- 同站冲突策略直接影响用户对计划结果的理解。
- “等待”会形成隐藏排队，和当前成熟产品口径不一致。

修正方式：

- 改为“跳过并记录 `station_busy`，不隐藏排队”。

当前判断：

- 已修正。

## 五、多计划同站冲突评估

当前默认策略已经确定：

- 同一计划内部，同一从站多个水路按顺序执行。
- 不同从站之间可以并行执行。
- 不同计划之间，同一从站冲突时不排队。
- 后触发计划项跳过并记录 `station_busy`。
- 保存计划时发现明显重叠，应提示“同一从站不能同时运行，实际触发时后一个计划项可能跳过”。
- 用户希望顺序执行时，应把水路合并到同一个计划。

评估结论：

- 该策略成熟、可解释、不过度设计。
- 该策略避免隐藏延后执行导致用户误判实际浇水时间。
- 该策略不需要复杂优先级、跨计划队列或补偿执行模型。
