## 一、总体判断

本评估结论已被 `25-核心运行与增强保护降级原则.md` 修正，不能作为进入实施计划的最终依据。

当前方案层还需要按“核心运行不被增强能力阻断”的原则重新复评，暂不进入实施计划编写阶段。

注意：

- 不写实施计划。
- 不开始正式业务代码。
- 先继续完善和复评方案层逻辑。

当前方案层已覆盖：

- 硬件启用范围。
- 主控/从站职责边界。
- 一个从站一个水源一个流量计。
- 多从站并行、同一从站单任务。
- 主阀、自吸泵、低液位、流量计、分支阀联动。
- 从站状态机。
- RS485 协议和 payload。
- 配置模型和同步语义。
- 运行调度模型。
- 流量保护算法。
- 自动计划冲突规则。
- Web 状态流和允许操作。
- 告警、故障、事件、运行记录生命周期。
- LCD 和本地按键最小边界。
- 不支持能力和过度设计边界。

## 二、当前阶段必须补齐的问题

本次评估原先未发现阻塞实施计划编写的 P0/P1 方案层缺口；但用户复核后发现“增强保护未就绪不应阻断核心运行”的原则未被充分体现。因此该结论需要修正。

评估过程中发现并已修正的问题：

### P0：核心运行与增强保护边界需要修正

问题描述：

- 标准流速、电压检测校准、流量计计量校准属于增强能力。
- 原结论中把电压检测未校准设为自动计划阻断条件，偏离“增强能力不阻断核心运行”的原则。

修正结果：

- 已新增 `25-核心运行与增强保护降级原则.md`。
- 已把标准流速未学习、电压检测未校准、流量计计量未校准统一改为“允许运行，但提示对应增强保护或计量能力未启用”。

当前判断：

- 需要按 25 号原则重新做完整方案复评。

### 已修正：从站状态机中残留具体实现名

问题描述：

- `03` 中状态灯说明残留具体实现名，属于实施阶段细节。

修正结果：

- 已改为“具体闪烁频率属于实施阶段设计”。

当前判断：

- 已收敛。

## 三、可以留到实施阶段的问题

以下问题不影响当前方案成熟度，但实施计划阶段必须逐项展开。

### 存储介质和断电保护

问题：

- 主控配置、运行记录、事件记录、从站安全配置最终如何保存，还没有决定具体介质、格式和断电保护算法。

为什么不影响当前方案成熟度：

- 当前方案已定义业务语义、数据生命周期、损坏后的业务后果和同步关系。
- 具体 NVS、文件、EEPROM/IAP、双槽或其他策略属于实现选型。

实施阶段需要评估：

- 主控侧配置和记录保存方式。
- 从站侧安全配置持久化方式。
- 配置版本、CRC、写入失败和断电恢复策略。
- 写入频率和存储寿命。

### 基础库缺口

问题：

- `Esp32Base` 和 `Stc8hBase` 是否已经具备全部需要的 Web、配置、文件、事件、RS485、CRC、ADC、PWM、持久化等能力，需要实施阶段逐项核对。

为什么不影响当前方案成熟度：

- 当前方案只定义应复用的能力类别和业务边界。
- 是否补基础库 API 属于实施阶段工程设计。

实施阶段需要评估：

- 哪些能力可以直接复用。
- 哪些能力属于基础库通用能力缺口。
- 哪些能力必须留在本业务项目中。

### RS485 实测参数

问题：

- `request_timeout_ms`、轮询周期、离线判定、疑似地址冲突识别，在真实线缆和多从站条件下需要实测。

为什么不影响当前方案成熟度：

- 协议语义、命令、payload、重试边界和安全后果已经定义。

实施阶段需要评估：

- 最大从站数量下状态刷新周期。
- 运行中从站轮询优先级。
- 总线异常、同地址冲突、A/B 反接、噪声场景。

### 电压检测实测

问题：

- 分压比例、ADC 参考、换算误差、阀门和泵动作时的瞬态电压还需要实测。

为什么不影响当前方案成熟度：

- 当前方案已定义电压校准状态、未校准降级策略和告警语义。电压检测未校准不阻断核心运行。

实施阶段需要评估：

- 电压换算公式。
- 采样滤波。
- 低压告警、低压停机、高压停机阈值。
- 校准失败时 Web 提示和保护降级策略。

### 阀门 PWM 和流量计参数

问题：

- 阀门吸合时间、保持比例、PWM 频率、流量计最低脉冲频率、标准流速采样窗口需要实测。

为什么不影响当前方案成熟度：

- 当前方案已定义默认降级策略：阀门可 100% 保持、流量计计量未校准时水量估算不可用或仅供参考、未学习标准流速时不启用低/高流量保护。

实施阶段需要评估：

- 阀门可靠吸合和温升。
- 流量计抖动、最低流量、重复校准误差。
- 现场管路波动对低/高流量误判的影响。

### Web UI 具体设计

问题：

- 页面布局、字段分组、移动端适配、文案风格还没有设计。

为什么不影响当前方案成熟度：

- 当前方案已定义 Web 必须覆盖的业务操作、状态流和允许/禁止动作。

实施阶段需要评估：

- 信息架构。
- 页面流程。
- 用户确认弹窗。
- 状态提示文案。
- 移动端可用性。

## 四、是否存在过度设计

本次评估未发现仍被纳入当前产品范围的明显过度设计。

已明确排除：

- 多主控。
- 从站主动上报总线架构。
- Web 任意透传 RS485 帧。
- 通用寄存器映射 UI。
- 一个从站多个流量计。
- 一个从站内部多个水路并行。
- 云平台计划。
- App 推送。
- 脚本规则引擎。
- 天气自动跳过。
- 土壤湿度自动决策。
- 从站本地执行自动计划。
- 主控板预留的独立小设备输入输出功能。

说明：

- 文档中出现这些能力时，均处于“不支持”或“当前产品不引入”的上下文，不是当前方案范围。

## 五、是否存在实施细节提前固化

本次评估未发现阻塞性提前固化。

已经避免固化：

- 不决定工程目录。
- 不决定具体文件格式。
- 不决定 NVS/LittleFS key 或路径。
- 不决定 EEPROM/IAP 地址。
- 不决定双槽、日志式或镜像式算法。
- 不决定分区表。
- 不决定 route 数量。
- 不决定具体类名、文件名、函数名。
- 不决定基础库 API 修改方案。

需要注意：

- `16` 和 `08` 中保留了参数范围和验证范围，这是产品方案层需要的边界，不是具体实现计划。
- 实施计划阶段可以调整参数默认值，但不能改变已经定义的安全语义，除非重新回到方案层评审。

## 六、最终建议

当前不进入实施计划。

下一步应使用 `26-核心逻辑复评提示词.md`，并按以下重点重新评估：

1. 哪些能力是核心运行前提，哪些只是增强保护。
2. 增强保护未配置、未学习、未校准时，是否都降级为提示而不是阻断。
3. 已启用增强保护且检测到明确危险时，是否能及时阻断或停机。
4. 多计划同一从站冲突策略是否足够成熟、可解释且不过度设计。
5. Web 是否把“保护未启用”和“设备故障/禁止运行”明确区分。

复评通过后，才能考虑是否进入实施计划。
