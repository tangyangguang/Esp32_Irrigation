## 一、总体判断

当前方案已经基本满足“核心运行最小可用，增强保护可降级”的原则。

本轮复评结论：

- 标准流速未学习时，允许手动和计划运行，只是不启用低/高流量保护。
- 流量计计量未校准时，允许运行，但水量估算不可用或仅供参考。
- 电压检测未校准时，允许运行，只是不启用电压告警和电压停机保护。
- 增强保护已经启用且检测到明确危险时，仍可阻断运行或本地停机。
- Web 状态矩阵已经区分“保护未启用”和“禁止运行”。
- 多计划同一从站冲突采用“保存时预警、触发时跳过并记录”的可解释策略，不做隐藏排队。

当前仍不进入实施计划。

原因：

- 用户已明确要求当前阶段继续完善和反复验证方案。
- 本轮复评刚修正了核心/增强原则相关旧口径，需要再由用户或独立会话复核一次。
- 多计划同一从站冲突策略已由用户确认：不排队、不隐藏，直接跳过并记录。

## 二、当前阶段必须修正的问题

本轮复评发现 1 个 P1 问题，已在本轮修正。

### P1：流量计计量未校准时，状态数据不能假装有准确水量

问题描述：

- 方案已经允许流量计计量未校准时继续运行。
- 但 `flow_rate_ml_min`、`total_ml`、`flow_ml` 等字段如果继续以普通数值显示，用户可能误以为水量和流速准确。

涉及文档：

- `07-Web API与数据结构.md`
- `13-RS485详细Payload与主控重试规则.md`
- `20-流量计校准与流量保护算法.md`
- `23-告警事件与运行记录生命周期.md`

为什么属于方案层问题：

- 这是业务语义问题，不是实现细节。
- 如果不定义清楚，Web 和记录会把增强计量能力误表达为核心能力。

修正方式：

- `20` 增加 `pulse_rate_per_min`，明确计量未校准时仍可显示脉冲率。
- `13` 增加 `flow_pulse_rate_min` 和 `flow_value_flags`，区分脉冲有效、计量已校准、ml/min 有效、累计水量可估算。
- `07` 增加 `flow_estimation_quality`，区分 `valid`、`reference_only`、`unavailable`。
- `23` 增加 `disabled_protections` 和 `estimation_quality` 记录要求。

当前判断：

- 已修正，不再阻塞下一轮复评。

|---|---|---|
| 低液位已触发 | 禁止启动或运行中停机 | 通过 |

| 高流量 | 已学习标准流速后默认停机并锁定 | 通过 |

| 运行超时 | 从站本地安全停机 | 通过 |

| 配置无效 | 禁止启动，要求重新同步或维护处理 | 通过 |

说明：

- “未校准”“未学习”“未启用增强能力”不属于危险条件。
- “已启用并确认异常”才触发阻断或停机。

## 五、多计划同站冲突评估

当前默认策略：

- 同一计划内，同一从站的多个水路按顺序执行。
- 不同从站可以并行执行。
- 不同计划之间，同一从站不做隐藏排队。
- 保存计划时，Web 预检查同一从站明显时间重叠并提示风险。
- 实际触发时，如果同一从站已有活动运行，后触发计划项跳过并记录 `station_busy`。
- 用户希望顺序执行时，应把这些水路合并到同一个计划。

评估：

- 该策略比“跨计划自动排长队”更可解释。
- 该策略比“静默跳过”更成熟，因为保存时有预警，触发时有记录。
- 不引入复杂调度规则，不属于过度设计。

用户已确认：

- 接受“不同计划同站冲突默认跳过并记录”作为产品默认行为。
- 不采用同站冲突延后执行或隐藏排队。

- 脚本规则引擎。
- 天气自动跳过。
- 土壤湿度自动决策。

本轮复评未发现新的提前实现细节固化。

仍不决定：

- 具体文件格式。
