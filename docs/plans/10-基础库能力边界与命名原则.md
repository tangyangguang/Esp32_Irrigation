## 当前阶段原则

当前阶段需要利用基础库能力来约束方案边界，但不做实施层决策。

原则：

- 基础库已有的通用能力，业务方案不得重新发明一套概念。
- 基础库能力只负责通用技术能力；灌溉业务模型留在本项目。
- 当前阶段只判断“应该复用哪类基础能力”，不决定具体函数、类、文件和目录。
- 如果发现缺少通用能力，只记录为实施阶段评估项，不在当前方案文档中设计补丁。
- 不把基础库的技术名词直接变成用户可见业务概念。

## Esp32Base 能力边界

ESP32 主控优先复用 `Esp32Base` 的通用能力类别：

| 能力类别 | 在本项目中的用途 | 方案边界 |
|---|---|---|
| Web 壳层和认证 | 承载灌溉 Web 页面和操作入口 | Web 业务对象由本项目定义 |
| 配置保存能力 | 保存主控侧业务配置和系统参数 | 配置模型由 `04`、`15` 定义 |
| 文件和导出能力 | 承载备份、运行历史、导出数据 | 文件格式实施阶段再定 |
| 事件和日志能力 | 记录业务事件、系统诊断和维护信息 | 业务事件语义由本项目定义 |
| 网络和时间能力 | 支撑 Web、计划调度、时间显示 | 调度规则由本项目定义 |
| OTA 和维护能力 | 主控固件维护入口 | 不扩展为从站 RS485 在线升级 |
| 看门狗和健康状态 | 保证主控长期运行可诊断 | 不替代业务故障模型 |

不应放进 `Esp32Base` 的内容：

- 水源、水路、主阀、分支阀、自吸泵等灌溉对象。
- 从站安全配置语义。
- 流量计校准和标准流速学习规则。
- 灌溉计划调度规则。
- 低液位、无水、流量异常等业务故障策略。

## Stc8hBase 能力边界

STC 从站优先复用 `Stc8hBase` 的通用能力类别：

| 能力类别 | 在本项目中的用途 | 方案边界 |
|---|---|---|
| GPIO | 输入、输出、拨码、指示灯、继电器 | 引脚和业务角色由本项目映射 |
| UART/RS485 | 从站协议收发 | 协议命令和业务 payload 由本项目定义 |
| 定时器/soft timer | 状态机超时、去抖、运行计时 | 业务状态和阈值由本项目定义 |
| PWM | 阀门保持能力 | 是否降功率由实测和配置决定 |
| ADC | 系统电压采样 | 电压业务阈值由本项目定义 |
| CRC/缓冲 | 协议帧校验和收发可靠性 | 帧格式由 `02`、`13` 定义 |
| 看门狗 | 从站长期运行安全 | 不替代通信丢失安全停机策略 |
| 持久化基础能力 | 保存从站必要安全配置 | 具体介质和算法实施阶段再定 |

不应放进 `Stc8hBase` 的内容：

- 灌溉状态机。
- 主控调度逻辑。
- Web 配置语义。
- 水路计划和运行记录。
- 特定 PCB 的完整业务映射。

## 业务概念命名

方案层统一使用以下概念：

| 中文概念 | 英文标识建议 | 定义 |
|---|---|---|
| 主控 | `Controller` | ESP32 主控板和主控固件 |
| 从站 | `Station` | STC8H 分站板，一个从站对应一个真实水源 |
| 水源 | `WaterSource` | 一个从站管理的实际供水来源 |
| 水路 | `Zone` | 一个可独立运行的分支灌溉通道 |
| 主阀 | `MasterValve` | 可选，某从站所有分支水路前置总阀 |
| 分支阀 | `ZoneValve` | 对应一路水路的电磁阀 |
| 自吸泵 | `Pump` | 可选，通过干接点输出控制外部泵继电器 |
| 流量计 | `FlowMeter` | 必选，一个从站只支持一个 |
| 低液位 | `LowLevel` | 可选输入，用于水源液位保护 |
| 运行会话 | `RunSession` | 一次手动或计划触发的灌溉运行 |
| 运行命令 | `RunCommand` | 主控下发给从站的单次启动/停止请求 |
| 安全配置 | `SafetyConfig` | 从站本地安全运行所需的最小配置投影 |
| 配置同步状态 | `ConfigSyncState` | 主控判断从站配置是否可运行的状态 |
| 故障 | `Fault` | 会影响运行安全或禁止运行的问题 |
| 告警 | `Alarm` | 需要用户关注但不一定禁止运行的问题 |
| 事件 | `Event` | 可解释的低频业务过程记录 |

## 禁止混用的概念

- 不把“从站”和“水源”混成一个概念。从站是硬件节点，水源是业务对象；当前产品中二者一一对应。
- 不把“水路”和“阀输出”混成一个概念。阀输出是硬件资源，水路是用户配置后的业务资源。
- 不把“主阀”当成可独立运行水路。
- 不把“流量计输入”和“低液位输入”混成同一种传感器。
- 不把“故障清除”理解为“问题已修复”。清除只是解除可恢复状态；真实条件仍异常时必须再次触发。
- 不把“主控通信失败”理解为“从站已停机”。从站是否停机只能通过后续状态或本地安全策略推断。

## 枚举和状态命名原则

- 状态名表达“当前处于什么状态”，例如 `IDLE`、`RUNNING`、`FAULT`。
- 动作名表达“请求做什么”，例如 `START_ZONE`、`STOP`、`CLEAR_FAULT`。
- 原因名表达“为什么停机或拒绝”，例如 `NO_FLOW`、`LOW_LEVEL`、`CONFIG_MISMATCH`。
- 配置字段名带单位，例如 `_ms`、`_sec`、`_mv`、`_ml_min`。
- 用户可见文案用中文业务语言，内部标识用稳定英文短语。

## 当前阶段不决定

以下内容不在当前阶段决定：

- `Esp32Base` 或 `Stc8hBase` 的具体 API 修改。
- 工程目录、include 方式、构建系统。
- NVS key、文件路径、二进制结构、JSON 结构。
- EEPROM/IAP 地址、双槽、日志式或镜像式算法。
- 具体类名、函数名、任务名和文件名。

这些内容进入实施阶段后，根据已经确认的业务模型再设计。
