## 总体架构

系统由一个主控和一个或多个从站组成。

- 主控运行在 ESP-WROOM-32E 上。
- 从站运行在 STC8H8K64U-45I-LQFP48 上。
- 主控固件基于 `Esp32Base` 基础库开发。
- 从站固件基于 `/Users/tyg/dir/codex_dir/Stc8hBase` 基础库开发。
- 主控通过 RS485 轮询和控制从站。
- 一个从站代表一个真实水源。
- 一个从站内部同一时间只运行一路分支水路。
- 多个从站代表多个独立水源，主控需要支持多个从站同时运行。
- Web 是完整操作入口。
- LCD 和本地按键只做最小现场辅助。

## 固件基础库边界

ESP32 主控侧：

- 使用 `Esp32Base` 作为应用基础库。
- 以 `Esp32Base` 的 Web、API、配置、文件、事件记录、网络、OTA、系统诊断和运行时能力作为优先复用对象。
- Web 页面样式、基础路由、认证、静态资源、文件系统维护和系统页能力应优先沿用 `Esp32Base` 已有规则。
- 业务代码只实现灌溉领域对象、页面、API、调度器、RS485 主站和传感器/外设适配。
- 不在本项目中另起一套 Web 框架、配置框架、事件日志框架或设备生命周期框架。
- 底层 Arduino ESP32 Core 的直接使用应被限制在 `Esp32Base` 没有覆盖、且确有硬件适配必要的地方。

STC 从站侧：

- 使用 `/Users/tyg/dir/codex_dir/Stc8hBase` 作为从站基础库。
- 优先复用其中的 GPIO、UART、定时器、PWM、ADC、持久化、看门狗、外部中断、CRC、ring buffer、soft timer 和滤波等通用模块。
- 应用侧只实现本项目专用的板级引脚、RS485 从站协议、输入角色映射、运行状态机、阀门/泵联动和故障策略。
- 不把产品业务状态机、Web 配置语义或主控调度逻辑放入 `Stc8hBase`。
- 从站工程应保持可裁剪，未使用模块不应占用 ROM、RAM、外设或中断资源。

基础库接入原则：

- 具体实现前先阅读两个基础库的 README、AGENTS 和相关 docs，以它们的当前文档作为接入依据。
- 每个模块设计前必须先按 `10-基础库能力边界与命名原则.md` 做基础库能力映射。
- 基础库已有或相近能力必须优先复用，不在本项目重复设计同类基础能力。
- 如果业务需要的能力属于基础库通用能力，优先评估是否补充基础库；如果属于本项目业务能力，则留在灌溉项目内。
- 修改基础库属于独立高影响范围，必须单独说明影响并确认后再做。
- 业务代码命名、状态命名、事件 token 和配置字段命名遵守 `10-基础库能力边界与命名原则.md`。

## 主控固件职责

主控固件负责：

- Web 页面和 API。
- Wi-Fi/网络连接。
- RTC 时间管理。
- SHT30 温湿度读取。
- LCD2004 最小状态显示。
- 主控按键最小辅助操作。
- 主控配置存储。
- 从站业务配置管理。
- 从站安全配置生成和同步。
- RS485 主站通讯。
- 多从站调度。
- 手动运行。
- 自动计划。
- 流量计校准流程。
- 标准流速学习流程。
- 事件记录和告警管理。

主控固件不负责：

- 直接操作从站阀门 GPIO。
- 在从站通信丢失后假定输出仍可控。
- 用 LCD 或本地按键完成完整配置。
- 同一从站内并行运行多个分支水路。

## 从站固件职责

从站固件负责：

- 读取拨码地址。
- 保存必要安全配置。
- 校验配置版本和 CRC。
- RS485 从站应答。
- 输入角色映射。
- 流量计脉冲计数。
- 低液位检测。
- 系统电压检测。
- 自吸泵干接点输出。
- 主阀输出。
- 分支阀输出。
- 阀门 PWM 吸合和保持。
- 本地运行状态机。
- 通信超时安全停机。
- 运行超时安全停机。
- 故障状态和状态灯。
- 本地维护/识别按键。

从站固件不负责：

- 自动计划。
- Web 配置。
- 多水源管理。
- 一个从站多个流量计。
- 从站主动抢总线上报。
- 本地按键启动灌溉。

## 主控运行模型

主控运行模型由多个周期任务组成。

核心循环：

1. 处理 Web/API 请求。
2. 周期读取 RTC 和 SHT30。
3. 周期轮询从站状态。
4. 根据计划创建运行任务。
5. 维护每个运行中的从站任务。
6. 处理故障、锁定和事件记录。
7. 更新 LCD 最小状态。

主控需要支持多个 `StationRunSession` 同时存在。

约束：

- 同一个从站最多一个活动 `StationRunSession`。
- 不同从站可以同时有活动 `StationRunSession`。
- 主控对每个运行中的从站都需要持续轮询。
- 某个从站通信失败或故障，不应阻塞其他从站轮询和运行。

## 从站运行模型

从站运行模型以主循环和定时中断组成。

主循环职责：

- 处理 RS485 请求。
- 更新运行状态机。
- 检查通信超时。
- 检查运行超时。
- 读取低液位输入。
- 读取电压。
- 更新状态灯。
- 喂狗。

中断职责：

- 流量计脉冲计数。
- PWM 或定时器节拍。
- 必要的系统 tick。

约束：

- 中断内只做短操作，避免复杂逻辑。
- 输出安全关闭逻辑必须能在主循环及时执行。
- RS485 收包和发包不能阻塞流量计计数。
- 从站任何复位路径都必须默认关闭输出。

## RS485 调度模型

主控作为唯一主站。

轮询优先级：

1. 正在运行的从站。
2. 有未处理故障的从站。
3. 在线空闲从站。
4. 离线重试从站。

运行中从站轮询频率高于空闲从站。

建议：

- 运行中从站：`500 ms..1 s` 查询一次。
- 空闲从站：`1..3 s` 查询一次。
- 离线从站：退避重试。

主控发送命令时：

- 同一时间总线上只允许一个未完成请求。
- 请求超时后按策略重试。
- 重试失败后标记从站通信异常。
- 运行中的从站通信异常时，从站本地也会按 `comm_timeout_ms` 安全停机。

## 多从站并行运行

多从站并行是产品能力。

主控调度器需要做到：

- 对不同从站可以同时创建运行任务。
- 每个从站任务独立记录 `run_id`。
- 每个从站独立维护运行时间、流量、状态和故障。
- 某个从站停止或故障，不自动停止其他从站。
- Web 首页和运行页可以同时展示多个运行中的从站。

同一从站内：

- 只允许一路分支水路运行。
- 同一个计划中针对同一从站的多个水路按顺序执行。
- 主阀和自吸泵属于该从站本地联动资源。

## 配置同步流程

配置同步由主控发起。

流程：

1. Web 修改从站安全相关配置。
2. 主控生成新的安全配置和版本号。
3. 主控保存业务配置。
4. 主控通过 `SET_SAFE_CONFIG` 下发到从站。
5. 从站校验并保存配置。
6. 主控读取或接收配置版本确认。
7. 配置同步成功后，从站可运行。

同步失败时：

- Web 显示从站配置未同步。
- 主控不允许对该从站启动运行。
- 已运行的从站不应被新配置覆盖；修改安全配置前应要求该从站空闲。

## 数据职责

主控侧负责管理：

- 保存主控系统配置。
- 保存从站业务配置。
- 保存水路配置。
- 保存计划。
- 保存流量计校准参数。
- 保存标准流速。
- 保存事件记录、故障记录和运行记录。

从站侧负责保留：

- 能判断自身安全运行所需的必要配置。
- 能报告当前配置版本和校验状态。
- 能报告最近故障和停止原因。

FRAM：

- 主控板有 FRAM 硬件预留。
- 本产品当前不启用 FRAM。
- 当前方案不依赖 FRAM 完成业务闭环。

## 安全策略

主控安全策略：

- 不对离线从站启动运行。
- 不对配置未同步从站启动运行。
- 不对锁定水路启动运行。
- 不覆盖从站本地安全停机原因。
- 不因一个从站故障自动停止其他从站，除非明确配置全局联动策略。

从站安全策略：

- 上电默认全部输出关闭。
- 配置无效不允许运行。
- 通信超时本地安全停机。
- 运行超时本地安全停机。
- 低液位触发本地停机。
- 已启用电压保护并确认电压严重异常时本地停机。
- 本地按键不能启动灌溉。

## Web/API 与固件关系

Web/API 只操作主控的业务对象。

例如：

- 用户点击启动水路，Web 调用主控 API。
- 主控 API 创建运行任务。
- 运行任务通过 RS485 下发 `START_ZONE`。
- 从站接受后执行本地状态机。
- 主控轮询状态并推送给 Web。

Web 不直接操作 RS485 帧。
