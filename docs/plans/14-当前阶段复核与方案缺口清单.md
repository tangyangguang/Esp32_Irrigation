## 总体结论

当前方案方向成立，但还不能进入实施计划，也不能开始写正式业务代码。

已经成立的方向：

- 主控板只承担主控、人机交互、配置、调度、记录和 RS485 管理。
- 主控板预留的 ADC、主控侧传感器输入、主控侧阀输出、干节点继电器、FRAM 等独立小设备能力在本产品中不启用。
- 从站板承担 485 通信、8 路阀输出、1 路流量计、可选低液位、可选自吸泵干接点输出、状态灯、本地按键、拨码地址和系统电压检测。
- 一个从站只对应一个真实水源，只支持一个流量计。
- 多个从站可以同时运行。
- 同一个从站内部同一时间只运行一路分支水路。
- 主阀可选，自吸泵可选，低液位可选。
- Web 是完整操作入口。
- LCD 和本地按键只承担最小现场辅助，不承担复杂配置。
- RS485 采用主控轮询，从站不主动上报。
- Web 不透传任意 RS485 帧。

经过用户复核，`24-方案成熟度评估结论.md` 中关于增强保护阻断运行的结论需要修正。当前已新增 `25-核心运行与增强保护降级原则.md`，方案层需要按该原则重新复评，暂不进入实施计划编写。

## 已收敛的方案问题

### 硬件启用范围

已通过 `01` 收敛：

- 明确主控板本产品只用供电、ESP32、按钮、两个指示灯、RS485、LCD2004 I2C、外接 SHT30。
- 明确从站板本产品使用 RS485、8 路阀、4 路输入中的流量计和可选低液位、干接点输出中的可选自吸泵、状态灯、按键、拨码、电压检测。
- 明确预留但当前不用的硬件不参与软件模型。

### 主控/从站职责

已通过 `03`、`04`、`06` 收敛：

- 主控拥有完整业务配置、计划、Web、调度和记录。
- 从站只保留本地安全运行所需的信息。
- 从站在主控通信异常时必须能本地安全停机。

### 基本通信模型

已通过 `02`、`13` 收敛：

- 主控单主站轮询。
- 从站只应答请求，不主动发起总线通信。
- 多从站并行运行通过主控调度和轮询优先级实现。
- 同地址冲突只能标记为疑似地址冲突或总线异常，不能承诺自动准确识别具体设备。

### 枚举和错误码

已通过 `11` 收敛：

- 从站状态、停止原因、协议结果、故障标志、Web 错误码使用同一套语义。
- 后续新增状态或故障必须先回到 `11` 统一命名。

## 当前阶段复核结果与剩余问题

### 已收敛：运行调度和多从站并行模型

已通过 `19-运行调度与多从站并行模型.md` 收敛：

- 主控不是全局单任务系统。
- 多个从站可以同时运行。
- 同一个从站内部严格单任务。
- `RunSession` 明确区分手动、计划、校准、学习和维护运行。
- 同站冲突、停止优先级、通信丢失和运行结束结果已有统一解释。

剩余风险：

- 运行状态必须在 Web 页面中表达清楚，避免“已发停止命令”和“已确认停止”混淆。

### 已收敛：计划调度与运行冲突规则

已通过 `21-计划调度与运行冲突规则.md` 收敛：

- 自动计划由主控执行，从站不保存计划。
- 同一计划内同站顺序、跨站并行。
- 自动计划不抢占手动运行。
- 不同计划同站冲突默认跳过，不跨计划排长队。
- 错过执行默认不补跑。
- 计划修改只影响未来触发，运行中使用启动快照。
- 计划记录必须区分完成、部分完成、失败、跳过和取消。

剩余风险：

- Web 页面需要把这些状态和允许动作明确展示，避免用户误解。

### P1：计划实测和时间源边界仍需保留

问题：

- 计划业务规则已经收敛，但主控时间可靠性、断电恢复后的时间状态和 RTC 校时流程仍需要在实施阶段验证。

风险：

- 如果时间源不可靠，自动计划可能错过或误触发。

推荐补法：

- 当前阶段保留“时间无效则不执行自动计划”的业务策略。
- 实施阶段再验证 RTC、网络校时和断电恢复行为。

### 已收敛：流量计校准与流量保护算法

已通过 `20-流量计校准与流量保护算法.md` 收敛：

- 一个从站一个流量计。
- 流量计校准生成从站级 `pulses_per_liter`。
- 标准流速学习生成水路级 `baseline_ml_min`。
- 无水保护不依赖标准流速。
- 低/高流量保护依赖标准流速。
- 待机漏水是空闲状态下的水源级告警。
- 从站负责本地快速保护，主控负责校准、学习、告警解释和用户确认。

剩余风险：

- Web 页面需要明确显示哪些流量保护已启用、哪些因未学习标准流速而未启用。

### P1：流量参数实测边界仍需保留

问题：

- 业务算法已经收敛，但具体流量计脉冲特性、最低可测流量、脉冲抖动和现场管路波动必须实机验证。

风险：

- 如果参数不合理，会造成误停机或漏报。

推荐补法：

- 当前方案文档只保留业务规则和保守默认策略。
- 实施阶段再根据实物测试确认采样窗口、最低脉冲频率和参数范围。

### 已收敛：Web 页面状态流与允许操作矩阵

已通过 `22-Web页面状态流与允许操作矩阵.md` 收敛：

- 首页、从站、手动运行、配置、输入输出、流量、计划、故障和维护页面的允许/禁止动作已定义。
- 明确区分状态过期、通信丢失、已发停止命令、已确认停止。
- 明确配置已保存不等于从站配置已同步。
- 明确清除故障不等于问题已修复。

剩余风险：

- 后续 UI 设计必须遵守这些业务状态，不得为了页面简化而隐藏关键禁止原因。

### P1：Web 页面具体信息架构仍需在 UI 阶段细化

问题：

- Web 业务状态和允许动作已经收敛，但具体页面布局、字段分组、移动端交互和文案风格属于后续 UI 设计。

风险：

- 页面设计如果压缩过度，可能隐藏关键风险提示。

推荐补法：

- 当前阶段只保留业务状态矩阵。
- UI 阶段按矩阵设计页面，不改变业务语义。

### 已收敛：告警、事件与运行记录生命周期

已通过 `23-告警事件与运行记录生命周期.md` 收敛：

- 告警、故障、事件、运行记录和计划记录的边界已区分。
- 告警生命周期、故障生命周期、清除条件和记录要求已定义。
- 清除故障不删除历史记录，清除告警不等于问题已修复。
- 计划记录和运行记录的关系已明确。

剩余风险：

- 具体保存数量、保存天数、导出格式和存储介质属于实施阶段设计。

### P2：记录保留策略实施阶段需量化

问题：

- 当前阶段只定义了业务边界，尚未量化记录保留数量、保存天数和导出格式。

风险：

- 保存容量和查询性能需要实施阶段结合基础库能力评估。

推荐补法：

- 当前阶段不继续细化存储实现。
- 实施阶段再根据业务记录量和基础库能力定保存策略。

### 已收敛：本地按键维护功能

已通过 `17-故障清除与本地维护策略.md` 补齐：

- 短按只做识别和状态显示。
- 长按只用于清除满足条件的可恢复故障或退出 `OFFLINE_LOCAL_SAFE`。
- 上电按住只进入识别显示，不恢复出厂、不清配置。
- 运行中按键不启动、不停止、不切换水路。

剩余风险：

- 实施阶段需要确保按键去抖和长按确认不会误触发。

## 当前阶段明确不做的事

以下问题属于实施阶段，不应在当前文档中继续细化：

- 工程目录。
- 编译系统。
- 分区表。
- 具体文件格式。
- EEPROM/IAP 地址。
- 双槽、日志式、镜像式等存储算法。
- NVS key、文件路径、JSON 字段落盘格式。
- 具体类名、任务名、函数名、文件名。
- 基础库 API 修改方案。
- 烧录和接线步骤。

这些不是被忽略，而是必须等业务模型收敛后再设计。

## 下一步建议

当前不在本文中写实施计划。

`24-方案成熟度评估结论.md` 已被 `25-核心运行与增强保护降级原则.md` 修正。当前不进入实施计划，下一步应按最新原则重新评估全套方案。

- 标准流速未学习时，允许手动和计划运行，只是不启用低/高流量保护。
- 流量计计量未校准时，允许运行，但水量估算不可用或仅供参考。
- 电压检测未校准时，允许运行，只是不启用电压保护。
- 多计划同一从站冲突时，默认不跨计划隐藏排队，保存配置时应提示重叠风险，触发时冲突项跳过并记录。

这些策略还需要用新的复评提示词检查一轮。
