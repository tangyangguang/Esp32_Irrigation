## 当前阶段目标

当前阶段只解决产品方案层问题：

- 逻辑是否闭环。
- 模型是否稳定。
- 概念是否统一。
- 状态是否完整。
- 流程是否可解释。
- 边界是否清楚。
- 安全策略是否覆盖核心风险。
- Web 是否覆盖完整业务操作。
- 主控和从站职责是否不会混乱。
- 预留硬件是否明确排除。

当前阶段不解决：

- 工程目录。
- 构建系统。
- 分区表。
- 具体文件格式。
- EEPROM/IAP 具体地址。
- 基础库 API 修改。
- 代码模块拆分。
- 烧录命令。
- 实机接线步骤。

这些内容进入实施计划阶段再讨论。

## 允许讨论的内容

当前阶段允许讨论：

- 一个从站是否只能对应一个水源。
- 多从站是否可以同时运行。
- 同一从站是否只能运行一路水路。
- 流量计、低液位、自吸泵、主阀、分支阀之间的业务关系。
- 低液位触发后属于告警、故障还是锁定。
- 无水、高流量、低流量的业务后果。
- 配置同步失败时主控和从站分别处于什么状态。
- 通信丢失时从站如何保证安全。
- Web 上用户能看到什么、能操作什么、不能操作什么。
- LCD 和本地按键的最小职责。
- 哪些硬件预留在本产品中明确不用。

## 不在当前阶段决定的内容

当前阶段不决定：

- 从站安全配置最终用 EEPROM、Flash 还是其他方式保存。
- 是否采用双槽、日志式、镜像式或其他存储算法。
- 主控配置最终用 NVS、LittleFS、二进制文件还是 JSON 文件。
- `Esp32Base` 采用哪个 profile。
- `Stc8hBase` 或 `Esp32Base` 需要补哪些具体 API。
- Web API 最终注册多少个 route。
- 固件目录放在哪里。
- 每个模块的类名、文件名和函数名。

这些不是不重要，而是属于工程实现设计，不是当前方案层工作的输出。

## 进入下一阶段的条件

只有满足以下条件，才进入实施计划阶段：

- 当前全部方案文档的核心概念没有互相矛盾。
- 用户确认硬件启用范围和不用的预留功能。
- 用户确认主控/从站职责边界。
- 用户确认 Web 是完整操作入口，LCD 只做最小显示。
- 用户确认故障和锁定策略的业务含义。
- 用户确认配置模型和状态机能表达真实使用场景。
- 用户确认当前方案不包含过度设计项。
- 用户确认当前方案没有用“以后再说”逃避成熟产品必须定义的业务逻辑。

进入下一阶段后，再另写实施计划文档，编号届时按文档目录顺序确定。当前编号继续优先用于补齐方案层文档。
