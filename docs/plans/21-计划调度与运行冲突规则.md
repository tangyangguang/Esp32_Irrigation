# 21 计划调度与运行冲突规则

状态：成熟产品方案层文档。本文定义自动计划的业务模型、触发条件、同站顺序、跨站并行、计划冲突、跳过规则和记录要求。不定义调度器线程、定时器实现、数据文件格式或代码结构。

## 核心原则

- 自动计划由主控执行。
- 从站不保存自动计划。
- 自动计划不能绕过运行准入规则。
- 不同从站可以并行执行计划。
- 同一从站内计划水路按顺序执行。
- 自动计划不抢占手动运行。
- 自动计划不自动清除故障。
- 自动计划不自动修改配置。
- 自动计划不自动重启因故障停止的水路。

## 计划对象

主控用 `Schedule` 表示一个自动计划。

`Schedule` 至少包含：

```text
schedule_id
name
enabled
start_time
repeat_rule
station_items
created_by
updated_at
```

`station_items` 表示该计划涉及的从站和水路：

```text
station_id
zone_sequence
duration_sec
enabled
```

规则：

- 一个计划可以包含多个从站。
- 每个从站下面可以包含多个水路。
- 同一个从站下的多个水路按 `zone_sequence` 顺序运行。
- 不同从站的计划项可以并行运行。
- 一个计划不能把同一从站同一水路重复列入同一次执行。

## 重复规则

当前产品支持本地周期计划，不引入云端规则引擎。

推荐支持：

| 类型 | 含义 |
|---|---|
| `once` | 单次执行 |
| `daily` | 每天固定时间执行 |
| `weekly` | 每周指定星期固定时间执行 |
| `interval_days` | 每隔 N 天固定时间执行 |

不支持：

- 云端天气自动跳过。
- 脚本规则。
- 复杂条件表达式。
- 多主控协同计划。
- 从站主动发起计划。

说明：

- 如果以后需要天气或土壤湿度策略，应先作为独立方案评估，不混入当前第一版计划模型。

## 计划触发前置条件

主控执行计划前必须检查：

- 主控时间有效。
- 计划已启用。
- 当前触发时间匹配计划重复规则。
- 计划至少包含一个启用的从站水路项。
- 对应从站在主控配置中启用。

单个从站计划项启动前还必须检查：

- 从站在线。
- 无疑似地址冲突或总线异常。
- 从站配置已同步。
- 从站未故障、未锁定、未处于 `OFFLINE_LOCAL_SAFE`。
- 从站没有活动 `RunSession`。
- 目标水路启用。
- 目标水路允许计划运行。
- 目标水路未锁定。
- 流量计输入已配置。
- 当前无已确认电压故障；电压检测未校准不阻断计划运行。
- 请求时长在允许范围内。

如果水路未学习标准流速：

- 默认仍可计划运行。
- 但本次运行不启用低/高流量保护。
- 计划记录必须标记“标准流速未学习，保护不完整”。
- Web 不应提供“未学习标准流速就禁止计划运行”的默认策略；如果未来要增加这种严格模式，必须重新回到方案层评审。

## 计划执行模型

一次计划触发生成一个 `ScheduleRun`。

`ScheduleRun` 包含多个从站执行分支：

```text
schedule_run_id
schedule_id
triggered_at
state
station_results
```

同一 `ScheduleRun` 内：

- 每个从站最多一个当前活动水路。
- 一个从站完成当前水路后，再执行该从站队列中的下一水路。
- 不同从站的队列互不等待。
- 某个从站失败或跳过，不影响其他从站继续执行。

`ScheduleRun` 状态：

| 状态 | 含义 |
|---|---|
| `pending` | 已触发，尚未开始任何水路 |
| `running` | 至少一个从站正在执行或等待同计划下一水路 |
| `completed` | 所有可执行水路正常结束 |
| `partial` | 部分水路完成，部分跳过或失败 |
| `skipped` | 全部应执行水路都未启动即跳过 |
| `failed` | 至少尝试启动或运行，但没有任何水路成功完成 |
| `cancelled` | 启动前被用户取消 |

## 同站顺序规则

同一从站内的计划水路按配置顺序执行。

每一路水路结束后：

- 如果正常完成，继续下一路。
- 如果用户停止该水路，默认停止该从站在本次计划中的后续水路。
- 如果本地安全故障停机，停止该从站后续水路。
- 如果通信丢失进入 `lost_contact`，停止该从站后续水路。
- 如果该水路被跳过，继续检查下一路。

说明：

- “跳过”表示该水路未启动，不等同于运行失败。
- “失败”表示尝试启动或运行后异常结束。

## 跨站并行规则

不同从站代表不同真实水源。

默认规则：

- 不同从站计划项可同时启动。
- 某从站的水路顺序不影响其他从站。
- 某从站故障不停止其他从站。
- 某从站通信丢失不停止其他从站。

例子：

```text
计划 A：
  从站 1：Z1 10 分钟 -> Z2 10 分钟
  从站 2：Z1 15 分钟
```

执行：

- 从站 1 的 Z1 和从站 2 的 Z1 可以同时开始。
- 从站 1 的 Z2 必须等从站 1 的 Z1 结束后再开始。
- 从站 2 完成后不等待从站 1。

## 手动运行与计划冲突

手动运行优先保持用户当前动作。

规则：

| 场景 | 处理 |
|---|---|
| 计划触发时，同一从站正在手动运行 | 该从站计划项跳过，记录 `station_busy` |
| 计划触发时，其他从站正在手动运行 | 不影响本计划可运行从站 |
| 计划运行中，用户请求同一从站手动运行 | 拒绝，提示该从站正在计划运行 |
| 计划运行中，用户请求其他从站手动运行 | 允许，按普通手动准入检查 |
| 用户停止计划中的某一路 | 停止该运行；该从站本次计划后续水路默认取消 |
| 用户停止全部 | 所有活动计划和手动运行都发起停止 |

说明：

- 自动计划不抢占手动运行。
- 手动运行也不自动插队到同一从站计划队列。

## 多计划冲突

多个计划可能在相同或接近时间触发。

不同从站：

- 可以并行执行。

同一从站：

- 如果从站已有活动运行，后触发计划项默认跳过并记录 `station_busy`。
- 不同计划之间不默认排长队，避免用户难以理解实际执行时间。
- 用户如果需要顺序运行，应把这些水路放进同一个计划。

同一时间多个计划同时触发且都包含同一从站：

- 主控按计划排序规则选择一个先执行。
- 其他涉及该从站的计划项跳过并记录冲突。

计划排序规则：

1. 触发时间更早者优先。
2. 触发时间相同，`schedule_id` 更小者优先。

计划配置预检：

- 保存计划时，Web 应检查同一从站是否存在明显时间重叠的计划。
- 发现重叠时不强制禁止保存，但必须提示“同一从站不能同时运行，实际触发时后一个计划项可能跳过”。
- 如果用户希望这些水路顺序执行，应提示用户合并到同一个计划。
- 这种策略优先保证行为可解释，不引入跨计划隐藏排队。

## 错过执行

错过执行指计划触发时主控未能执行，例如主控断电、时间无效或系统刚启动。

默认策略：

- 不自动补跑已经错过的计划。
- 主控恢复后记录错过事件。
- 下一次符合重复规则的时间再执行。

原因：

- 灌溉和水泵控制有现场条件，补跑可能与用户预期不一致。
- 成熟产品应避免断电恢复后突然补跑放水。

当前产品不提供补跑配置；错过执行只记录事件，等待下一次符合重复规则的时间再执行。

## 计划修改与运行中快照

计划运行开始后，当前 `ScheduleRun` 使用启动时的计划快照。

规则：

- 修改计划只影响未来触发。
- 正在运行的水路不因计划修改自动改变时长。
- 尚未启动的同计划后续水路默认仍按启动快照执行。
- 如果用户希望立即停止或取消后续水路，应执行停止或取消动作。

禁用计划：

- 只阻止未来触发。
- 不自动停止已经运行中的 `ScheduleRun`。
- Web 必须提示“是否同时停止当前运行”。

删除计划：

- 不删除历史运行记录。
- 如果该计划当前正在运行，Web 必须先要求用户选择停止当前运行或保留运行到结束。

## 跳过原因

计划项未启动时，必须记录跳过原因。

常见跳过原因：

```text
time_invalid
schedule_disabled
station_disabled
station_offline
station_address_conflict
station_config_pending
station_fault
station_locked
station_busy
zone_disabled
zone_locked
zone_not_schedule_allowed
low_level_active
voltage_fault
validation_failed
```

规则：

- 跳过原因面向用户可解释。
- 跳过不清除故障。
- 跳过不自动重试同一次计划项。
- 跳过记录保留在计划运行记录中。

## 计划失败和部分完成

`ScheduleRun` 结果：

| 结果 | 条件 |
|---|---|
| `completed` | 所有应执行水路都正常完成 |
| `partial` | 至少一路完成，且至少一路跳过或失败 |
| `skipped` | 全部应执行水路都未启动即跳过 |
| `failed` | 至少尝试启动或运行，但没有任何水路成功完成 |
| `cancelled` | 用户在启动前取消 |

说明：

- “部分完成”是成熟产品必须明确展示的结果，不能简单显示成功。
- “全部跳过”不是运行失败，必须显示跳过原因。
- 某一路因故障停机时，该水路结果是失败；同计划其他从站可能仍完成。

## 用户停止计划

Web 应提供：

- 停止单个正在运行的水路。
- 停止某个从站本次计划剩余水路。
- 停止整个 `ScheduleRun`。
- 停止全部运行。

规则：

- 停止单个水路只影响对应从站当前运行。
- 停止某个从站本次计划剩余水路，不影响其他从站。
- 停止整个 `ScheduleRun` 会停止该计划触发产生的所有活动运行。
- 停止全部会停止所有活动运行，不区分来源。

## 记录要求

每次计划触发必须记录：

- 计划 ID 和名称。
- 触发时间。
- 主控时间是否有效。
- 每个从站每个水路的结果。
- 跳过原因。
- 失败原因。
- 用户停止动作。
- 是否发生通信丢失。
- 是否发生本地安全停机。
- 实际运行时长和估算用水量。

计划记录和运行记录关系：

- `ScheduleRun` 是计划层记录。
- `RunSession` 是单次水路运行记录。
- 一个 `ScheduleRun` 可以包含多个 `RunSession`。

## Web 展示要求

计划页面必须能显示：

- 计划是否启用。
- 下次预计执行时间。
- 最近一次执行结果。
- 最近一次部分完成或失败原因。
- 每个从站水路的顺序。
- 未校准或未学习标准流速的风险提示。
- 与当前手动运行或其他计划的冲突提示。

运行中页面必须能显示：

- 该运行来自哪个计划。
- 同一计划中还有哪些水路待执行。
- 哪些从站已完成、跳过、失败或仍在运行。

## 不支持的计划能力

当前产品不支持：

- 云平台计划。
- App 推送计划。
- 天气自动跳过。
- 土壤湿度自动决策。
- 脚本规则引擎。
- 从站离线后本地自行执行计划。
- 多主控协同计划。
- 计划自动抢占手动运行。
- 计划因故障清除后自动立即重跑。
