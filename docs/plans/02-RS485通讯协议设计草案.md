# 02 RS485 通讯协议设计草案

状态：成熟产品协议草案。本文目标是一次性定义可长期使用的主控-从站通讯边界；设计应完整覆盖本灌溉系统需要的多从站、安全配置、运行控制、状态轮询和故障处理，但不做脱离本产品场景的通用化扩展。

关联文档：

- `01-新版硬件启用范围与系统边界.md`
- `11-统一枚举与错误码表.md`
- `13-RS485详细Payload与主控重试规则.md`

## 设计目标

- 主控板通过 RS485 管理一个或多个从站。
- 一个从站代表一个真实水源，最多 8 路阀输出，最多 1 个流量计。
- 通讯协议优先保证稳定、安全和可维护性，不追求高吞吐。
- 从站不能依赖主控实时在线才能保证安全；通讯异常时，从站需要能本地安全停机。
- 协议要支持灵活配置输入、阀输出、主阀、自吸泵、低液位和流量计。
- 协议面向本产品固定硬件，不设计成通用工业寄存器平台。
- 协议允许后续固件升级时保留版本字段，但不在当前设计中引入复杂兼容层。

## 设计边界

必须支持：

- 多个 RS485 从站。
- 从站拨码地址。
- 从站安全配置同步。
- 配置版本和 CRC 校验。
- 单路运行命令和本地运行看门狗。
- 周期状态轮询。
- 从站本地故障和安全停机。
- 主控识别从站拒绝运行的原因。

明确不支持：

- 多主控。
- 一个从站多个流量计。
- 从站主动抢总线上报。
- 广播运行命令。
- JSON/文本协议。
- 高速连续采样数据上传。
- 通用寄存器映射模型。

## 协议选择

采用自定义二进制协议，借鉴 Modbus RTU 的主从轮询和 CRC 思路。

不采用 Modbus RTU 作为第一选择，原因：

- 本系统需要配置版本、运行状态机、故障状态、阀/输入资源映射等结构化数据。
- 用 Modbus 寄存器也能实现，但会让业务语义分散到很多寄存器地址中，后期维护不直观。

不采用 ASCII 或 JSON 协议，原因：

- 字节数更大。
- STC 从站解析成本更高。
- 长期运行控制场景中，二进制帧更适合。

## 基本通讯模式

- 主控永远主动发起通讯。
- 从站只响应主控请求，不主动发送，避免 RS485 总线冲突。
- 从站地址来自 4 位拨码开关。
- 地址范围为 `1..15`。
- 地址 `0` 保留，不用于运行命令。

## 波特率

默认波特率使用 `19200 bps`。

原因：

- 灌溉控制对速度要求低。
- 一条短命令通常只有几十字节，低波特率也足够。
- 保守波特率对长线、户外干扰、接线质量更友好。
- 如实机验证线缆、接地、干扰和从站数量都稳定，可以提供配置项切换到 `38400 bps`。

发送时间粗略估算：

| 波特率 | 60 字节帧 | 100 字节帧 |
|---:|---:|---:|
| 9600 bps | 约 62 ms | 约 104 ms |
| 19200 bps | 约 31 ms | 约 52 ms |
| 38400 bps | 约 16 ms | 约 26 ms |
| 115200 bps | 约 5 ms | 约 9 ms |

产品默认不使用 `115200 bps` 作为 RS485 控制波特率，除非后续实机验证证明现场环境足够稳定且确有必要。

## 帧格式

```text
SOF        2 字节   固定 0xA5 0x5A
VER        1 字节   协议版本
ADDR       1 字节   从站地址，1..15
SEQ        1 字节   包序号，用于匹配应答
CMD        1 字节   命令
LEN        1 字节   Payload 长度
PAYLOAD    N 字节
CRC16      2 字节   从 VER 到 PAYLOAD 的 CRC
```

说明：

- `SOF` 用于快速找到帧起点。
- `VER` 用于协议升级识别。
- `SEQ` 用于主控识别应答是否对应当前请求。
- `LEN` 限制在 `0..200`，避免从站内存压力。
- `CRC16` 采用 Modbus CRC16 算法，方便验证和调试。

## 应答规则

- 每条关键命令必须有应答。
- 应答帧使用相同 `SEQ`。
- 应答中需要包含结果码。
- 从站解析失败、CRC 错误或命令非法时，不执行动作。
- CRC 错误可以不应答，由主控超时重试。
- 命令合法但业务拒绝时必须应答错误码，例如配置版本不一致、未配置、正在运行、故障锁定。

## 命令集

| 命令 | 方向 | 用途 |
|---|---|---|
| `PING` | 主控 -> 从站 | 探测在线 |
| `GET_INFO` | 主控 -> 从站 | 读取固件版本、硬件能力、拨码地址 |
| `GET_STATUS` | 主控 -> 从站 | 轮询从站运行状态 |
| `GET_CONFIG_SUMMARY` | 主控 -> 从站 | 读取配置版本和配置 CRC |
| `SET_SAFE_CONFIG` | 主控 -> 从站 | 写入从站必要安全配置 |
| `GET_SAFE_CONFIG` | 主控 -> 从站 | 读取从站当前安全配置，用于维护核对 |
| `START_ZONE` | 主控 -> 从站 | 启动某一路水路 |
| `STOP` | 主控 -> 从站 | 停止当前运行 |
| `CLEAR_FAULT` | 主控 -> 从站 | 清除可恢复故障 |
| `FACTORY_RESET_CONFIG` | 主控 -> 从站 | 清空从站本地配置，需谨慎 |

## 命令编号

命令编号按功能分段，避免后续扩展时混乱。

| 编号 | 命令 | 说明 |
|---:|---|---|
| `0x01` | `PING` | 在线探测，最小命令 |
| `0x02` | `GET_INFO` | 读取设备信息和硬件能力 |
| `0x03` | `GET_STATUS` | 读取当前运行状态 |
| `0x04` | `GET_CONFIG_SUMMARY` | 读取配置版本和 CRC |
| `0x10` | `SET_SAFE_CONFIG` | 写入从站必要安全配置 |
| `0x11` | `GET_SAFE_CONFIG` | 读取从站当前安全配置，用于维护核对 |
| `0x20` | `START_ZONE` | 启动某一路水路 |
| `0x21` | `STOP` | 停止当前运行 |
| `0x22` | `CLEAR_FAULT` | 清除可恢复故障 |
| `0x30` | `FACTORY_RESET_CONFIG` | 清空从站本地配置 |

预留区间：

- `0x01..0x0F`：查询和状态类命令。
- `0x10..0x1F`：配置类命令。
- `0x20..0x2F`：运行控制类命令。
- `0x30..0x3F`：维护类命令。
- `0x40..0x7F`：保留给本产品后续明确功能扩展。
- `0x80..0xFF`：暂不使用，避免和应答标志、错误扩展混淆。

## 应答 Payload 基本格式

所有需要应答的命令，应答 Payload 以统一结果码开头：

```text
result_code    1 字节
data           N 字节，按命令定义
```

主控判断应答时先看：

1. 帧 CRC 是否正确。
2. 地址是否匹配。
3. `SEQ` 是否匹配。
4. `CMD` 是否为对应命令。
5. `result_code` 是否为成功。

## 故障标志

故障标志统一定义见 `11-统一枚举与错误码表.md`。

`fault_flags` 使用 32 位位图。一个状态应答里可以同时携带多个故障标志。

| 位 | 标志 | 含义 |
|---:|---|---|
| 0 | `FAULT_CONFIG_INVALID` | 本地安全配置无效 |
| 1 | `FAULT_COMM_TIMEOUT` | 运行中通讯超时 |
| 2 | `FAULT_LOW_LEVEL` | 低液位触发 |
| 3 | `FAULT_NO_FLOW` | 无水或无脉冲 |
| 4 | `FAULT_HIGH_FLOW` | 高流量 |
| 5 | `FAULT_LOW_FLOW` | 低流量 |
| 6 | `FAULT_VOLTAGE_LOW` | 系统电压过低 |
| 7 | `FAULT_VOLTAGE_HIGH` | 系统电压过高 |
| 8 | `FAULT_RUN_TIMEOUT` | 运行超时 |
| 9 | `FAULT_OUTPUT_FORCED_OFF` | 输出被本地安全策略强制关闭 |
| 10 | `FAULT_ADDRESS_CONFLICT` | 主控检测到地址冲突后标记，不一定由从站本地判断 |
| 11 | `FAULT_STORAGE_ERROR` | 从站本地配置保存或读取失败 |

保留位：

- 位 12 到 31 预留。

## 配置同步策略

主控保存完整业务配置。

从站保存必要安全配置：

- 输入用途映射。
- 流量计输入。
- 低液位输入和有效电平。
- 主阀输出。
- 自吸泵输出。
- 阀门 PWM 吸合/保持参数。
- 通信超时时间。
- 单次最大运行时间。
- 当前必要的流量保护参数。

同步流程：

1. 主控轮询从站 `GET_CONFIG_SUMMARY`。
2. 从站返回本地配置版本和 CRC。
3. 主控发现版本一致，允许运行。
4. 主控发现版本不一致，下发 `SET_SAFE_CONFIG`。
5. 从站校验配置合法后保存，并返回新的版本和 CRC。
6. 从站配置无效或保存失败时，拒绝运行。

## 运行命令策略

运行命令使用短命令，不携带完整硬件配置。硬件资源映射、主阀、自吸泵、输入用途、PWM 参数等保存在从站本地；`START_ZONE` 只携带本次运行必须变化的参数，例如运行时长、流量计参数和保护阈值。

`START_ZONE` 携带：

```text
config_version
run_id
zone_index
duration_sec
expected_flow_ml_min
本次流量保护参数
```

从站本地已经保存：

- 哪一路输入是流量计。
- 哪一路输入是低液位。
- 哪一路阀是主阀。
- 自吸泵是否启用。
- 阀门 PWM 参数。
- 通信超时。
- 单次最大运行限制。
- 安全输入有效电平。

从站收到 `START_ZONE` 后：

1. 校验配置版本。
2. 校验水路是否有效。
3. 校验当前是否空闲且无不可恢复故障。
4. 开启本地运行状态机。
5. 返回已接受或拒绝原因。

## Payload 定义来源

本文只保留协议总览、命令边界和状态语义，不再重复定义每个命令的详细 Payload。

详细字段、字节序、单位、长度、CRC 范围、重试规则和地址/总线冲突处理，以 `13-RS485详细Payload与主控重试规则.md` 为唯一实现依据。

这样处理的原因：

- 避免 `02` 和 `13` 两处同时维护字段，导致 CRC 宽度、位图长度或字段顺序不一致。
- 代码结构体、协议解析、Web 维护页和测试用例都只对齐 `13`。
- `02` 只负责解释为什么采用当前协议模型，以及命令集和安全边界。

## 状态轮询策略

主控周期性发送 `GET_STATUS`。

轮询周期：

- 空闲从站：每 `1..3` 秒轮询一次。
- 当前运行从站：每 `500 ms..1 s` 轮询一次。
- 离线从站：退避重试，例如 `3 s`、`5 s`、`10 s`。

状态应答包含：

```text
state
fault_flags
active_zone
valve_bitmap
input_bitmap
flow_pulse_count
flow_rate_ml_min
voltage_mv
uptime_sec
config_version
last_stop_reason
```

## 通讯异常安全策略

从站运行中如果通讯丢失：

- 短时间丢包：继续运行，内部累计通讯异常时间。
- 超过配置的通讯超时时间：立即关闭自吸泵、主阀和所有分支阀。
- 通讯恢复后：保持停止状态，等待主控重新下发命令。
- 不自动恢复丢失前的运行。

这些动作属于底层安全策略，不允许用户关闭。

## 产品不支持的通讯能力

- 从站主动上报。
- 多主控。
- 一个从站多个流量计。
- 广播运行命令。
- JSON/文本协议。
- 高速连续采样数据上传。
- 通用寄存器映射模型。
