# 00 项目背景

## 当前目标

当前重建阶段的目标不是修补旧原型，而是重新建立这个 ESP32 灌溉项目的完整产品方案：

- 先确认硬件边界。
- 再确认产品定位、对象模型、状态模型、业务规则和信息层级。
- 再设计低保真原型，确认页面数量、页面边界、信息层级和任务流。
- 低保真原型确认后，再设计最终用户可理解、可操作、可评审的高保真原型。
- 高保真原型确认后，才进入编码、测试和交付阶段。

前四轮已经失败，主要风险是沿着旧方案不断局部修补，导致页面、状态、对象和任务流越来越复杂，但用户仍无法形成稳定判断。因此当前重建阶段从 `pcb/` 重新开始。

## 产品定位

本项目定位为智能浇水/灌溉系统，不限定为家庭庭院、温室、小农场、小菜地、雨棚或露天等具体场景。当前重建阶段按产品对象、硬件能力和业务规则建模，不按场景标签人为拆分功能。

当前已经重新确认的背景：

- 系统是一个 ESP32 主控加从站的灌溉控制系统。
- 主控板只做主控，不作为直接阀控设备。
- 系统必须至少有一个从站才能完成灌溉控制。
- 一个从站对应一个真实水源。
- 从站负责水源侧电磁阀输出和传感输入。
- 主控通过 RS485 控制从站，并提供 Web、时间、状态汇总和后续本地显示基础。
- 主控附近可外接 SHT30 或同类 I2C 温湿度传感器，用于观察主控附近温湿度。
- 本地 LCD2004 和四个业务按键属于现场操作入口，但当前设计顺序是先把 Web 完整功能逻辑确认清楚，再从中抽取最小现场能力。
- 用户最终需要一个高保真 Web 原型；但当前必须先完成逻辑方案和低保真原型。
- 安装、日常使用和维护可以先按同一个用户处理，不设计多角色、多权限或多租户。
- 当前目标产品不按第一版、第二版或 MVP 拆分；本轮应把期望功能一次性设计完整，后续只作为优化、修复或硬件小改继续演进。
- 当前目标产品需要支持自动灌溉、Web 管理和现场手动控制。酷炫可视化不是目标重点，逻辑完整、状态清楚和操作可靠更重要。

当前不能直接继承的结论：

- 不能直接继承旧页面结构。
- 不能直接继承旧 API、协议、对象模型或记录模型。
- 不能直接继承旧固件骨架。
- 不能直接继承旧文档里的主控/从站职责划分。
- 不能因为旧方案写过某个能力，就把它视为当前重建阶段已确认能力。

## 当前重建阶段成功标准

当前重建阶段至少要完成以下确认，才进入低保真原型：

1. 硬件边界清楚：哪些能力来自当前 PCB，哪些资料缺失，哪些只是可能能力。
2. 用户对象清楚：用户看到和操作的对象是什么，例如设备、水源、水路、计划、事件等是否成立。
3. 状态模型清楚：运行、停止、故障、维护、配置、同步、不可用等状态如何区分。
4. 任务流清楚：用户如何完成日常浇水、自动计划、异常处理、配置和维护。
5. 信息层级清楚：每个页面优先回答什么问题，哪些内容不能进入普通用户路径。
6. 原型依据清楚：低保真和高保真原型只能表达已确认方案，不能借视觉设计补功能空洞。

## 当前关键风险

- 不能把产品人为限制为家庭、庭院、雨棚、露天等具体使用场景；这会把通用灌溉对象拆碎。
- 主控板上存在多种硬件资源，但“存在于 PCB”不等于“当前目标产品必须启用”。
- 从站板也存在多种资源，但同样不能因为存在就全部进入当前目标产品范围。
- 主控 + 从站结构已经确认，后续必须围绕这个结构建模，不能再回到单主控直接控制路线。
- 系统参数只承载系统级参数；水源级和水路级配置回到对应对象页面，不再用一个“设置”页包所有低频项。
- 如果先做高保真视觉，再补对象和状态模型，会重复前四轮失败路径。
