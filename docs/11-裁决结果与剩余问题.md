# 11 裁决结果与剩余问题

## 已裁决

### 计划

- 自动计划使用统一周期模型：周期长度、周期起始日期、周期内执行日、开始时间列表。
- 每天、按星期、浇几停几都由统一周期模型表达。
- 计划触发时，水源忙碌或不具备启动条件则跳过本次步骤，生成事件，不排队，不补跑。
- 停用计划只阻止后续未触发任务，不停止正在运行的任务。
- 断电、时间无效或离线导致错过计划，不补跑。

### 手动运行

- 手动浇水不进入队列，不抢占当前任务。
- 水源正在运行时，不能手动启动同一水源的另一路。
- 手动时长不设全局默认值。
- 手动时长可以来自用户本次输入、上次手动时长，或从某个计划的对应水路步骤带入。
- 最大单次运行时长是系统参数，默认 120 分钟，同时约束计划和手动运行。

### 校准和参考流量

- 水路校准采集稳定流量默认按 60 秒设计。
- 校准成功后需要用户确认保存参考流量。
- 校准不生成普通浇水记录，只生成校准事件。
- 正常完成的浇水记录要保存稳定流速。
- 用户可以从正常浇水记录中把稳定流速采用为该水路参考流量。

### 事件和恢复

- 事件只分三类：信息、提示、阻断。
- 发生中事件不能手动清除，只能自动恢复。
- 已恢复事件保留历史，不要求用户确认。
- 校准、配置变更、计划跳过、用户停止等进入事件，不混入浇水记录。

### 通信和离线

- 从站离线后，正在运行任务按异常停止处理。
- 离线恢复后不补跑错过的计划。
- RS485 通信速率、轮询周期和离线失败次数属于系统参数。
- 原型只表达在线、离线和禁止启动状态，不展示底层轮询细节。

### 记录保留

- 浇水记录和事件历史保留策略先不在产品方案中定死。
- 实现阶段根据文件系统容量、基础库能力和循环覆盖策略确定。
- 产品层面只要求容量满时自动循环覆盖，不能因为记录满而影响灌溉控制。

## 低保真阶段再确认

以下内容不在当前逻辑文档中继续细化，进入低保真原型时通过页面评审确认：

1. 顶层页面数量是否最终固定为 7 个。
2. 诊断、维护、统计、天气、通知、OTA、LCD 是否都不做顶层页面。
3. 每个页面的具体信息密度、区块顺序和入口位置。
